## 5. 下一步计划

后续开发采用"先复现稳定, 再双主线推进"的结构。双主线分别是:

- 产品主线: 把 Cocanvas 做成真正适合头脑风暴、信息整理、团队讨论和输出结论的看板工具。
- 分布式主线: 保留课程作业属性, 强化一致性、容错、可用性、可观测性、可扩展性和可验证性。

### 5.3 分布式主线: 非功能能力

目标: 答辩时能清楚展示 Cocanvas 是一个扎实的分布式系统。

计划:

- 可用性:
  - backend1/backend2 故障切换
  - Redis 短暂不可用时的降级策略
  - MySQL 重启后的恢复验证
  - 网络断开后的自动重连
- 一致性:
  - HLC 测试
  - CRDT 属性级 LWW 测试
  - tombstone 防复活测试
  - 乱序 op 到达测试
  - 多人并发编辑测试
- 可观测性:
  - 状态面板
  - 当前连接节点
  - room 路由结果
  - pending ops 数量
  - 最近恢复 ops 数
  - snapshot 版本
  - 节点在线状态
- 可扩展性:
  - 大房间压测脚本
  - cursor 限流
  - shape-preview 合并
  - 批量 op 协议
  - 大白板视口裁剪或虚拟化
- 安全与权限:
  - 房间访问码
  - 只读 / 编辑权限
  - op payload 校验
  - WebSocket join token

验收标准:

- 可以演示节点故障、重连恢复、CRDT 收敛、历史恢复。
- 可以用状态面板解释当前系统内部状态。
- 关键一致性规则有自动化测试和文档支撑。

### 5.4 历史回放与数据一致性

目标: 既能实时协作, 也能安全查看过去状态。

计划:

- 拆分实时 `shapeStore` 和只读 `replayStore`。
- 历史回放不再污染实时协作状态。
- 当前历史恢复是完整重放 ops, 后续恢复为 snapshot + delta。
- 给 snapshot 增加完整性校验或版本字段。
- 增加历史恢复自动化测试。

验收标准:

- 历史回放和实时编辑互不影响。
- snapshot + delta 能在数据量增大时保持可用。
- 旧坏 snapshot 不会再次污染恢复结果。
