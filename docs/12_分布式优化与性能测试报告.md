# 12 分布式优化与性能测试报告

> 更新日期: 2026-05-30  
> 测试环境: Docker Desktop 4.37.1, Docker Engine 27.4.0, Docker Compose v2.31.0  
> 当前阶段: Cocanvas 分布式协作链路从“功能演示”升级为具备分片广播、可靠持久化确认、快照增量恢复、节点发现优化和可重复性能测试能力的版本。

## 1. 优化摘要

本轮优化聚焦分布式实时协作的可靠性、扩展性和恢复性能。

已完成:

- Redis Pub/Sub 从单一全局频道改为 64 个 hash 分片频道。
- op 处理顺序改为先持久化成功, 再应用本地副本、发送 ack 和跨节点广播。
- 新增 `op-ack` 协议, 前端 pending outbox 只有收到服务端 ack 后才删除。
- 历史恢复改为优先读取最新 snapshot, 再回放 snapshot 之后的增量 op。
- snapshot payload 保留每个属性自己的 HLC/writer 版本, 避免恢复后 CRDT 版本不可比较。
- Redis 节点发现去掉 `KEYS`, 改为 `cocanvas:nodes` Set + 节点 TTL key。
- 一致性哈希环增加缓存, 节点列表不变时不重复构建。
- 前端 pending op 从内存队列升级为 `localStorage` outbox, 按 `roomId:userId` 隔离。
- Nginx WebSocket 代理增加长连接读写超时。
- snapshot 写入改为只有一致性哈希 owner 节点负责, 避免多后端重复写同一房间 snapshot。
- 新增 `scripts/distributed-check.mjs`, 用于 REST/WebSocket 分布式验收和基础性能压测。

尚未完成:

- WebSocket 慢客户端背压和发送队列隔离。
- cursor / shape-preview 的瞬态消息本地化或低可靠通道优化。

## 2. 代码变更范围

后端:

- `RedisRealtimeBroadcaster`: 发布到 `cocanvas:room-events:{shard}` 分片频道。
- `RedisPubSubConfig`: 启动时订阅 64 个分片频道。
- `RedisNodeRegistry`: 使用 Redis Set 维护节点索引, 不再扫描 `KEYS`。
- `NodeRouter`: 缓存一致性哈希环。
- `HistoryService`: 提供 `tryRecordOperation`, 使用 snapshot + 增量 op 恢复, 并限制 snapshot 只由 owner 节点写入。
- `RoomReplicaService`: 拆分 HLC 合并和已提交 op 应用, 增加 versioned snapshot。
- `CollabWebSocketHandler`: op 落库失败不广播; 落库成功后发送 `op-ack`。
- `OpAckMessage`: 新增服务端 ack 消息类型。

前端:

- `Room.tsx`: pending op 落 localStorage, 发送 op 后等待 `op-ack` 删除, 重连后自动 flush。
- `shapeStore.ts`: snapshot 恢复支持版本化属性值。
- `protocol.ts`: 增加 `op-ack` 服务端消息类型。

基础设施:

- `nginx.conf`: WebSocket location 增加 `proxy_read_timeout 1h` 和 `proxy_send_timeout 1h`。

测试脚本:

- `scripts/distributed-check.mjs`: 创建房间、双节点 WebSocket 连接、跨节点广播验证、op ack 延迟统计、REST 路由性能统计、历史恢复查询。

## 5. 功能验收

使用脚本:

```powershell
node scripts\distributed-check.mjs
```

脚本覆盖:

- `/api/health`。
- `/api/cluster/nodes`。
- 创建测试房间。
- 重复查询房间路由。
- 分别连接 `/ws/backend1/collab` 和 `/ws/backend2/collab`。
- 发送首条 op 并等待发送方 `op-ack`。
- 确认另一个后端连接收到跨节点 op 广播。
- 连续发送 N 条 op, 统计 ack 延迟。
- 查询历史恢复接口。

一次基础结果:

```text
health=ok nodes=backend1,backend2
room=perfmprzoe8c routed=ws://localhost:8088/ws/backend2/collab
route_get count=300 avg=6.34ms p50=5.82ms p95=8.37ms
ws_cross_node ok first_ack=310.99ms
ws_ops count=300 total=1972.52ms throughput=152.1ops/s ack_avg=6.56ms ack_p50=6.26ms ack_p95=9.21ms
history snapshot=replay-from-start-perfmprzoe8c ops=292 latency=98.00ms
```

说明:

- 首条跨节点 ack 有一次冷启动/连接建立影响, 后续压测明显稳定。
- 脚本最初使用本机 `Date.now()` 传 `at`, 与容器时间存在边界偏差, 可能导致历史查询少量 op。脚本已改为默认不传 `at`, 使用后端默认最大时间。

## 6. 性能测试结果

### 6.1 顺序 WebSocket op 压测: 1500 ops

命令:

```powershell
$env:COCANVAS_OPS='1500'
$env:COCANVAS_ROUTE_REQUESTS='1000'
node scripts\distributed-check.mjs
```

结果:

```text
route_get count=1000 avg=4.32ms p50=4.18ms p95=6.00ms
ws_cross_node ok first_ack=7.35ms
ws_ops count=1500 total=7023.70ms throughput=213.6ops/s ack_avg=4.67ms ack_p50=4.46ms ack_p95=6.56ms
history snapshot=replay-from-start-perfmprzr5ts ops=1501 latency=21.66ms
```

容器资源快照:

```text
backend1 CPU: 83.36%, memory: 338.6MiB
backend2 CPU: 106.88%, memory: 340.1MiB
redis CPU: 2.72%, memory: 15.48MiB
mysql CPU: 10.10%, memory: 472.3MiB
```

结论:

- Redis 压力很低, 分片 Pub/Sub 不是瓶颈。
- op ack 延迟主要消耗在后端处理和 MySQL 落库。
- 1500 条顺序 op 下 ack p95 约 6.56ms, 对课程演示和小中型协作房间是健康的。

### 6.2 并发 REST 路由/API 压测

测试参数:

- 总请求数: 2000
- 并发数: 80
- 接口: `/api/rooms/{roomId}`

结果:

```text
route_concurrent total=2000 concurrency=80 elapsed=956.51ms rps=2090.9 avg=36.37ms p50=26.20ms p95=113.31ms p99=342.36ms
```

容器资源快照:

```text
backend1 CPU: 5.91%, memory: 356.7MiB
backend2 CPU: 6.96%, memory: 343.5MiB
redis CPU: 0.21%, memory: 16.06MiB
mysql CPU: 133.97%, memory: 512.4MiB
```

结论:

- 高并发 REST 查询下 p99 偏高, 主要瓶颈更像 MySQL 房间查询, 而不是 Redis 节点发现或 hash ring。
- 如果继续优化 REST 路由性能, 下一步应考虑 room metadata 本地缓存/Redis cache, 而不是继续优化一致性哈希本身。

### 6.3 snapshot 后历史恢复

等待 snapshot 周期后查询:

```powershell
Invoke-RestMethod http://localhost:8088/api/rooms/perfmprzr5ts/history
```

结果:

```text
snapshot: 命中最新 snapshot
ops: 0
```

结论:

- 历史恢复已经从全量操作回放降为 snapshot 命中 + 增量回放。
- 对长生命周期房间, 恢复成本不会再随全部历史 op 数线性增长。

### 6.4 单 owner snapshot 验证

问题:

- 测试发现同一房间 snapshot 会被两个后端重复写。

修复:

- `HistoryService.saveSnapshot` 中增加 owner 判断。
- 只有 `nodeRouter.routeRoom(roomId).nodeId()` 等于当前 `NodeIdentity.nodeId()` 的节点写 snapshot。

验证:

```text
room=ownercheck1
ops=201
等待多个 snapshot 周期后 snapshots=1
```

结论:

- 多节点重复 snapshot 写入问题已修复。

## 7. 当前剩余风险

### 7.1 WebSocket 广播仍缺少背压

`RoomSessionRegistry.broadcastInRoom` 仍同步逐个 `sendMessage`。慢客户端可能拖慢同房间广播。

建议下一步:

- 每个 session 一个发送队列。
- cursor / shape-preview 超限可丢弃。
- op 超限则断开连接, 要求客户端重连后从历史恢复。

### 7.2 cursor / shape-preview 仍走跨节点广播

cursor 和 shape-preview 属于瞬态状态, 不需要和 op 同级可靠性。

建议下一步:

- cursor 限频继续保留。
- 可进一步按节点本地连接情况决定是否发布跨节点消息。
- 或拆出低优先级/可丢弃频道。

### 7.3 REST 房间查询高并发 p99 偏高

80 并发下 `/api/rooms/{roomId}` p99 达到 342ms, MySQL CPU 较高。

建议下一步:

- 给房间元数据增加 Redis cache 或本地 Caffeine cache。
- 创建/更新/归档房间时主动失效缓存。
- `wsUrl` 路由结果可短 TTL 缓存, 节点列表变化时失效。

## 8. 结论

本轮优化后, Cocanvas 的分布式链路已经具备以下答辩亮点:

- 多后端节点通过 Redis 心跳注册和一致性哈希路由。
- Redis Pub/Sub 使用分片频道降低全局广播浪费。
- op 使用 HLC/CRDT 合并, 并通过 MySQL 操作日志持久化。
- 服务端持久化成功后才 ack 和广播, 失败由客户端 pending outbox 重试。
- 历史恢复使用 versioned snapshot + 增量 op, 避免从头回放。
- snapshot 只由 owner 节点写入, 避免多副本重复持久化。
- 已有 Docker 化压测脚本和可复现测试数据。

整体上, 系统已经从“分布式概念展示”进入“可解释、可压测、可恢复”的阶段。
