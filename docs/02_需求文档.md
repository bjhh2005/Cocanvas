# Cocanvas 需求文档

## 1. 项目概述

Cocanvas 是一个面向多人实时协作的分布式白板系统,核心定位是 Miro 的简化复刻 + 分布式系统能力演示。系统允许多个用户同时进入一个"房间",在共享的无限画布上并发地创建、移动、修改图形元素,所有操作通过分布式后端实时同步给房间内所有成员,并支持任意时间点的历史回放。

项目的设计目标不是追求 Miro 的功能完备度,而是在受控的功能子集下,**完整展示分布式系统的核心问题与解决方案**:WebSocket 集群下的房间路由、跨节点消息广播、CRDT 并发冲突消解、节点故障切换、操作日志持久化与回放、最终一致性收敛等。

## 2. 术语定义

| 术语 | 含义 |
| --- | --- |
| 房间 (Room) | 一个独立的协作单元,拥有唯一 roomId,内部包含一份共享画布状态和加入其中的所有用户 |
| 图形 (Shape) | 画布上的一个可编辑元素,拥有前端生成的全局唯一 UUID,终生不变 |
| 操作 (Operation) | 对画布状态的一次原子修改,如 create / update / move / delete,是 CRDT 同步的基本单位 |
| HLC | Hybrid Logical Clock,混合逻辑时钟,作为操作的逻辑时间戳,用于跨节点排序与冲突消解 |
| 副本 (Replica) | 同一房间在不同客户端 / 服务节点上的画布状态拷贝 |
| 快照 (Snapshot) | 某一时刻房间画布的全量状态序列化结果,用于加速历史回放与故障恢复 |

## 3. 功能需求

### 3.1 用户与房间

- 用户可以创建一个新房间,系统为其分配唯一的 roomId
- 用户可以通过 roomId 加入已存在的房间
- 进入房间时,系统为该用户分配一个会话内唯一的用户 ID 与显示颜色(用于光标和操作归属标识)
- 同一房间允许至少 4 名用户同时在线协作(演示规模),设计上不应对此数量做硬上限

### 3.2 画布与图形编辑

- 画布为无限大平面,支持平移与缩放
- 支持的图形类型限定为:**矩形、圆形、文本、连线(箭头)、图片**,不支持其他类型
- 每个图形拥有以下基本属性:
  - 全局唯一 UUID(前端生成,创建后不可变)
  - 位置(绝对坐标 x, y)
  - 尺寸(宽 w, 高 h;圆形为半径)
  - 视觉属性(填充色、描边色、描边宽度等)
  - 文本图形额外携带文本内容
  - 连线额外携带起止图形的 UUID 引用
- 支持的图形操作:
  - **create**:在指定位置创建一个新图形
  - **delete**:删除指定 UUID 的图形
  - **update**:修改图形的视觉属性或文本内容
  - **move**:修改图形的位置
  - **resize**:修改图形的尺寸
- 当被引用的图形(连线的端点)被删除时,关联的连线应级联失效

### 3.3 实时协同

- 房间内任一用户的操作应实时广播给同房间的所有其他用户
- 每位用户的鼠标光标位置实时同步,以其分配的颜色显示在其他用户的视图中
- 多个用户对**同一图形的同一属性**同时进行修改时,系统必须按确定性规则消解冲突,且所有副本最终收敛到相同状态
- 冲突消解规则:
  - 采用基于操作的 CRDT 模型
  - 数值与离散属性(位置、尺寸、颜色、文本等)采用 **LWW(Last-Writer-Wins)** 策略
  - 写入先后顺序按 **HLC 时间戳** 比较,HLC 相同时按 nodeId 字典序打破平局
- 操作必须满足因果一致性:依赖于前序操作的后续操作(如先 create 再 update),在所有副本上的应用顺序必须保持因果先后

### 3.4 历史记录与时光回放

- 房间内所有操作必须持久化存储,不因节点重启或用户离线而丢失
- 操作日志表至少包含字段:房间 ID、操作 ID、用户 ID、操作类型、操作载荷、HLC 时间戳、写入时间
- 系统定期对房间画布状态生成全量快照(建议每 N 个操作或每 T 分钟触发一次)
- 前端提供时间轴滑块 UI,用户拖动滑块可以查看房间在任意历史时刻的画布状态
- 历史回放从距目标时刻最近的快照开始,叠加其后到目标时刻的增量操作,避免从零重放
- 历史回放视图为只读,不影响当前协作中的画布状态

### 3.5 节点故障与恢复

- 当某个 WebSocket 服务节点宕机时,其承载的所有房间应能被重新路由到其他存活节点
- 客户端在 WebSocket 连接断开后必须自动尝试重连
- 重连成功后,客户端必须重新加入原房间,并从后端拉取最新快照 + 增量操作完成状态重建
- 故障切换过程中,已成功持久化的操作不应丢失;未持久化但在客户端缓存的操作应在重连后重新提交

## 4. 非功能需求

### 4.1 性能

- 同节点内协同操作的端到端延迟(用户 A 操作 → 用户 B 看到)应低于 100ms(局域网环境)
- 跨节点协同操作的端到端延迟应低于 300ms
- 单个房间支持的图形数量不少于 1000 个,在此规模下画布交互(拖动、缩放)应保持流畅(≥ 30 FPS)
- 单个房间支持至少 10 名用户同时在线编辑而无明显卡顿

### 4.2 可用性

- 单个 WebSocket 节点故障不应导致整个系统不可用
- 故障切换 + 客户端重连完成时间应在 10 秒内
- 已持久化到 MySQL 的操作必须保证不丢失

### 4.3 一致性

- 系统提供**最终一致性**保证:在没有新操作产生的前提下,所有副本最终收敛到相同的画布状态
- 系统提供**因果一致性**保证:具有因果依赖的操作在所有副本上按相同的因果顺序应用
- 系统**不**提供强一致性(线性一致性)保证,这是在 CAP 权衡中为可用性与分区容忍性做出的设计选择

### 4.4 可扩展性

- WebSocket 服务节点支持水平扩展,新增节点后能够通过一致性哈希自动承担部分房间的负载
- 房间到节点的路由应满足:同一房间的用户尽可能被路由到同一节点,减少跨节点通信开销
- 操作日志的 MySQL 存储支持按 roomId 哈希分片,支持后续扩展数据库节点
- Redis 作为跨节点 Pub/Sub 广播通道,频道按房间维度划分(如 `room:<roomId>:ops`)

### 4.5 可观测性

- 服务端应记录关键事件日志:连接建立 / 断开、房间路由、操作接收 / 广播 / 持久化、节点故障
- 系统应暴露基本运行指标:每节点当前 WebSocket 连接数、每房间活跃用户数、操作处理延迟、Redis Pub/Sub 消息速率

### 4.6 容错性

- 网络分区导致同房间用户被割裂到不同节点时,系统不应崩溃,各分区内可继续操作,分区恢复后通过 CRDT 合并收敛
- 客户端发送的操作在因网络问题暂时无法到达服务端时,应在本地缓存并在网络恢复后重传

## 5. 技术约束

- 前端:React + Konva(画布渲染)
- 实时通信:WebSocket
- 跨节点广播:Redis Pub/Sub
- 路由策略:一致性哈希(房间 → 节点)
- 时钟方案:Hybrid Logical Clock (HLC)
- 持久化:MySQL,按 roomId 分片
- 冲突消解:基于操作的 CRDT,LWW + HLC

## 6. 非目标(明确不做)

为避免范围蔓延,以下功能明确不在本项目范围内:

- 用户注册、登录、权限管理(房间默认对所有获得 roomId 的人开放)
- 富文本编辑(文本仅支持纯文本)
- 图形组合 / 分组 / 图层管理
- 评论、@提及、通知系统
- 移动端适配(仅保证桌面浏览器可用)
- 端到端加密、传输加密以外的安全机制
- 跨房间的数据共享或模板市场
