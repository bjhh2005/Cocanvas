# Cocanvas

## 一、整体评价：高价值、高匹配、高回报
- **课程契合度极高**：WebSocket集群、Pub/Sub跨节点通信、CRDT冲突消解、服务发现与一致性哈希、操作日志回放……几乎每一块都是分布式系统经典问题的直接映射，教授一眼就能看出你们在解决“真正的分布式难题”。
- **演示效果炸裂**：四人四色光标在同一个大屏上实时飞动，kill掉一个后端节点协作无感继续，再加时光回溯滑块拖拽历史——这种可视化冲击力，答辩通过率会大幅提升。
- **技术深度足够区分度**：尤其是CRDT实现，在本科生/硕士生课程中属于进阶内容，一旦做出来就是降维打击。

## 二、Miro复刻的难点与应对
Miro的核心是**无限画布 + 实时协同 + 便签/图形/连线等丰富元素**，你们不需要复刻它的全部，但要抓住精髓：

### 1. 画布与图形模型（成员D）
- **建议锁定图形子集**：矩形、圆形、文本、连线（箭头）、图片，不要贪多。这些已经足够展示协同冲突（两人同时拖拽同一个矩形的位置/大小、同时修改文本）。
- **坐标与变换**：用绝对坐标 + 矩阵变换，避免嵌套的变换混乱。每个图形一个唯一ID（UUID），由前端生成，终生不变，这是CRDT操作定位的基石。
- **画布状态管理**：前端维护一份完整的图形状态，基于CRDT的操作流来更新。推荐使用**Konva**，它对大量形状的命中检测和事件处理比Fabric更友好，且文档更清晰。

### 2. CRDT同步算法（成员B，核心难点）
这是项目能否封神的关键。你们要做的是**基于操作的CRDT**，而不是全量同步。
- **选择适合的CRDT结构**：图形对象可以用**LWW-Element-Set（最后写入胜出元素集）** 的变体，每个图形属性（位置、尺寸、颜色等）独立带上向量时钟或逻辑时间戳。更严格的做法是实现一个简化的**RGA（Replicated Growable Array）** 来处理图形之间的顺序，但代价较大，建议先用“每个图形独立冲突解决”的方案。
- **冲突解决策略**：对于位置/大小等数值属性，最简单且直观的冲突解决是**最后修改者获胜（LWW）**，配合Hybrid Logical Clock（混合逻辑时钟）以接近物理时间。操作日志按 (nodeId, hlc) 排序，后者覆盖前者。**一定要在答辩时讲清楚为什么最终一致、如何保证冲突后所有副本收敛到相同状态。**
- **操作定义**：create, delete, update(shapeId, attrs), move(shapeId, newX, newY) 等。每个操作携带依赖的父操作ID，实现因果一致性。
- **向量时钟 vs 混合逻辑时钟**：如果不想引入中心化组件，可使用向量时钟，但节点增多时钟维度膨胀。用HLC（混合逻辑时钟）更简洁，且支持跨节点排序。建议采用HLC + nodeId作为操作唯一标识，LWW按HLC比较，同时平局时按nodeId打破。

### 3. 多节点WebSocket集群 + Redis Pub/Sub（成员A & B）
- **房间一致性路由**：按roomId做一致性哈希，将同一房间的所有用户尽可能路由到同一组服务节点（比如主从两个节点），这样房内消息走内存广播，跨节点才走Redis Pub/Sub，大幅降低延迟和跨节点带宽。一致性哈希可以用Ketama算法或多级哈希环。
- **节点故障切换**：当某个节点掉线时，其所服务的房间会被重新哈希到其他节点。但此时用户WebSocket会断开，需要前端实现**自动重连并重新加入房间**，然后从后端获取当前快照+增量操作日志进行状态重建。这个故障切换过程如果能在答辩时丝滑展示，会很加分。
- **Redis Pub/Sub的使用**：作为跨节点广播通道，可以按房间频道订阅（如 `room:<roomId>:ops`），所有服务节点订阅它，收到操作后本地广播给该房间内连接到本节点的所有用户。同时成员C的历史服务也监听此频道以持久化操作。

### 4. 历史记录与时光回放（成员C）
- **操作日志存储**：MySQL分片可以按roomId哈希分片。表结构：（room_id, operation_id, user_id, op_type, op_payload_json, hlc_timestamp, created_at）。操作ID由(HLC, nodeId)构成。
- **定期快照**：每隔固定操作数（如100个操作）或固定时间对房间画布状态做一次全量快照（JSON），存储到对象存储或MySQL单独表。回放时从最近快照开始重放后续增量操作，避免从零重放。
- **时光回放滑块**：前端根据滑块时间戳，向历史服务请求“该时间点之前最近快照 + 此后增量操作”，拿到后前端重放（但不要应用到当前编辑器状态，而是在一个隐藏画布上重放，或者直接渲染为静态帧）。可考虑用Canvas录制的方式，但最简单就是重放操作序列。
- **回放性能优化**：如果操作数非常多，可预计算关键帧（关键时间的完整状态）存储，加快随机访问。

## 三、可能被低估的坑与应对策略
1. **CRDT实现的正确性验证**：
   - 建议成员B在写核心算法时，用**混沌测试**：随机生成多个操作并发应用到不同副本，最终对比状态是否一致。可以写一个简单的模拟测试框架。
   - 考虑图形之间的依赖（如连线关联两个图形），删除一个图形时连线的处理，要有明确的规则（级联删除，将连线操作视为依赖父图形而因果失效）。

2. **前端性能**：
   - 当画布上有大量图形时（比如数千个），React状态频繁更新可能导致卡顿。建议用Konva直接操作Canvas节点，尽量减少React render。
   - 多光标直接用绝对定位的div或SVG元素跟随鼠标移动，不要放到Canvas内部。

3. **网络分区与脑裂**：
   - 如果网络分区导致同一房间的用户被割裂到不同节点（因为路由不感知分区），可能出现短暂的操作发散，恢复后通过操作序列合并。这正好能讲CAP和最终一致性。
