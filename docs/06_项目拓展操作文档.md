# 06 项目拓展操作文档

> **读者**：参与今天（Step 7-27）编写代码的四位成员。
> **目的**：指出哪些地方今天如果写得"太死"，后续扩展 CRDT、Redis 集群、历史回放等功能时会被迫大改；提供具体的"写法建议"，让你在不多写一行多余代码的前提下保住扩展口。

---

## 一、总原则

**今天的目标是跑通，不是设计完美系统。** 但以下两类写法代价不同：

| 类型 | 今天代价 | 后续代价 |
|---|---|---|
| **写死具体实现**（if/else 堆逻辑、直接 new 依赖、URL 字面量） | 省 5 分钟 | 重构一个文件甚至多个文件 |
| **留一个接口/常量**（哪怕实现只有一个） | 多写 3 行 | 只换实现，不改调用处 |

下面按模块逐一说明。

---

## 二、后端

### 2.1 广播层——最高优先级

**问题**：Step 13/15 里 `CollabWebSocketHandler` 直接调用 `RoomSessionRegistry.broadcastInRoom()`。这在单机没问题。Step 36 需要换成 Redis Pub/Sub，如果你把广播调用写散在 handler 里，到时候要全文搜索替换。

**建议**：今天就抽一个接口，哪怕只有一个实现。

```java
// com/cocanvas/pubsub/OpsBroadcaster.java
public interface OpsBroadcaster {
    void broadcast(String roomId, String jsonPayload, WebSocketSession exceptSession);
}

// com/cocanvas/pubsub/LocalOpsBroadcaster.java
@Component
@ConditionalOnProperty(name = "ops.broadcaster", havingValue = "local", matchIfMissing = true)
public class LocalOpsBroadcaster implements OpsBroadcaster {
    private final RoomSessionRegistry registry;
    // ... 直接调 registry.broadcastInRoom
}
```

`CollabWebSocketHandler` 只依赖 `OpsBroadcaster`，不知道有没有 Redis。
Step 36 时只需新建 `RedisPubSubBroadcaster` 并切 `application.yml` 里的一行配置。

---

### 2.2 RoomSessionRegistry——不要暴露内部 Map

**问题**：Step 13 建的 `RoomSessionRegistry` 内部是 `ConcurrentHashMap<String, Set<WebSocketSession>>`。如果其他类直接拿 Map 操作，Step 37（多节点）时 Map 会分散在每个 JVM 里，没法跨节点查 session。

**建议**：
- `RoomSessionRegistry` 只暴露三个方法：`join` / `leave` / `broadcastInRoom`
- **不要**提供 `getSessions(roomId)` 返回裸 Set，调用方如果需要遍历，让它通过 `broadcast` 走
- 内部 Map 加 `private`，不要 getter

这样 Step 36 把 registry 换成基于 Redis 的实现时，调用方零改动。

---

### 2.3 消息分发——用 switch/Map 不要用 if-else 链

**问题**：Step 15 的 `handleTextMessage` 需要分发 join/cursor/op 三种消息。如果写成：

```java
if (msg instanceof JoinMessage) { ... }
else if (msg instanceof CursorMessage) { ... }
else if (msg instanceof OpMessage) { ... }
```

Step 21 加 `op`，Step 47 加 `ping`，每次都要回来改这个文件，且越来越长。

**建议**：用 Java 17+ switch 表达式或 `Map<Class, MessageHandler>` 策略模式：

```java
// 方案 A：Java 17 switch（最简单，今天推荐）
switch (msg) {
    case JoinMessage m   -> handleJoin(m, session);
    case CursorMessage m -> handleCursor(m, session);
    case OpMessage m     -> handleOp(m, session);
    default              -> sendError(session, "unknown_type", null);
}
```

每次新增消息类型只加一个 `case`，不改已有逻辑。

---

### 2.4 `wsUrl` 不要写死字符串

**问题**：Step 12 `RoomController` 的响应里 `wsUrl` 写死 `"ws://localhost:8080/ws/collab"`。Step 45 要改成由一致性哈希路由到对应节点。

**建议**：今天提取成配置项，而不是字面量：

```yaml
# application.yml
cocanvas:
  ws-base-url: ws://localhost:8080
```

```java
@Value("${cocanvas.ws-base-url}")
private String wsBaseUrl;

// 返回
return wsBaseUrl + "/ws/collab";
```

Step 45 时只需把这个值的生成逻辑改成从 `NodeRouter` 取，`RoomController` 本身不动。

---

### 2.5 `CollabWebSocketHandler` 里的 session 状态——用 attributes，不用字段

**问题**：Step 15 需要在断连时知道"这个 session 属于哪个 roomId/userId"。如果用 handler 的成员变量（如 `Map<WebSocketSession, String> sessionToRoom`）存储，线程安全麻烦，也不利于后续节点化。

**建议**（开发计划里已说明，这里强调）：
```java
// 进房时写
session.getAttributes().put("roomId", joinMsg.roomId());
session.getAttributes().put("userId", joinMsg.userId());

// 断连时读
String roomId = (String) session.getAttributes().get("roomId");
```

`session.getAttributes()` 是线程安全的 Map，天然随 session 存活，不用额外维护。

---

### 2.6 `Op` 类——今天忽略 `hlc` 字段，但不要把它过滤掉

**问题**：Step 14 写 `OpMessage` DTO 时，如果直接把 `hlc` 字段**不加**进去（因为 Step 30 才用），等 Step 30 时要改 DTO + Jackson 配置 + 所有调用处。

**建议**：今天就加上 `hlc` 字段，用 `@JsonIgnoreProperties(ignoreUnknown = true)` + 允许 null：

```java
public record OpMessage(
    String type, String msgId, String roomId, String userId,
    String hlc,  // 今天传 null 或不传都行，Jackson 忽略
    Op op
) implements InboundMessage {}
```

字段存在但业务代码今天不读它，Step 30 时只加读取逻辑，不改数据结构。

---

### 2.7 持久化入口——今天不写，但留好注入点

**问题**：Step 40 要在每条 op 处理时异步写 MySQL。如果 Step 15 把所有 op 处理逻辑都堆在 `handleOp()` 里，Step 40 会在里面插一段 `repository.save()`，耦合度高、测试困难。

**建议**：今天在 `CollabWebSocketHandler.handleOp()` 里加一行注释占位（不写实现）：

```java
private void handleOp(OpMessage msg, WebSocketSession session) {
    // TODO Step 40: opLogService.persist(msg);   <-- 只加注释，不实现
    broadcaster.broadcast(msg.roomId(), serialize(opBroadcast), session);
}
```

Step 40 时按注释位置注入 `OpLogService`，改动范围只有这一行。

## 三、前端

### 3.1 `WSClient`——事件发布订阅，不要回调地狱

**问题**：Step 10 的 `WSClient` 如果用单个 `onMessage(handler)` 回调，Step 17 注册 join 处理、Step 18 注册 cursor 处理、Step 24 注册 op 处理，最后会变成一个 switch 散落在业务组件里：

```ts
// 反例：散落在 Room.tsx 里
ws.onMessage(msg => {
  if (msg.type === 'joined') { ... }
  else if (msg.type === 'cursor') { ... }
  else if (msg.type === 'op') { ... }
})
```

**建议**：`WSClient` 支持**按 type 订阅**：

```ts
class WSClient {
  private handlers = new Map<string, ((msg: any) => void)[]>()

  on<T>(type: string, handler: (msg: T) => void) {
    if (!this.handlers.has(type)) this.handlers.set(type, [])
    this.handlers.get(type)!.push(handler)
    return () => { /* 返回取消订阅函数 */ }
  }

  private dispatch(msg: OutboundMessage) {
    this.handlers.get(msg.type)?.forEach(h => h(msg))
  }
}
```

各个 store / 组件只订阅自己关心的类型，互不干扰：

```ts
// connectionStore.ts
ws.on('joined', msg => { ... })
ws.on('peer-joined', msg => { ... })

// cursorStore.ts
ws.on('cursor', msg => { ... })

// shapeStore.ts
ws.on('op', msg => { ... })
```

Step 47 加 `pong`、Step 31 加 snapshot，只需各自新增一个 `ws.on()`，不改已有代码。

---

### 3.2 `shapeStore`——内部结构留给 CRDT 的空间

**问题**：Step 22 建 `shapeStore`，如果直接把 shape 存成扁平对象：

```ts
// 反例
shapes: Map<string, { x: number, y: number, fill: string, ... }>
```

Step 31 改成 CRDT `ShapeReplica`（每个属性带 hlc + writerId）时，整个 Map 结构要重写，`applyOp` 逻辑也要重写。

**建议**：今天就把 `applyOp` 封装成独立函数，数据结构变化时只改这一个函数：

```ts
// store/shapeStore.ts

// 今天的简单结构（Step 22-27）
type ShapeData = { shapeId: string; shapeType: ShapeType; attrs: Attrs }

// applyOp 是唯一接触数据结构的地方
function applyOp(shapes: Map<string, ShapeData>, op: Op): Map<string, ShapeData> {
  switch (op.opType) {
    case 'create': return new Map(shapes).set(op.shapeId, { ... })
    case 'update': /* 合并 attrs */ ...
    case 'delete': { const m = new Map(shapes); m.delete(op.shapeId); return m }
  }
}
```

Step 31 只需把 `ShapeData` 换成 `ShapeReplica` 并重写 `applyOp` 的 LWW 逻辑，**Konva 渲染层（`Canvas.tsx`）一行不改**，因为渲染只关心最终的 attrs，不关心内部存储格式。

---

### 3.3 渲染层——从 store 派生显示数据，不要直接渲染 ShapeReplica

**建议**：在 `Canvas.tsx` 和 `shapeStore` 之间加一层派生选择器：

```ts
// 派生选择器：把内部格式转成 Konva 可以直接用的扁平对象
const displayShapes = useShapeStore(s =>
  [...s.shapes.values()]
    .filter(sh => !sh.tombstone)   // 排除已删除
    .map(sh => flattenAttrs(sh))   // 🟡 Step 31 后 flattenAttrs 从 ShapeReplica 提取 value
)
```

今天 `flattenAttrs` 直接返回 `sh.attrs`；Step 31 改成从每个字段的 `.value` 提取——Konva 渲染代码完全不变。

---

### 3.4 重连逻辑——今天不做，但不要把 `join` 写进组件

**问题**：Step 17 在 `Room.tsx` 的 `useEffect` 里直接写 `ws.connect() → ws.send(join)`。Step 47 加重连后，重连时也要重发 join，如果 join 逻辑散在组件里，就要在两处维护。

**建议**：把"连接并 join"封装进 `connectionStore` 的一个 action：

```ts
// store/connectionStore.ts
connect: async (roomId: string, userId: string, ...) => {
  const { wsUrl } = await fetchRoom(roomId)  // 🟡 Step 47 重连时重新拿 wsUrl
  const ws = new WSClient(wsUrl)
  ws.connect()
  ws.onOpen(() => ws.send({ type: 'join', roomId, userId, ... }))
  set({ ws, status: 'connecting' })
}
```

`Room.tsx` 只调 `connectionStore.connect(roomId)`，不关心内部细节。Step 47 的重连逻辑只在这一个函数里改。

---

### 3.5 `userId` 与 `displayName`——存 localStorage，不要每次随机生成

**问题**：Step 17 如果每次 `crypto.randomUUID()` 生成 userId，刷新页面后 userId 变了，对端会收到 `peer-left(旧id)` + `peer-joined(新id)`，历史消息里的 userId 对不上。

**建议**：
```ts
// utils/identity.ts
export function getOrCreateUserId(): string {
  return localStorage.getItem('userId') ?? (() => {
    const id = crypto.randomUUID()
    localStorage.setItem('userId', id)
    return id
  })()
}
```

`displayName` 同理存 localStorage，第一次随机生成后固定，后续可做"设置昵称"功能直接复用。

## 四、协议层

### 4.1 `type` 字段——不要用数字枚举

后端 `@JsonTypeInfo(use = NAME)` 用字符串类型名；前端 TypeScript 的 type 也用字符串字面量联合类型。

**不要**用 `0=join, 1=cursor, 2=op` 这样的整数，虽然省带宽，但：
- 调试时 WS 帧不可读
- 加新类型要协调双方改枚举，容易漏
- Jackson 多态不友好

字符串 type 的协议，后续加新消息类型只需双方加一个 case，不改现有字段。

### 4.2 保留 `msgId` 字段

今天后端不用 `msgId`，但：
- Step 48 故障切换时需要幂等去重（同一条 op 不要重复 apply）
- `error` 消息里 `refMsgId` 已经用到了
- 日志排查离不开它

**不要**因为"今天用不到"就把 `msgId` 从 DTO 里删掉。

### 4.3 op 消息的 `shapeType` 在 update/delete 里也传

开发计划 Step 27 里 delete 只有 `shapeId`，不含 `shapeType`。这在今天没问题（delete 不需要知道类型）。但 Step 31 的 CRDT 日志回放时，需要知道图形类型才能正确反序列化 attrs。

**建议**：即使是 update/delete，也把 `shapeType` 带上，后端原样转发。

## 五、配置与部署

### 5.1 用 `application-dev.yml` 隔离环境配置

今天所有地址（8080、5173、redis、mysql）都写在 `application.yml`。Step 34/39 加了 Redis/MySQL 后，如果不分环境，本地跑单节点也要起两个依赖。

**建议**：

```
backend/
  src/main/resources/
    application.yml          # 只放通用配置：server.port, cocanvas.ws-base-url
    application-dev.yml      # 本地：spring.data.redis.host=localhost, ops.broadcaster=local
    application-prod.yml     # 生产：ops.broadcaster=redis
```

启动时加 `-Dspring.profiles.active=dev`，不同环境一键切换，不用改代码。

### 5.2 `docker-compose.yml` 不要把后端硬绑死端口

目前 `docker-compose.yml` 已经把 Redis/MySQL 起好了。Step 37 要起两个后端实例时，如果 compose 里写死 `ports: "8080:8080"`，第二个实例会端口冲突。

**建议**：后端服务不加固定端口映射（用 `expose` 代替 `ports`），在命令行 `--server.port` 指定端口，或用 nginx 做反向代理统一入口（Step 37 之后）。
