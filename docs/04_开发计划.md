# 04 后续开发计划

> 更新日期: 2026-05-29
> 当前基线: 已完成多人实时协作白板、Redis Pub/Sub 多节点广播、MySQL 操作日志与历史恢复、一致性哈希路由、断线重连与 pending op 补偿, 以及产品化白板工具集。

## 1. 当前状态判断

项目已经不再处于从零搭建阶段。早期路线图中的核心链路基本完成:

- 前端: React + Vite + Zustand + Konva 白板工作台已成型, 支持 Sticky、Text、常用形状、Connector、Pen、Comment、Frame、多选、复制粘贴、Fit 和 PNG 导出。
- 后端: Spring Boot 原生 WebSocket、房间 session 管理、HLC/CRDT 属性级 LWW、Redis Pub/Sub、MySQL 操作日志、snapshot、历史恢复、一致性哈希路由和节点注册已落地。
- 基础设施: 根目录 Docker Compose 可启动 nginx、frontend、backend1、backend2、redis、mysql, 并通过 nginx 暴露统一入口。
- 当前主要风险不在"能否跑通基本 demo", 而在可靠性验证、文档一致性、回放体验、工程化质量和演示脚本。

## 2. P0: 复现与演示稳定性

目标: 确保任意合作者可以在一台干净机器上复现当前进度, 并稳定完成答辩演示。

任务:

1. 统一本地环境要求。
   - 明确后端需要 Java 21 toolchain, 当前 Gradle 配置会拒绝仅安装 Java 24 的环境。
   - README 和 docs 中的命令路径改成相对路径或当前仓库路径, 避免 `E:\Corcanvas`、`D:\Cocanvas` 等历史路径混用。
   - 为 Docker Desktop 未启动、80/6379/3307 端口占用、镜像拉取失败分别保留排错步骤。

2. 补齐一键验证脚本。
   - 新增 `run.bat verify` / `run.sh verify` 或单独 `scripts/verify.*`。
   - 至少执行: `docker compose config --quiet`、后端测试、前端 TypeScript 编译、前端 Vite 构建。
   - 验证脚本只做校验, 不清理数据卷。

3. 固化演示 checklist。
   - 两浏览器同房间实时光标。
   - 创建、编辑、拖动、删除、多选同步。
   - Pen 绘制过程 preview。
   - Connector 跟随被连接对象移动。
   - 刷新后历史恢复。
   - 停掉一个 backend 后重连并恢复。
   - 断线期间 pending op 恢复后补发。

验收标准:

- 新合作者按文档可在 30 分钟内启动项目。
- 本地验证命令结果可复制到阶段总结中。
- 演示流程每一步有明确观察点和失败时排查入口。

## 3. P1: 历史回放与数据一致性

目标: 把已有历史恢复能力升级为可演示、可测试、低风险的数据能力。

任务:

1. 拆分实时 store 与 replay store。
   - 当前历史恢复仍会进入实时 `shapeStore`。
   - 新增只读 replay store, 时间轴查看历史时不污染实时协作状态。

2. 恢复 snapshot + delta 模式。
   - 当前为了规避旧坏 snapshot, 历史恢复采用从空状态重放全部 ops。
   - 先增加 snapshot 完整性校验或版本字段, 再逐步恢复为最近 snapshot + 后续 ops。

3. 增加历史与 CRDT 测试。
   - 覆盖 create/update/delete 乱序应用。
   - 覆盖 tombstone 防复活。
   - 覆盖相同 HLC 下 writerId tie-break。
   - 覆盖跨节点远端 op 同步到本节点副本后 snapshot 不丢数据。

验收标准:

- 回放 UI 与实时协作 UI 互不影响。
- 历史接口在较多 ops 下仍能稳定返回当前状态。
- 关键 CRDT 行为有自动化测试。

## 4. P1: 前端产品化完善

目标: 让白板工具更接近可持续迭代的产品界面, 减少演示时的交互毛刺。

任务:

1. 工具栏图标化。
   - 当前 Toolbar 仍有部分字母工具入口。
   - 引入 lucide 图标或现有图标系统, 保留 hover tooltip。

2. 完善对象属性面板。
   - 统一 fill、stroke、strokeWidth、fontSize、zIndex、resolved 等属性的编辑入口。
   - 对 Connector、Comment、Frame、Pen 做专属状态展示。

3. 改进导出。
   - 当前 PNG 导出是当前视口。
   - 增加"导出全部内容范围"选项。

4. 改进批量操作协议。
   - 当前多选拖动释放后逐个对象发送 update op。
   - 设计 batch op 或事务式消息, 降低消息数量并方便历史记录聚合。

验收标准:

- 工具入口无需说明文字也能被识别。
- 复杂白板上常见操作不遮挡、不跳动、不溢出。
- 导出结果与用户选择的范围一致。

## 5. P2: 离线与可靠性增强

目标: 提升异常网络下的数据安全性。

任务:

1. pending op 持久化。
   - 当前 pending 队列只在内存中。
   - 可先落 localStorage, 后续视复杂度升级 IndexedDB。

2. 增加应用层心跳。
   - 浏览器不能主动发 WebSocket 协议层 ping。
   - 使用业务 `ping/pong` 消息维护 lastActiveTime。

3. 增加冲突与重连可观测性。
   - 事件日志区展示 pending 数、重连次数、最近恢复 ops 数。
   - 后端增加关键链路日志: route room、join、record op、publish、subscribe、snapshot。

验收标准:

- 页面刷新前未发送成功的 pending op 不会静默丢失。
- 网络断开、服务重启、节点切换时用户能看到明确状态。

## 6. P2: 后端工程化与运维

目标: 让项目更容易维护、调试和部署。

任务:

1. 整理接口文档。
   - `docs/05_接口文档.md` 仍带有早期 Step 标记, 需要升级到当前 v2 状态。
   - 明确 nginx `/api/` 会剥离前缀, 后端保留 `/api/rooms` 和 `/rooms` 兼容路径。

2. 增加数据库迁移。
   - 当前 JPA `ddl-auto=update` 适合开发, 不适合稳定演示和部署。
   - 引入 Flyway 或 Liquibase 管理 schema。

3. 增加基础监控端点。
   - 保留 `/health`。
   - 扩展 `/api/cluster/nodes` 和必要的房间调试信息。

验收标准:

- 文档描述与实际代码路由一致。
- 数据库 schema 可重复初始化。
- 节点状态、房间路由、历史恢复问题有可查询入口。

## 7. 建议推进顺序

1. 先做环境与复现文档修正, 因为它影响所有合作者继续接手。
2. 再做自动化验证脚本, 把"我这里能跑"变成可重复的项目能力。
3. 随后补历史/CRDT 测试, 保住分布式系统核心叙事。
4. 最后做前端交互打磨、导出增强和回放 UI, 用于提升演示完成度。
