# 04 后续开发计划

> 更新日期: 2026-05-29
> 当前基线: 已完成多人实时协作白板、Redis Pub/Sub 多节点广播、MySQL 操作日志与历史恢复、一致性哈希路由、断线重连与 pending op 补偿, 以及产品化白板工具集。

## 1. 当前状态判断

项目已经不再处于从零搭建阶段。早期路线图中的核心链路基本完成:

- 前端: React + Vite + Zustand + Konva 白板工作台已成型, 支持 Sticky、Text、常用形状、Connector、Pen、Comment、Frame、多选、复制粘贴、Fit 和 PNG 导出。
- 后端: Spring Boot 原生 WebSocket、房间 session 管理、HLC/CRDT 属性级 LWW、Redis Pub/Sub、MySQL 操作日志、snapshot、历史恢复、一致性哈希路由和节点注册已落地。
- 基础设施: 根目录 Docker Compose 可启动 nginx、frontend、backend1、backend2、redis、mysql, 并通过 nginx 暴露统一入口。
- 当前主要风险不在"能否跑通基本 demo", 而在可靠性验证、文档一致性、回放体验、工程化质量和演示脚本。

后续开发需要同时服务两条主线:

- 产品主线: Cocanvas 要更像一个真正能支持头脑风暴、信息整理和团队讨论的看板, 而不是只具备基础绘图能力的白板 demo。
- 分布式主线: Cocanvas 仍然是分布式系统课程作业, 需要在一致性、可用性、容错、可观测性、可扩展性和可验证性上拿得出手。

## 2. P0: 复现与演示稳定性

目标: 确保任意合作者可以在一台干净机器上复现当前进度, 并稳定完成答辩演示。

任务:

1. 统一本地环境要求。
   - 明确后端需要 Java 21 toolchain, 当前 Gradle 配置会拒绝仅安装 Java 24 的环境。
   - README 和 docs 中的命令路径改成相对路径或当前仓库路径, 避免 `E:\Corcanvas`、`D:\Cocanvas` 等历史路径混用。
   - 为 Docker Desktop 未启动、80/6379/3307 端口占用、镜像拉取失败分别保留排错步骤。

2. 补齐一键验证脚本。
   - 新增 `run.bat verify` / `run.sh verify` 或单独 `scripts/verify.*`。
   - 至少执行: `docker compose config --quiet`、后端测试、前端 TypeScript 编译、前端 Vite 构建。
   - 验证脚本只做校验, 不清理数据卷。

3. 固化演示 checklist。
   - 两浏览器同房间实时光标。
   - 创建、编辑、拖动、删除、多选同步。
   - Pen 绘制过程 preview。
   - Connector 跟随被连接对象移动。
   - 刷新后历史恢复。
   - 停掉一个 backend 后重连并恢复。
   - 断线期间 pending op 恢复后补发。

验收标准:

- 新合作者按文档可在 30 分钟内启动项目。
- 本地验证命令结果可复制到阶段总结中。
- 演示流程每一步有明确观察点和失败时排查入口。

## 3. P1: 历史回放与数据一致性

目标: 把已有历史恢复能力升级为可演示、可测试、低风险的数据能力。

任务:

1. 拆分实时 store 与 replay store。
   - 当前历史恢复仍会进入实时 `shapeStore`。
   - 新增只读 replay store, 时间轴查看历史时不污染实时协作状态。

2. 恢复 snapshot + delta 模式。
   - 当前为了规避旧坏 snapshot, 历史恢复采用从空状态重放全部 ops。
   - 先增加 snapshot 完整性校验或版本字段, 再逐步恢复为最近 snapshot + 后续 ops。

3. 增加历史与 CRDT 测试。
   - 覆盖 create/update/delete 乱序应用。
   - 覆盖 tombstone 防复活。
   - 覆盖相同 HLC 下 writerId tie-break。
   - 覆盖跨节点远端 op 同步到本节点副本后 snapshot 不丢数据。

验收标准:

- 回放 UI 与实时协作 UI 互不影响。
- 历史接口在较多 ops 下仍能稳定返回当前状态。
- 关键 CRDT 行为有自动化测试。

## 4. P1: 头脑风暴看板能力完善

目标: 让白板工具从"能画"升级为"适合头脑风暴、归纳、讨论和产出结论"的协作看板。

任务:

1. 看板对象更丰富。
   - 增加任务卡/想法卡, 支持标题、正文、标签、优先级、状态、负责人等字段。
   - 增加图片或附件占位对象, 便于放参考材料。
   - 增加投票对象或投票标记, 支持团队快速筛选想法。
   - 增加分组/泳道对象, 让 Frame 不只是视觉区域, 还能承载分类语义。

2. 头脑风暴工作流。
   - 支持从"发散"到"收敛"的流程: 快速创建便签、批量归类、投票、排序、输出总结。
   - 增加模板: SWOT、用户旅程、四象限、问题-原因-方案、Kanban、Retro。
   - 增加一键整理: 按标签、颜色、作者、状态排列卡片。
   - 增加搜索与筛选, 支持在大型白板中快速定位内容。

3. 协作讨论增强。
   - Comment 从轻量状态升级为讨论线程, 支持作者、时间、回复、resolved 状态。
   - 增加 @ 提及或轻量提醒字段, 后续可接通知系统。
   - 增加在线成员列表与跟随视角, 支持点击某个成员跳转到其当前视口附近。

4. 工具栏图标化。
   - 当前 Toolbar 仍有部分字母工具入口。
   - 引入 lucide 图标或现有图标系统, 保留 hover tooltip。

5. 完善对象属性面板。
   - 统一 fill、stroke、strokeWidth、fontSize、zIndex、resolved 等属性的编辑入口。
   - 对 Connector、Comment、Frame、Pen 做专属状态展示。

6. 改进导出和产出物。
   - 当前 PNG 导出是当前视口。
   - 增加"导出全部内容范围"选项。
   - 增加 Markdown/JSON 导出, 便于把头脑风暴结果整理成报告。

7. 改进批量操作协议。
   - 当前多选拖动释放后逐个对象发送 update op。
   - 设计 batch op 或事务式消息, 降低消息数量并方便历史记录聚合。

验收标准:

- 工具入口无需说明文字也能被识别。
- 一场 4-6 人的头脑风暴可以在 Cocanvas 中完成发散、分类、投票和总结。
- 复杂白板上常见操作不遮挡、不跳动、不溢出。
- 导出结果与用户选择的范围一致, 且能承载讨论结论。

## 5. P1: 分布式非功能能力增强

目标: 让项目在分布式课程语境下更强, 能明确展示一致性、容错、可用性、可扩展性和可观测性。

任务:

1. 可用性与故障恢复。
   - 当前 pending 队列只在内存中。
   - 可先落 localStorage, 后续视复杂度升级 IndexedDB。
   - 增加应用层心跳, 使用业务 `ping/pong` 消息维护 lastActiveTime。
   - 完整验证 backend 节点故障、Redis 短暂不可用、MySQL 重启、网络断开后的恢复行为。
   - 明确降级策略: Redis 不可用时单节点本地协作是否继续, MySQL 不可用时是否暂停写操作或进入只读/内存模式。

2. 一致性与冲突消解。
   - 为 HLC、属性级 LWW、tombstone、防乱序覆盖补齐自动化测试。
   - 增加并发编辑场景测试: 多人同时拖动、同时改文本、删除与更新乱序到达。
   - 将 CRDT 合并规则写入协议/设计文档, 让答辩时可以从算法到实现讲清楚。

3. 可观测性。
   - 事件日志区展示 pending 数、重连次数、最近恢复 ops 数。
   - 后端增加关键链路日志: route room、join、record op、publish、subscribe、snapshot。
   - 增加调试端点: 节点列表、房间路由结果、房间在线人数、当前 snapshot 版本、最近 op 序号。
   - 为演示准备可视化状态面板, 展示当前连接节点、重连状态、pending ops、历史恢复结果。

4. 可扩展性与性能。
   - 设计大房间压测脚本, 模拟多客户端 cursor/op 高频更新。
   - 对 cursor、shape-preview、drag update 做限流和合并策略。
   - 大白板渲染侧考虑虚拟化、视口裁剪或按层渲染。
   - 后端评估房间副本内存占用和 snapshot 触发策略。

5. 安全性与权限。
   - 当前任何拿到 roomId 的用户都能加入, 后续增加房间访问码或只读/编辑权限。
   - 对入站 op 做基础校验, 防止非法 shapeType、超大 payload、高频刷消息。
   - 为 WebSocket join 增加轻量 token 或签名字段, 至少能说明鉴权设计。

验收标准:

- 页面刷新前未发送成功的 pending op 不会静默丢失。
- 网络断开、服务重启、节点切换时用户能看到明确状态。
- 答辩时可以用故障注入和状态面板证明系统具备分布式容错能力。
- 关键一致性规则有测试和文档支撑。

## 6. P2: 后端工程化与运维

目标: 让项目更容易维护、调试和部署。

任务:

1. 整理接口文档。
   - `docs/05_接口文档.md` 仍带有早期 Step 标记, 需要升级到当前 v2 状态。
   - 明确 nginx `/api/` 会剥离前缀, 后端保留 `/api/rooms` 和 `/rooms` 兼容路径。

2. 增加数据库迁移。
   - 当前 JPA `ddl-auto=update` 适合开发, 不适合稳定演示和部署。
   - 引入 Flyway 或 Liquibase 管理 schema。

3. 增加基础监控端点。
   - 保留 `/health`。
   - 扩展 `/api/cluster/nodes` 和必要的房间调试信息。

验收标准:

- 文档描述与实际代码路由一致。
- 数据库 schema 可重复初始化。
- 节点状态、房间路由、历史恢复问题有可查询入口。

## 7. 建议推进顺序

1. 先做环境与复现文档修正, 因为它影响所有合作者继续接手。
2. 再做自动化验证脚本, 把"我这里能跑"变成可重复的项目能力。
3. 然后并行推进两条主线:
   - 产品主线先做卡片/标签/投票/模板, 让头脑风暴场景更完整。
   - 分布式主线先做故障恢复、CRDT 测试、状态面板和压测脚本, 让课程作业更有说服力。
4. 最后做前端交互打磨、导出增强和回放 UI, 用于提升演示完成度。
