## 第一阶段:项目骨架与最小联通

### Step 1 — 建仓库与目录骨架

在 Git 上创建一个空仓库,本地 clone 后建好顶层目录:`frontend/`、`backend/`、`protocol/`、根目录放 `README.md` 和 `.gitignore`。`README.md` 简单写项目名和"待补充";`.gitignore` 把 `node_modules/`、`build/`、`.idea/`、`.vscode/`、`.env` 这些常见忽略项加上。`protocol/` 下先放一个空 `README.md` 占位。提交一次,推到远端,确认所有成员能 clone 并切分支。**验证标准**:四个成员各自能 clone 仓库,能切到自己的分支并 push。

### Step 2 — docker-compose 起 Redis 和 MySQL

在根目录写 `docker-compose.yml`,定义两个服务:Redis 7 暴露 6379,MySQL 8 暴露 3306 并初始化数据库 `cocanvas`、root 密码设个简单的(如 `cocanvas123`)。本地跑 `docker-compose up -d`,然后用 `redis-cli ping` 和 `mysql -h 127.0.0.1 -u root -p` 分别连进去确认能通。**验证标准**:两个容器能起,本地能连。这一步当前用不到 Redis 和 MySQL,但提前起来让大家熟悉环境。

### Step 3 — 初始化后端 Spring Boot 项目

用 [start.spring.io](https://start.spring.io) 生成项目,选 Gradle Kotlin DSL、Java 21、Spring Boot 3.x,依赖只勾 **Spring Web** 和 **Spring Boot DevTools**(WebSocket、Redis、JPA 后面再加,先保持依赖干净)。下载解压到 `backend/`,在 IDEA 里打开,跑 `./gradlew bootRun`,确认 8080 端口能起来、访问 `http://localhost:8080` 返回 Whitelabel Error Page(说明 Spring MVC 在跑,只是没路由)。**验证标准**:`bootRun` 成功,浏览器能访问 8080。

### Step 4 — 后端加第一个 HTTP 端点

在 `com.cocanvas.controller` 包下建 `HealthController`,加一个 `GET /health` 返回 `{"status": "ok"}`。重启服务,浏览器访问 `http://localhost:8080/health` 看到 JSON 输出。**验证标准**:能看到 JSON 响应。这一步在确认 Spring MVC + Jackson 链路通畅。

### Step 5 — 初始化前端 Vite 项目

`cd frontend`,跑 `pnpm create vite . --template react-ts`,然后 `pnpm install`、`pnpm dev`,确认 5173 能起来、浏览器看到 Vite + React 默认页。再装两个依赖:`pnpm add zustand konva react-konva`。**验证标准**:Vite 跑起来,浏览器能看到默认欢迎页。

### Step 6 — 前端调通后端 health 端点

在前端 `src/` 下建 `network/api.ts`,写一个 `fetchHealth()` 函数用 `fetch('http://localhost:8080/health')`。改 `App.tsx`,在 `useEffect` 里调这个函数,把结果显示在页面上。这时浏览器控制台会报 CORS 错误,这是预期的——你需要回到后端,加一个 `CorsConfig` 类(`@Configuration` + `@Bean WebMvcConfigurer`),允许来源 `http://localhost:5173`。重启后端,刷新前端,看到页面显示 `status: ok`。**验证标准**:前端页面显示后端返回的 status。这一步搞通了 CORS、HTTP 通信、JSON 序列化整条链,**这是整个项目第一次真正意义上的"前后端联通"**,值得团队庆祝一下。

## 第二阶段:WebSocket 最小双向通信

### Step 7 — 后端加 WebSocket 依赖与配置类

在 `backend/build.gradle.kts` 加依赖 `org.springframework.boot:spring-boot-starter-websocket`。在 `com.cocanvas.config` 下建 `WebSocketConfig`,加 `@Configuration` + `@EnableWebSocket`,实现 `WebSocketConfigurer`,在 `registerWebSocketHandlers` 方法里先暂时什么都不注册,只让类能编译过。重启服务确认仍能正常启动。**验证标准**:服务能起,WebSocket 框架装上但还没用上。

### Step 8 — 实现一个最简单的 Echo WebSocketHandler

在 `com.cocanvas.ws` 下建 `EchoWebSocketHandler` 继承 `TextWebSocketHandler`,重写三个方法:`afterConnectionEstablished` 打印 "connected: {sessionId}";`handleTextMessage` 把收到的消息原样回发给客户端;`afterConnectionClosed` 打印 "disconnected: {sessionId}"。回到 `WebSocketConfig.registerWebSocketHandlers` 把它注册到路径 `/ws/echo`,并 `.setAllowedOrigins("*")`。重启服务。**验证标准**:服务能起,日志没报错。

### Step 9 — 用 Chrome DevTools 手动测 Echo

打开 Chrome DevTools 的 Console,输入 `const ws = new WebSocket('ws://localhost:8080/ws/echo')`,然后 `ws.onmessage = e => console.log('got:', e.data)`,再 `ws.send('hello')`。应该看到 console 打印 `got: hello`,同时后端日志打印 connected 和收到的消息。**验证标准**:浏览器收到 echo 回应,后端日志看到连接和消息。这一步证明 WebSocket 通道彻底跑通,纯手工测试,不依赖前端代码,排错最容易。

### Step 10 — 前端封装 WebSocket 连接管理

在前端 `src/network/` 下建 `websocket.ts`,导出一个 `WSClient` 类,构造函数接 URL,方法包括 `connect()`、`send(data)`、`onMessage(handler)`、`close()`。内部用一个 `WebSocket` 实例,暴露事件订阅接口。**先不做重连,先不做心跳,先不做消息队列**。在 `App.tsx` 里实例化它连 `/ws/echo`,加一个 input 框和按钮,点击按钮调 `send`,把收到的消息显示在页面上。**验证标准**:在前端页面输入文字点发送,页面上显示回显消息。这一步把第 9 步的手工测试迁移到前端代码里。

## 第三阶段:房间机制与多客户端广播

### Step 11 — 协议文档初稿

在根目录 `protocol/` 下建 `messages.md`,定义最初的几条消息格式:客户端到服务端的 `join`(字段 type、msgId、roomId、userId、displayName、color)、`cursor`(type、msgId、roomId、userId、x、y);服务端到客户端的 `joined`(type、roomId、peers 数组)、`peer-joined`(type、userId、displayName、color)、`peer-left`(type、userId)、`cursor`(转发用,type、userId、x、y)、`error`(type、code、message)。每条消息列出字段名、类型、含义、示例 JSON。**验证标准**:文档评审通过,所有成员对协议有共识。这一步看似不写代码,实则是后面所有人能并行干活的前提。

### Step 12 — 后端 HTTP 创建房间端点

在 `com.cocanvas.controller` 下建 `RoomController`,实现 `POST /api/rooms` 返回 `{ roomId, wsUrl }`。roomId 用 `UUID.randomUUID().toString()` 简化为前 8 位即可;wsUrl 当前阶段写死 `ws://localhost:8080/ws/collab`。再加 `GET /api/rooms/{roomId}` 返回 `{ roomId, exists: true, wsUrl }`(当前不存数据,exists 永远 true)。用 curl 或 Postman 测一下。**验证标准**:`POST /api/rooms` 返回带 roomId 的 JSON。

### Step 13 — 后端 RoomSessionRegistry 与 CollabWebSocketHandler 骨架

在 `com.cocanvas.ws` 下建 `RoomSessionRegistry`,加 `@Component`,内部用 `ConcurrentHashMap<String, Set<WebSocketSession>>` 存房间→会话集合(value 用 `ConcurrentHashMap.newKeySet()`),提供 `join(roomId, session)`、`leave(roomId, session)`、`broadcastInRoom(roomId, message, exceptSession)` 三个方法。再建 `CollabWebSocketHandler` 继承 `TextWebSocketHandler`,**先只实现三个回调的空壳**,只打印日志,不调 registry。在 `WebSocketConfig` 注册它到 `/ws/collab`。重启服务,用 DevTools 手测连一下 `/ws/collab` 确认能连上。**验证标准**:能连上 `/ws/collab`,后端日志显示连接建立。

### Step 14 — 后端协议 DTO 与 Jackson 多态反序列化

在 `com.cocanvas.protocol.inbound` 下建抽象基类 `InboundMessage`(普通抽象类或 sealed interface),用 `@JsonTypeInfo(use = NAME, property = "type")` 加 `@JsonSubTypes` 列出子类型。建两个 record 子类:`JoinMessage` 和 `CursorMessage`,字段对应协议文档。类似地在 `outbound` 包建 `JoinedMessage`、`PeerJoinedMessage`、`PeerLeftMessage`、`CursorBroadcastMessage`。写一个**单元测试**:用 ObjectMapper 反序列化一段 `{"type":"join", ...}` 的 JSON,断言能正确得到 `JoinMessage` 实例。**验证标准**:单元测试通过。这一步把消息协议层独立验证,跟 WS 解耦。

### Step 15 — 后端把消息流串起来

回到 `CollabWebSocketHandler`,注入 `RoomSessionRegistry` 和 `ObjectMapper`。在 `handleTextMessage` 里把 payload 反序列化为 `InboundMessage`,用 `if instanceof` 或 switch 模式匹配分发:收到 `JoinMessage` 调 `registry.join`、回发 `JoinedMessage` 给当前 session、广播 `PeerJoinedMessage` 给房间其他人;收到 `CursorMessage` 直接广播 `CursorBroadcastMessage`(把 userId、x、y 转发出去)。`afterConnectionClosed` 里要清理 registry——但这时有个问题:session 自己不知道在哪个房间。**解决**:在 `JoinMessage` 处理时,把 roomId 和 userId 存到 session.getAttributes() 里,断开时从 attributes 读出来。这一步代码量稍大,但**结构清晰**:一个文件管所有消息分发。**验证标准**:用两个 DevTools 标签页分别连 `/ws/collab`,先各自 send 一条 `join`(roomId 相同),然后 A 发 `cursor`,B 应该收到 `cursor` 广播。

### Step 16 — 前端 connectionStore 与房间页骨架

前端用 React Router(`pnpm add react-router-dom`),改造 `App.tsx` 为路由配置:`/` 是 Home 页(一个"创建房间"按钮),`/room/:roomId` 是 Room 页(暂时只显示 roomId 文字)。在 `store/` 下建 `connectionStore.ts`,用 Zustand 管理 `{ status: 'idle'|'connecting'|'connected'|'disconnected', roomId, userId, wsClient }`。Home 页的按钮点击后调 `POST /api/rooms`,拿到 roomId 后 `navigate('/room/' + roomId)`。**验证标准**:点首页按钮,跳转到 `/room/xxx` 页面,URL 里有 roomId。

### Step 17 — 前端在 Room 页连接 WS 并发 join

Room 页在 `useEffect` 里:从 URL 拿 roomId,从 connectionStore 拿 userId(没有就 `crypto.randomUUID()` 生成一个存进去),实例化 `WSClient` 连接 `/ws/collab`,连上后 send `join` 消息。订阅 `onMessage`,先简单 console.log。**验证标准**:打开 `/room/xxx` 页面,后端日志显示有人 join 进了房间;开两个标签页用同一个 roomId,后端日志显示两个用户都 join 了。

### Step 18 — 前端发送光标位置

Room 页加一个全屏的 div,监听 `onMouseMove`,每 50ms 节流(用 lodash 的 throttle 或自己写)发送一条 `cursor` 消息。在 connectionStore 或单独的 `userStore` 里维护一个 `Map<userId, {x, y, color, displayName}>` 存远端用户。处理 `cursor` 广播消息时更新这个 map。**先不渲染**,只 console.log 接到了什么,验证消息流向。**验证标准**:A 标签页移动鼠标,B 标签页 console 打印接收到的 cursor 数据,数据正确。

### Step 19 — 前端渲染远端光标

在 Room 页加一个 `CursorLayer` 组件,从 store 读远端用户 map,渲染若干个绝对定位的 div(每个是一个带颜色的小圆点 + 用户名标签),根据 x/y 设置 `transform: translate(...)`。颜色和用户名从 join 时就传好,服务端在 `peer-joined` 广播里带过来。**验证标准**:**两个浏览器窗口同一个 roomId,一个移动鼠标,另一个看到色点跟随移动**——这是第一阶段的"联通验证"目标。到这一步,第一阶段就完成了,值得 demo 一下并打个 tag。

## 第四阶段:图形 CRUD(无冲突,服务端简单转发)

### Step 20 — 协议扩展:图形操作

在 `protocol/messages.md` 加新消息:客户端到服务端的 `op`(type、msgId、roomId、userId、op 子对象,op 子对象有 opType: create/update/delete、shapeId、shapeType: rect/circle/text、attrs: {x, y, w, h, fill, ...});服务端到客户端的 `op` 广播(同结构,加上来源 userId)。先只支持矩形,圆形和文本后面再加。

### Step 21 — 后端加 OpMessage 处理

在后端 protocol 包加 `OpMessage` 和 `OpBroadcastMessage` record,定义嵌套的 `Op` 类型。在 `CollabWebSocketHandler` 的分发逻辑里加一个分支:收到 `OpMessage` 直接构造 `OpBroadcastMessage` 广播给房间其他人。**不做任何业务校验、不做冲突处理**,纯转发。**验证标准**:用 DevTools 手测,发一条 op 消息,房间内其他客户端能收到。

### Step 22 — 前端 shapeStore 与 Konva 画布

新建 `store/shapeStore.ts`,内部状态是 `Map<shapeId, Shape>`,提供 `applyOp(op)` 方法处理 create/update/delete。在 Room 页加 `Canvas` 组件,用 react-konva 的 `Stage` 和 `Layer`,从 store 读 shapes 渲染成 `Rect`。**先不能交互**,只渲染。在 store 里塞两个测试矩形,确认能显示。**验证标准**:画布上看到两个测试矩形。

### Step 23 — 前端工具栏与本地创建矩形

在 Room 页加一个 `Toolbar` 组件,有个"添加矩形"按钮。点击后:`crypto.randomUUID()` 生成 shapeId,构造 op 对象 `{ opType: 'create', shapeId, shapeType: 'rect', attrs: {x: 100, y: 100, w: 100, h: 80, fill: '#3498db'} }`,**先调本地** `shapeStore.applyOp(op)` 把矩形画出来,**再调 WSClient.send** 发出去。**验证标准**:点按钮,本地立刻看到新矩形,后端日志收到 op,但其他客户端还看不到(因为前端还没处理 op 广播)。

### Step 24 — 前端处理 op 广播消息

WSClient 的 onMessage 里加分支:收到 type=op 的消息,调 `shapeStore.applyOp(op)`。但这里要避免**自己的消息再处理一遍**——服务端转发时会带 userId,前端可以判断如果 userId 是自己就忽略(因为本地已经 applyOp 过了)。或者后端转发时直接不发回原始 sender(broadcastInRoom 的 exceptSession 参数就是干这个的)。**用后者更简洁**。**验证标准**:A 点按钮加矩形,B 立刻看到同样的矩形出现。这是图形协同的第一个里程碑。

### Step 25 — 拖动矩形:本地交互

让 Konva 的 Rect 设置 `draggable={true}`,监听 `onDragMove` 事件,每 50ms 节流。先**只更新本地** store(直接改 shape 的 x/y 后触发 store re-render)。**验证标准**:鼠标能拖动矩形,本地看到流畅跟随。

### Step 26 — 拖动矩形:同步到其他客户端

在 onDragMove 节流回调里,除了更新本地 store,再发一条 `op` 消息(opType=update,shapeId,attrs={x, y})出去。B 客户端通过 op 广播应用到自己的 store,矩形也会跟着动。**验证标准**:A 拖矩形,B 看到矩形实时跟着动。这时**两人同时拖同一个矩形**会出现"打架"现象——位置在两个客户端之间反复跳,这正是引出 CRDT 的天然契机,可以截图留作答辩素材。

### Step 27 — 删除与多图形类型

加删除功能:选中矩形(onClick 把 shapeId 存到 selectedId)、按 Delete 键触发 op(opType=delete)、本地 + 远端同步删除。再加圆形(shapeType=circle,attrs 用 radius)和文本(shapeType=text,attrs 加 text 字段,Konva 的 Text 组件)。工具栏加对应按钮。**验证标准**:能加圆形和文本,能删除,所有操作在两个客户端同步。

到这一步,**第二阶段(无冲突协同 CRUD)完成**,系统已经像个"简版 Miro"了。再打个 tag。

## 第五阶段:HLC 与 CRDT 冲突消解

### Step 28 — 后端实现 HybridLogicalClock

在 `com.cocanvas.crdt` 下建 `HybridLogicalClock` 类,核心方法 `now()` 返回当前 HLC 时间戳(物理时间 + 逻辑计数),`update(remoteHlc)` 在收到外部消息时推进本地 HLC。具体算法照 Kulkarni 等人 2014 年的 HLC 论文实现即可,核心是取 `max(localPhysical, remotePhysical, wallClock)` 并维护 logical counter。注意线程安全,用 `AtomicReference` 持有 HLC 状态或加 synchronized。写单元测试覆盖三种情况:本地推进、收到更早远端、收到更晚远端。**验证标准**:单元测试通过。

### Step 29 — 前端 HLC 实现

前端用 TypeScript 在 `src/crdt/hlc.ts` 实现同样的逻辑,API 跟后端对齐(`now()` 和 `update(remote)`)。前端不需要复杂线程安全(JS 是单线程),但要注意持久化——刷新页面后 HLC 应该接得上(可选,简化方案是不持久化,刷新就重置)。写两三个 jest 或 vitest 测试。**验证标准**:测试通过,前后端 HLC 行为一致。

### Step 30 — 协议加 HLC 字段,贯通前后端

在 `protocol/messages.md` 里给所有 `op` 消息加 `hlc` 字段(字符串形式,如 `1699999999999.0.userId123`)。前端 op 发送前先 `hlc.now()` 取一个戳塞进消息;后端收到 op 后 `hlc.update(remoteHlc)` 更新本地时钟。**这一步先打通字段,不做冲突消解**。**验证标准**:用 DevTools 看 WS 消息能看到 hlc 字段被填充,前后端日志显示 HLC 在递增。

### Step 31 — 前端 CRDT 状态:每个属性带 HLC 戳

改造 `shapeStore`,内部存的不再是简单 `Shape`,而是 `ShapeReplica`,每个属性(x, y, w, h, fill, text 等)单独带一个 hlc 戳和写入者 userId。提供 `applyOp(op)` 实现 LWW 规则:对每个被修改的属性,比较新旧 hlc,新 hlc 大才覆盖,相等时按 userId 字典序打破平局。删除用墓碑标记(tombstone)避免"删除后又被远端 update 复活"。**验证标准**:写一个本地小测试,模拟两条乱序到达的 update,验证最终状态符合预期。

### Step 32 — 验证并发冲突收敛

人工测试:A 和 B 同时拖动同一个矩形,松手后两边应该最终落到同一个位置(谁的 HLC 大谁赢)。再做一个稍复杂场景:A 改颜色为红、B 同时改颜色为蓝,松手后两边应该是同一个颜色。**验证标准**:无论怎么并发操作,两边状态最终一致。这是 CRDT 的核心承诺,演示效果很强,值得录屏。

### Step 33 — 后端持有副本(为持久化做准备)

后端 `CollabService` 内部维护一份每个房间的 `ShapeReplica` 状态(同样的 LWW 逻辑,Java 版),收到每条 op 时本地应用一次。**这一步暂时不持久化**,只是让后端也有一份"真相",为后面快照和回放做准备。**验证标准**:后端日志能打印出当前房间的图形状态,跟前端一致。

到这一步,**第三阶段(CRDT 冲突消解)完成**,核心难点已过,答辩的主菜做好了。

## 第六阶段:Redis Pub/Sub 与多节点

### Step 34 — 后端加 Redis 依赖与 Lettuce 配置

`build.gradle.kts` 加 `org.springframework.boot:spring-boot-starter-data-redis`。在 `RedisConfig` 配置 LettuceConnectionFactory(localhost:6379)、RedisTemplate(String 序列化)、RedisMessageListenerContainer。先不订阅频道,只让连接装上。重启服务确认能起,看 actuator 的 redis health 是 UP。**验证标准**:服务启动,Redis 连接正常。

### Step 35 — 抽象 OpsBroadcaster 接口

在 `com.cocanvas.pubsub` 下定义接口 `OpsBroadcaster`,方法 `broadcast(roomId, opMessage, exceptSessionId)`。建实现 `LocalOnlyBroadcaster`(`@Component @ConditionalOnProperty(name="ops.broadcaster", havingValue="local", matchIfMissing=true)`),把原 `CollabService` 里直接调 `RoomSessionRegistry.broadcastInRoom` 的逻辑搬进来。重构 `CollabService` 改为依赖 `OpsBroadcaster`。**验证标准**:重构后所有原功能仍正常。这一步是结构调整,行为不变,但为下一步替换做铺垫。

### Step 36 — 实现 RedisPubSubBroadcaster

建 `RedisPubSubBroadcaster`(`@ConditionalOnProperty(name="ops.broadcaster", havingValue="redis")`)。`broadcast` 方法把消息序列化后 publish 到 `room:{roomId}:ops` 频道。同时建一个 `RedisOpsSubscriber`,启动时通过 RedisMessageListenerContainer 订阅 `room:*:ops`(模式订阅),收到消息后反序列化,从 channel 名解析 roomId,然后调 `RoomSessionRegistry.broadcastInRoom` 在本节点广播。**注意去重**:发送节点自己也会收到自己的 publish,需要在消息里带 senderNodeId 或在 broadcast 前先本地广播一次再 publish 出去(本地不重复处理)。建议后者,简单。**验证标准**:配置 `ops.broadcaster=redis`,本地起一个节点,功能仍正常(只是消息绕了一圈 Redis)。

### Step 37 — 起两个后端实例验证多节点

`./gradlew bootRun --args='--server.port=8080 --ops.broadcaster=redis'` 和 `./gradlew bootRun --args='--server.port=8081 --ops.broadcaster=redis'` 起两个实例。前端开两个标签页,**手动**让一个连 8080、另一个连 8081(可以临时在 connectionStore 里加个调试入口指定 wsUrl,或者直接改 fetch 的目标)。两个标签页加入同一个 roomId,在 A 拖矩形,看 B 是否能收到。这一步会暴露很多隐藏 bug:序列化不一致、订阅没起来、消息丢失等,要耐心调。**验证标准**:跨节点的两个客户端能协同。这是分布式属性第一次真正体现。

### Step 38 — 节点 ID 与简单服务发现

每个节点启动时生成 nodeId(可以是 hostname + port + 随机后缀),写到 Redis 的 `nodes` set 里,定时心跳 SETEX 一个 `node:{nodeId}:alive` key(TTL 10 秒),其他节点用 `KEYS node:*:alive` 或更高效的 SCAN 拿活节点列表。建 `NodeRegistry` 组件提供 `getAliveNodes()`。**验证标准**:起两个实例,通过日志或调试端点能看到双方都知道对方存在。

## 第七阶段:持久化与时光回放

### Step 39 — MySQL 加 JPA 依赖与基础表

`build.gradle.kts` 加 `org.springframework.boot:spring-boot-starter-data-jpa` 和 `mysql:mysql-connector-j`。`application-dev.yml` 配 datasource。建实体 `OperationEntity`(字段:id BIGINT 自增主键、roomId、operationId、userId、opType、payload TEXT JSON、hlc、createdAt)和 `SnapshotEntity`(roomId、snapshotId、payload TEXT、hlc、createdAt)。建对应 `OperationRepository`、`SnapshotRepository`。启动时让 JPA 自动建表(`ddl-auto: update`)。**验证标准**:服务启动,MySQL 里能看到表已创建。

### Step 40 — 写入操作日志

在 `CollabService` 收到 op 时,异步(用 `@Async` 或直接扔到一个 `ExecutorService`)写入 `OperationRepository`。注意是**所有节点都会收到 op**(通过 Redis),如果每个节点都写,会重复 N 次——所以约定只有**首次收到 op 的节点**(原始 sender 所在的节点)负责写。简单做法:Redis 广播之前先写库,广播只负责其他节点的实时通知,不再触发持久化。**验证标准**:操作几下后,MySQL 表里能看到对应记录,数量与操作次数一致。

### Step 41 — 周期性快照

建 `SnapshotService`,加 `@Scheduled(fixedDelay = 60000)` 每分钟对每个活跃房间生成快照:从 `RoomReplica` 拿当前状态,序列化为 JSON,写到 `SnapshotEntity`。也可以基于操作数触发(每 100 个 op 一次),用计数器实现。**验证标准**:等一两分钟后,Snapshot 表能看到记录。

### Step 42 — 历史查询 HTTP 端点

`HistoryController` 加 `GET /api/rooms/{roomId}/history?at={timestamp}`,逻辑是:查找 hlc ≤ timestamp 的最近一个 snapshot,然后查 snapshot 之后到 timestamp 之间的所有 operations,组合返回 `{ snapshot, ops }`。用 curl 测一下。**验证标准**:请求能返回合理的历史数据。

### Step 43 — 前端时光回放 UI

Room 页加一个 `TimelineSlider` 组件:一个 HTML range input,值域是房间创建时间到当前时间。拖动时调 `GET /api/rooms/{roomId}/history?at=...`,把返回的 snapshot 应用到一个**独立的回放 store**(不要污染实时 store),然后依次应用增量 ops,最终在一个**只读的覆盖层 Canvas** 里渲染。提供一个"退出回放"按钮回到实时模式。**验证标准**:拖动滑块,能看到画布回到历史状态;退出后回到当前实时状态。这是答辩里另一个视觉冲击点。

## 第八阶段:一致性哈希与故障切换

### Step 44 — 一致性哈希环

在 `com.cocanvas.routing` 下建 `ConsistentHashRing`,实现 Ketama 算法(每个节点 100~200 个虚拟节点,MD5 哈希到 long,排序后存 `TreeMap`)。提供 `addNode(nodeId)`、`removeNode(nodeId)`、`routeKey(key): nodeId`。写单元测试覆盖增删节点时 key 重映射的比例(应该接近 1/N)。**验证标准**:测试通过。

### Step 45 — NodeRouter 集成

建 `NodeRouter` Bean,启动时和心跳变化时根据 `NodeRegistry.getAliveNodes()` 维护内部的 ConsistentHashRing。改造 `RoomController.createRoom` 和 `getRoom`,wsUrl 不再写死,而是用 `nodeRouter.routeKey(roomId)` 得到目标 nodeId,再查这个 nodeId 对应的 host:port(各节点把自己的访问地址也注册到 Redis 里),拼成 wsUrl 返回。**验证标准**:起两个节点,创建多个房间,响应的 wsUrl 会分散到不同节点上。

### Step 46 — 前端按 wsUrl 连接对应节点

前端创建/加入房间后,严格用 HTTP 响应的 wsUrl 连接,不再写死。**验证标准**:两个房间被分配到不同节点,两个房间的客户端各自连各自的节点,功能正常。

### Step 47 — 前端重连机制

`WSClient` 加重连逻辑:`onclose` 事件触发后,启动指数退避重连(1, 2, 4, 8, 30 秒上限,带抖动),每次重连前**重新调 `GET /api/rooms/{roomId}`** 拿最新 wsUrl(因为节点可能换了),然后连接成功后重新发 join 消息,从后端拉最新 snapshot + 增量 ops 重建状态。**验证标准**:手动断网再恢复,客户端能自动重连并状态正确。

### Step 48 — 故障切换演示

杀掉一个后端节点(`Ctrl+C` 或直接 kill 进程)。Redis 的心跳 TTL 过期后,其他节点的 `NodeRegistry` 会发现节点掉线,触发 `ConsistentHashRing.removeNode`,这个节点上的房间被重新映射到其他节点。客户端的 WS 连接断开,触发重连,重新请求 wsUrl 时拿到新节点地址,连过去,重新 join,从 MySQL 拉最新状态恢复。**整个过程对用户应该是 10 秒内完成的"卡顿一下又恢复"**。提前准备一个演示脚本:两个客户端协作中,杀掉其中一个节点,几秒后协作继续。**验证标准**:故障切换在 15 秒内完成且数据无丢失。

到这一步,**所有核心目标完成**。
