# 阶段实现总结: 可靠协作链路与前端白板产品化

## 1. 背景

本阶段在已有多人协作白板基础上继续完成两条主线:

- 后端协作链路可靠性优化: 对应 `docs/04_开发计划.md` 早期路线中的三个 P0 项。
- 前端白板产品化实现: 对应 `docs/07_前端白板产品化功能设计与计划.md` 中的 Phase 1 到 Phase 6。

完成后, Cocanvas 从基础协作 Demo 升级为一个更接近真实在线白板工具的工作台。它不仅可以多人同步编辑基础图形, 还支持便签、文本、常用形状、连接线、画笔、多选、评论、Frame、复制粘贴、导出和断线恢复等能力。

## 2. 后端 P0 可靠性优化

### 2.1 P0: 真实按节点路由 WebSocket

目标是让一致性哈希路由不仅停留在计算层面, 而是真正影响浏览器连接到哪个后端节点。

已实现内容:

- `NodeInfo` 增加 `wsPath` 字段, 节点注册时携带自己的公开 WebSocket 路径。
- `RedisNodeRegistry` 和 `LocalNodeRegistry` 注册节点时写入 `host / port / wsPath`。
- `docker-compose.yml` 为两个后端节点配置不同的公开路径:
  - `backend1`: `/ws/backend1/collab`
  - `backend2`: `/ws/backend2/collab`
- `src/nginx/nginx.conf` 增加按节点区分的 WebSocket 转发:
  - `/ws/backend1/collab` -> `backend1:8080/ws/collab`
  - `/ws/backend2/collab` -> `backend2:8080/ws/collab`
- `RoomController` 通过 `NodeRouter` 计算 roomId 对应节点, 再返回该节点的 `wsUrl`。
- 前端 `Room.tsx` 在连接前读取房间接口返回的 `wsUrl`, 并通过 `resolveWsUrl()` 转成当前页面可访问的 WebSocket 地址。

效果:

- 不同 roomId 可以按一致性哈希结果连接到不同后端节点。
- 浏览器开发者工具中可以看到 WebSocket URL 命中 `/ws/backend1/collab` 或 `/ws/backend2/collab`。
- 节点故障后, 前端重连会重新查询房间路由, 获得新的 `wsUrl`。

涉及文件:

- `docker-compose.yml`
- `src/nginx/nginx.conf`
- `src/backend/java/src/main/java/com/cocanvas/cluster/NodeInfo.java`
- `src/backend/java/src/main/java/com/cocanvas/cluster/RedisNodeRegistry.java`
- `src/backend/java/src/main/java/com/cocanvas/controller/RoomController.java`
- `src/backend/java/src/main/java/com/cocanvas/routing/NodeRouter.java`
- `src/frontend/app/src/pages/Room.tsx`

### 2.2 P0: 重连后恢复 snapshot/ops

目标是避免客户端断线或后端节点切换后继续停留在旧画布状态。

已实现内容:

- 前端连接状态变为 `connected` 后会重新发送 `join`。
- 重新 join 后调用 `getRoomHistory(roomId, Date.now())` 拉取当前时间点的历史状态。
- `applyHistoryState()` 先应用 snapshot, 再顺序应用 snapshot 之后的 ops。
- 恢复期间收到的远端 op 会先进入 `bufferedOpsRef`, 等 snapshot/ops 恢复完成后再 replay。
- 恢复完成后调用 `fitViewportToContent()` 将视口移动到当前内容附近。
- 页面事件日志会记录恢复结果, 例如 restored ops、replayed ops、flushed pending ops。

后端支撑:

- `HistoryService.history(roomId, at)` 当前采用从空状态重放截至目标时间的全部 ops, 避免历史恢复命中过去已经写坏或不完整的 snapshot。
- `RoomReplicaService.snapshot(roomId)` 提供房间当前副本快照。
- `HistoryService.snapshotActiveRooms()` 周期性保存活跃房间快照。
- `RedisRoomEventSubscriber` 收到其他节点发布的远端 op 后, 会同步应用到本节点 `RoomReplicaService`, 保证各后端节点内存副本都能生成完整 snapshot。

效果:

- A 断线期间, B 创建或移动对象; A 重连后可以恢复到服务端最新画布状态。
- 后端节点切换后, 前端重新查询路由并恢复当前房间状态。

涉及文件:

- `src/frontend/app/src/pages/Room.tsx`
- `src/frontend/app/src/network/api.ts`
- `src/backend/java/src/main/java/com/cocanvas/controller/HistoryController.java`
- `src/backend/java/src/main/java/com/cocanvas/service/HistoryService.java`
- `src/backend/java/src/main/java/com/cocanvas/service/RoomReplicaService.java`

### 2.3 P0: 断线期间操作处理策略

本阶段采用早期计划中的方案 B: pending op 队列。

目标是避免用户断线时本地看似操作成功, 但服务端没有记录, 刷新或重连后静默丢失。

已实现内容:

- 所有本地操作在发送前都会通过 `stampOp()` 补齐稳定 `opId`、`hlc` 和 `writerId`。
- `sendShapeOp()` 在 WebSocket 未连接时不丢弃操作, 而是将 stamped op 放入 `pendingOpsRef`。
- update 类型 pending op 会按同一 shape 合并 attrs, 避免断线拖动产生过多重复 update。
- 重连成功后流程为:
  1. join 房间。
  2. 拉取 snapshot + ops 恢复服务端状态。
  3. replay 恢复期间缓存的远端 op。
  4. flush pending ops 到服务端。
- `HistoryService.recordOperation()` 按 `opId` 幂等去重, pending op 重放时同一个 op 不会重复写入操作日志。

效果:

- 断线期间创建、编辑、移动、删除对象会进入 pending 队列。
- 重连并完成恢复后 pending 操作会自动补发。
- 服务端操作日志通过 `opId` 去重, 降低重复重放风险。

涉及文件:

- `src/frontend/app/src/pages/Room.tsx`
- `src/backend/java/src/main/java/com/cocanvas/service/HistoryService.java`
- `src/backend/java/src/main/java/com/cocanvas/persistence/entity/OperationLogEntity.java`
- `src/backend/java/src/main/java/com/cocanvas/persistence/repository/OperationLogRepository.java`
- `src/frontend/app/src/types/protocol.ts`

## 5. 关键问题修复与交互优化

本阶段除按计划实现功能外, 还修复了多个实际使用中暴露的问题:

- Select 模式拖动物体释放后位置不正确:
  - 改为手动基于 pointer 和 canvas 坐标计算拖动位置。
- 拖动物体时远端只能看到释放后的瞬移:
  - 新增 `shape-preview` 通道, 拖动过程中广播临时位置。
- 连接线不跟随拖动中的物体:
  - 连接线端点改为读取当前渲染位置, 包括本地 dragState 和远端 preview。
- 连接线必须命中不可见锚点:
  - 改为命中目标对象本体或附近即可连接, 自动选择最近边框锚点。
- 创建房间后空白页, React 报 `Maximum update depth exceeded`:
  - 修复 Zustand selector 返回新数组导致的无限更新, 改用稳定 selector + `useMemo` 派生 selectedShapes。
- 左侧工具栏过长:
  - 新增滚动区域和折叠模式。
- 画笔实时性不足:
  - 绘制过程中也通过 `shape-preview` 广播临时路径。
- 短时间刷新物品存在, 过一段时间刷新后物品丢失:
  - 根因是 Redis Pub/Sub 订阅端过去只把远端 op 广播给本节点客户端, 没有同步到本节点 `RoomReplicaService` 内存副本。
  - 某些后端节点随后会基于空/旧内存副本写出不完整 snapshot, 历史恢复命中该 snapshot 后早期对象消失。
  - 修复方式一: 订阅端收到远端 op 后调用 `RoomReplicaService.applyRemote()`, 同步本节点副本, 但不重复写 MySQL 操作日志。
  - 修复方式二: 历史恢复暂时改为从空状态完整重放截至目标时间的全部 ops, 避免已有坏 snapshot 继续影响刷新恢复。

## 7. 当前边界与后续建议

已完成本阶段目标, 但仍有一些后续可优化点:

- 历史回放目前仍会把历史状态应用到实时 store, 后续可以拆出独立 replay store。
- 为了规避已有坏 snapshot, 当前历史恢复优先从操作日志完整重放到目标时间; 数据量变大后可在确保跨节点 snapshot 正确的基础上恢复为 snapshot + 增量 ops。
- 多选拖动目前释放时对每个对象发送 update op, 后续可设计 batch op 降低消息数量。
- pending op 队列在前端内存中, 刷新页面会丢失未 flush 的 pending op; 后续可以落到 localStorage 或 IndexedDB。
- PNG 导出目前导出当前 canvas 视口, 后续可支持导出完整内容范围。
- Comment 目前是轻量版本, 后续可以增加作者、时间、线程回复和 @ 提及。
- Frame 目前是视觉分区, 后续可以支持内部对象归属、演示模式和一键跳转。
- Toolbar 当前使用字母作为图标, 后续可以引入 lucide 图标提升可识别性。

## 8. 结论

本阶段完成了 Cocanvas 从基础协作画布到产品化白板工作台的重要升级:

- 后端链路更可信: WebSocket 真实按节点路由, 重连后恢复 snapshot/ops, 断线操作进入 pending 队列并在恢复后补发。
- 前端能力更完整: 支持便签、文本、形状库、连接线、画笔、多选、评论、Frame、快捷键、Fit 和 PNG 导出。
- 协作体验更连续: 拖动和画笔都具备实时 preview, 连接线能在对象移动中实时跟随。

这使项目在答辩或演示中可以讲清楚完整故事: 多人协作、分布式路由、故障恢复、离线操作补偿和真实白板交互体验。
