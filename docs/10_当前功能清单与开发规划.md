# 10 当前功能清单与开发规划

> 更新日期: 2026-05-29  
> 当前阶段: 产品化协作白板 + 房间管理主线已成型, 已加入会议流程和场景化讨论模板, 分布式协作链路可运行, 仍需补齐权限执行、多人语音、回放隔离和演示验证。

## 1. 当前产品定位

Cocanvas 当前已经从基础同步 Demo 演进为一个面向头脑风暴、会议协作和产品讨论的在线白板工作台。

当前产品主线可以概括为:

- 用户先进入房间控制台, 管理房间、身份和进入方式。
- 进入房间后, 多人可以在同一画布中创建、编辑、拖动、连接、分组和讨论内容。
- 主持人可以按“准备议题 -> 发散 -> 归类 -> 投票 -> 决策 -> 行动项”的流程推进讨论, 并一键插入当前阶段适合的讨论图。
- 白板内容通过 WebSocket 实时同步, 并通过操作日志与历史恢复保证刷新后可恢复。
- 系统保留分布式课程项目特征, 包含多后端节点、Redis Pub/Sub、一致性哈希、HLC/CRDT、MySQL 历史日志等能力。

已知限制:

- 密码当前用于进入校验, 仍属于开发级实现, 后续应改为带盐哈希、访问 token 或 session。

### 2.2 用户身份

已完成:

- 浏览器本地用户 ID。
- 自定义显示名称。
- 中文输入法友好输入:
  - 输入过程中使用本地草稿。
  - 失焦或 Enter 后提交。
  - 避免输入法组词时频繁写入全局状态。
- 推荐协作颜色。

- 用户身份保存到 localStorage。
- 房间控制台可编辑身份。

- WebSocket join 时使用当前显示名称和颜色。

已知限制:

- 当前身份是本地匿名身份, 没有账号系统。

### 2.3 房间进入与连接

已完成:

- 房间不存在或归档时阻止进入画布。
- 密码错误时停留在密码输入页。
- 密码输入只在提交时校验, 不会每输入一个字符就请求。
- 密码正确后保存真实 `wsUrl`, WebSocket 连接不再做无密码二次查询。
- WebSocket 状态展示:
  - connecting
  - connected
  - closed
  - error
- Peers 数量展示。
- 断线后自动重连。
- 重连后重新 join。
- 重连后历史恢复。
- 离线期间 pending op 队列, 重连后补发。

已知限制:

- pending op 目前主要在内存中, 页面完全关闭后可能丢失。
- WebSocket 鉴权还没有 token 化, 后续应把房间密码或授权结果换成短期 join token。

## 4. 当前验证方式

最近验证通过:

```powershell
docker compose exec -T frontend pnpm lint
docker compose exec -T frontend pnpm build
docker compose build backend1 backend2
```

已验证过的关键行为:

- 带 `Origin: http://localhost:8088` 的创建房间请求不再被 CORS 拦截。
- 密码房间错误密码不会进入画布。
- 密码房间正确密码可以拿到真实 `wsUrl`。
- WebSocket 协作通道可返回 `joined`。
- 远端光标使用画布世界坐标计算。

宿主机限制:

- 本机直接运行 `gradlew test` 需要 Java 21 toolchain。
- 当前可通过 Docker 镜像构建验证后端 Java 编译。

## 5. 当前主要问题与风险

### 5.1 产品风险

- 房间权限只是配置项, 还没有真正限制编辑行为。
- 语音入口只是本地麦克风开关, 还不是会议语音。
- 房间控制台仍可继续提升为更完整的团队空间:
  - 最近访问。
  - 收藏。
  - 房间所有者。
  - 成员列表。
  - 邀请链接。
  - 角色权限。
- 右侧 Inspector 还有继续产品化空间。

### 5.2 协作风险

- pending op 未持久化。
- profile 修改没有独立实时消息。
- WebSocket join 缺少 token。
- 权限没有后端强校验。

### 5.3 数据风险

- 没有数据库迁移工具。
- 密码哈希仍偏开发级。
- 历史恢复和实时状态隔离不足。
- 大房间数据量增长后, 完整 replay 会有性能压力。

### 5.4 性能风险

- Konva 大画布尚未虚拟化。
- 多选拖动会产生多条 update。
- 大量 cursor/preview 时需要更严格合并与采样。
- 前端 bundle 已超过 500 kB, 后续可考虑代码分割。

## 6. 后续开发规划

### P0: 把权限和房间安全做实

目标: 房间管理不只是 UI 字段, 而是真正控制用户行为。

任务:

- 创建房间后生成短期 join token。
- WebSocket join 校验 token。
- `permissionMode=view` 时前端禁用编辑工具。
- 后端对 op 做权限校验。
- 房间密码改为带盐哈希。
- 增加房间 owner / host 字段。
- 增加成员角色:
  - owner
  - editor
  - commenter
  - viewer

验收:

- 只读用户无法通过前端或直接 WebSocket 写入 op。
- 密码房间没有授权时无法拿到可用 wsUrl。

### P0: 稳定协作演示

目标: 双窗口、双节点、断线恢复演示稳定。

任务:

- 增加 `run.bat verify` / `run.sh verify`。
- 增加 WebSocket 连接诊断面板:
  - 当前 wsUrl。
  - 当前连接节点。
  - reconnect attempts。
  - pending ops。
  - last restored ops。
- 增加一键创建演示房间。
- 固化演示 checklist。

验收:

- 两个浏览器进入同一房间后, `WS: connected`, `Peers: 1`。
- 创建、拖动、删除、分组在 1 秒内同步。
- 停掉一个 backend 后可恢复连接。
