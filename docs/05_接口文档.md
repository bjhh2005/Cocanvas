## 1. 通用约定

### 1.1 基础地址
| 用途 | 地址 |
|---|---|
| HTTP API | `http://localhost:8080` |
| WebSocket | `ws://localhost:8080` |
| 前端开发服务器 | `http://localhost:5173` |

### 1.2 数据格式
- 编码：JSON，UTF-8
- 字段命名：**camelCase**
- 所有 ID（roomId、userId、shapeId、msgId、nodeId、snapshotId）：**字符串 UUID**（roomId 可截前 8 位简化，见 Step 12）
- 时间：毫秒时间戳 `Long`
- 颜色：CSS 十六进制字符串，如 `#3498db`

### 1.3 HLC 时间戳（🟡 Step 30 引入）
格式：`物理ms.逻辑计数.nodeId`，示例：`1716123456789.0.node-a1b2`

### 1.4 公共消息字段
所有 WebSocket 消息（无论方向）都至少包含 `type` 字段；客户端→服务端的消息额外要求 `msgId`，方便日志追踪与未来幂等。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `type` | string | ✅ | 消息类型，Jackson 多态反序列化 discriminator |
| `msgId` | string(UUID) | C→S ✅ | 客户端生成，每条消息唯一 |
| `roomId` | string | C→S ✅ | 冗余字段，便于日志与后续跨节点路由 |
| `userId` | string | C→S ✅ | 当前用户 |
| `ts` | long | 可选 | 客户端发送时刻毫秒 |
| `hlc` | string | 🟡 | Step 30 起 op 消息必填 |

### 1.5 CORS
后端允许来源：`http://localhost:5173`（Step 6 已配置）。

### 1.6 广播策略
所有 WS 广播一律使用 `broadcastInRoom(roomId, msg, exceptSession = sender)`——**原始发送者不收回自己的消息**。前端约定：本地先 `apply`，再 `send`（Step 24）。

创建房间。

**请求体**：空对象 `{}`（今天无任何参数）

**响应 200**
```json
{
  "roomId": "a1b2c3d4",

```

| 字段 | 类型 | 说明 |
|---|---|---|
| `roomId` | string | UUID 前 8 位 |
| `wsUrl` | string | 客户端应连接的 WS 地址 ✅ 今天写死；🟡 Step 45 起由一致性哈希决定 |

### 2.3 `GET /api/rooms/{roomId}` ✅ Step 12
查询房间是否存在 + 应连接的 wsUrl。

**响应 200**
```json
{
  "roomId": "a1b2c3d4",
  "exists": true,
  "wsUrl": "ws://localhost:8080/ws/collab"
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `exists` | bool | ✅ 今天恒为 `true`（无房间存在性校验）；🟡 Step 39 后基于 MySQL 真实判断 |
| `wsUrl` | string | 同上；🟡 Step 47 重连时调用本接口拿最新地址 |

**响应 404**（🟡 Step 39 后）
```json
{ "code": "room_not_found", "message": "room a1b2c3d4 not found" }
```

## 3. WebSocket 接口

### 3.1 连接

| 项 | 值 | 阶段 |
|---|---|---|
| 业务路径 | `/ws/collab` | ✅ Step 13 |
| 调试 echo | `/ws/echo` | ✅ Step 8（仅开发期，生产应下线） |
| 子协议 | 无 | — |
| 鉴权 | 无；userId 前端 `crypto.randomUUID()` 生成 | ✅ |
| 帧类型 | 仅 TEXT（JSON） | ✅ |
| `setAllowedOrigins` | `*` | ✅ Step 8 |

**连接生命周期**：
1. 客户端 `new WebSocket(wsUrl)` 建立连接
2. 服务端 `afterConnectionEstablished` 仅打印日志，**不**把 session 放进任何房间（房间归属由后续 `join` 消息决定）
3. 客户端**必须**发送的第一条业务消息是 `join`
4. 服务端处理 `join` 时：把 `roomId`、`userId`、`displayName`、`color` 写入 `session.getAttributes()`，调用 `RoomSessionRegistry.join(roomId, session)`
5. 断开时 `afterConnectionClosed` 从 `attributes` 读出 roomId/userId，调用 `registry.leave`，广播 `peer-left`

**Session 属性约定（后端 `session.getAttributes()`）**

| key | 类型 | 值 |
|---|---|---|
| `roomId` | String | join 时写入 |
| `userId` | String | join 时写入 |
| `displayName` | String | join 时写入 |
| `color` | String | join 时写入 |
| `nodeId` | String | 🟡 Step 38 |

### 3.2 客户端 → 服务端 消息

#### 3.2.1 `join` ✅ Step 11
进入房间，声明身份。

```json
{
  "type": "join",
  "msgId": "11111111-...-...",
  "roomId": "a1b2c3d4",
  "userId": "u-xxxx",
  "displayName": "Alice",
  "color": "#e74c3c"
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `displayName` | string | ✅ | 1~20 字符 |
| `color` | string | ✅ | CSS 十六进制，前端建议从一组预设色里随机选 |

服务端处理：
1. 写 session attributes
2. `registry.join(roomId, session)`
3. **回发**当前 session 一条 `joined`（含已在房间内的 peers）
4. **广播**给房间其他人一条 `peer-joined`

---

#### 3.2.2 `cursor` ✅ Step 11
上报鼠标位置（高频，**前端 50ms 节流**）。

```json
{
  "type": "cursor",
  "msgId": "...",
  "roomId": "a1b2c3d4",
  "userId": "u-xxxx",
  "x": 312.5,
  "y": 488.0
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `x`, `y` | number | 画布坐标系（左上角原点，单位 px，Stage 内坐标） |

服务端处理：原样构造 `cursor` 广播消息（仅去掉 `msgId`、`roomId`，保留 `userId`、`x`、`y`），`exceptSession=sender` 广播。

---

#### 3.2.3 `op` ✅ Step 20（拖动节流：≥50ms / 条）
图形操作。

```json
{
  "type": "op",
  "msgId": "...",
  "roomId": "a1b2c3d4",
  "userId": "u-xxxx",
  "hlc": "1716123456789.0.u-xxxx",
  "op": {
    "opType": "create",
    "shapeId": "s-abcd",
    "shapeType": "rect",
    "attrs": { "x": 100, "y": 120, "w": 100, "h": 80, "fill": "#3498db" }
  }
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `hlc` | string | 🟡 Step 30 起必填；今天忽略 | |
| `op.opType` | enum | ✅ | `create` / `update` / `delete` |
| `op.shapeId` | string | ✅ | 前端 `crypto.randomUUID()` |
| `op.shapeType` | enum | create 必填 | `rect` / `circle` / `text` |
| `op.attrs` | object | create 必填；update 为差量；delete 可省 | 见 §4.1 |

服务端处理（✅ Step 21）：**不做业务校验、不做冲突处理**，纯转发为 `op` 广播。

---

#### 3.2.4 `ping` 🟡 Step 47
心跳。客户端 30s 一次。

```json
{ "type": "ping", "msgId": "...", "ts": 1716123456789 }
```

服务端回 `pong`（不广播）。

### 3.4 典型时序

#### 时序 1：A 单人进房 ✅
```
A: connect ws://.../ws/collab
A → S: {"type":"join","roomId":"R","userId":"A",...}
S → A: {"type":"joined","roomId":"R","you":{...A},"peers":[]}
```

#### 时序 2：B 后到 ✅
```
B: connect
B → S: join
S → B: joined  (peers=[A])
S → A: peer-joined(B)
```

#### 时序 3：A 移动鼠标 ✅
```
A → S: {"type":"cursor","x":312,"y":488,...}     (50ms 节流)
S → B: {"type":"cursor","userId":"A","x":312,"y":488}
```

#### 时序 4：A 创建矩形 ✅
```
A 本地: shapeStore.applyOp(create)   // 先本地
A → S: {"type":"op","op":{"opType":"create",...}}
S → B: {"type":"op","fromUserId":"A","op":{...}}
B 本地: shapeStore.applyOp(create)
```

#### 时序 5：A 拖动矩形 ✅
```
A 本地: shape.x = newX   // 立即响应
A → S: op(update, attrs:{x,y})        每 50ms 一条
S → B: op 广播
B 本地: applyOp
```

> 两人同时拖同一矩形时会出现"位置跳变"——这是 Step 26 的预期现象，是引出 CRDT 的契机。

#### 时序 6：A 关闭页面 ✅
```
A: ws.close()
S: afterConnectionClosed → registry.leave → broadcast
S → B: {"type":"peer-left","userId":"A"}
```

#### 时序 7：跨节点协同 🟡 Step 37
```
A 连 node-1, B 连 node-2，同一房间 R
A → node-1: op
node-1 本地: registry.broadcastInRoom(R, op, exceptA)   // node-1 上无 B
node-1 → Redis: PUBLISH room:R:ops <op>
node-2 ← Redis: 收到，registry.broadcastInRoom(R, op, null)
node-2 → B: op
```

#### 时序 8：故障切换 🟡 Step 48
```
node-1 挂掉
node-2.NodeRegistry: TTL 过期，从环上摘除 node-1
A 的 ws onclose → 指数退避重连
A → GET /api/rooms/R → 拿到新 wsUrl(node-2)
A connect node-2 → send join
node-2 → A: snapshot (从 MySQL/内存恢复)
A 应用 snapshot，继续协作
```

## 4. 数据结构

### 4.1 Shape `attrs` 字段表

**rect** ✅ Step 23
| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `x`, `y` | number | ✅ | 左上角坐标 |
| `w`, `h` | number | ✅ | 宽高 |
| `fill` | string | ✅ | 填充色 |
| `stroke` | string | 可选 | 描边色 |
| `strokeWidth` | number | 可选 | 描边宽度 |

**circle** ✅ Step 27
| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `x`, `y` | number | ✅ | 圆心坐标 |
| `radius` | number | ✅ | 半径 |
| `fill` | string | ✅ | |
| `stroke`, `strokeWidth` | — | 可选 | |

**text** ✅ Step 27
| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `x`, `y` | number | ✅ | 文本左上角 |
| `text` | string | ✅ | 文本内容 |
| `fontSize` | number | ✅ | 字号 |
| `fill` | string | ✅ | 文字颜色 |
| `fontFamily` | string | 可选 | 默认 `sans-serif` |

### 4.2 Op 语义
- **create**：`attrs` 必须是**完整**初始属性
- **update**：`attrs` 是**差量**——只列变化的字段，其它字段保持不变
- **delete**：`attrs` 可为 `null` 或省略

### 4.3 ShapeReplica（🟡 Step 31，CRDT 内部结构）
前端 / 后端 `shapeStore` 内部存储结构，每个属性单独带 hlc + writerId：

```json
{
  "shapeId": "s-abcd",
  "shapeType": "rect",
  "tombstone": false,
  "fields": {
    "x":    { "value": 220, "hlc": "...", "writerId": "u-xxxx" },
    "y":    { "value": 180, "hlc": "...", "writerId": "u-xxxx" },
    "fill": { "value": "#3498db", "hlc": "...", "writerId": "u-yyyy" }
  }
}
```

LWW 规则：新 hlc > 旧 hlc 才覆盖；hlc 相等时按 `writerId` 字典序打破平局。

  "hlc": "1716120000000.0.node-a1b2",
  "shapes": [ /* Shape[] (扁平化后的当前状态) */ ],

## 5. 错误码表

| code | 触发场景 | HTTP / WS | 阶段 |
|---|---|---|---|
| `room_not_found` | 房间不存在 | 404 / WS error | 🟡 Step 39 |
| `invalid_message` | JSON 解析失败、字段缺失、字段类型错 | 400 / WS error | ✅ Step 11 |
| `unknown_type` | `type` 字段未识别 | WS error | ✅ Step 14 |
| `not_joined` | 在 join 之前发送了 cursor/op | WS error | ✅ Step 15 |
| `room_full` | 房间人数超限 | 409 / WS error | 🟡 预留 |
| `rate_limited` | 客户端发送过快 | WS error | 🟡 预留 |
| `internal_error` | 服务端异常 | 500 / WS error | ✅ |

## 6. 节流与频次约定（前端实现约束）

| 消息 | 最小间隔 | 说明 |
|---|---|---|
| `cursor` | ≥ 50ms / 条 | 鼠标移动用 lodash `throttle` 或自写 |
| `op` (update / 拖动) | ≥ 50ms / 条 | `onDragMove` 节流 |
| `op` (create / delete) | 无节流 | 用户主动触发，频次低 |
| `ping` | 30s / 次 🟡 | Step 47 |

> 后端今天不做速率限制；若前端实现违反约定打爆服务端，由前端兜底修。
