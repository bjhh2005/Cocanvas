# 功能清单

> 更新日期: 2026-05-31  
> 说明: 本文档描述当前代码已经具备的能力和仍然存在的限制。旧编号文档保留为历史设计材料, 当前功能状态以本文档为准。

## 1. 产品定位

Cocanvas 当前是一个多人实时协作白板工作台, 同时用于展示分布式协作系统的关键机制。

核心使用路径:

1. 用户进入房间控制台。
2. 创建或加入房间。
3. 在房间中多人协作编辑白板。
4. 系统通过 WebSocket 实时同步操作。
5. 断线或节点故障后, 客户端重连并通过历史恢复重建状态。

## 2. 房间能力

已完成:

- 房间列表。
- 创建房间。
- 自定义房间号。
- 自定义房间名称。
- 房间访问模式字段:
  - `link`
  - `password`
  - `private`
- 默认权限字段:
  - `edit`
  - `comment`
  - `view`
- 房间密码进入校验。
- REST 创建/查询房间时返回短期 `joinToken`。
- 登录账号创建房间时自动成为该房间 `owner`。
- 房间成员支持 `owner` / `edit` / `comment` / `view` 角色, 成员角色优先于房间默认权限。
- `owner` 可按用户名添加成员、调整角色和移除成员; 后端会保留至少一个 owner。
- 更新房间设置。
- 归档房间。
- 查询房间是否存在。
- 返回当前房间应连接的 WebSocket 地址。
- 会议语音入口开关字段。
- 后端基于 `permissionMode` 做 op 写入校验:
  - `view` 禁止写入 op。
  - `comment` 只允许 comment 对象相关 op。
- 房间元数据本地缓存, 创建、更新、归档时失效。

已知限制:

- `private` 已有成员权限模型, 但仍未强制成“仅成员可访问”的完整私有房间策略。
- 密码实现仍偏开发级, 后续应改为带盐哈希。
- 房间元数据缓存目前是单节点 10 秒内存缓存, 还不是 Redis/Caffeine 的可观测缓存。

## 5. 实时协作

已完成:

- Spring WebSocket 原生协议。
- 房间 join。
- `joined` 回执。
- `peer-joined` / `peer-left`。
- 远端光标广播。
- shape operation 广播。
- shape preview 广播。
- cursor / shape-preview 作为 transient 低优先级消息处理。
- 多用户瞬态交互优化:
  - 前端按 animation frame 合并远端 cursor / shape-preview, 避免每条瞬态消息触发一次渲染。
  - 本地 cursor 发送增加细微移动过滤, 减少多人房间中的无效坐标广播。
  - 后端发送队列对 cursor / shape-preview 做 latest-wins 合并, 慢客户端不会积压过期瞬态消息。
  - Redis 订阅节点在本地无房间会话时跳过 cursor / shape-preview 转发。
- 每个 WebSocket session 有独立发送队列。
- 慢客户端队列积压时:
  - cursor / preview 超限丢弃。
  - op 等可靠消息超限断开连接, 交由客户端重连和历史恢复。
- 客户端本地先应用 op；服务端收到后按"内存副本 -> ack -> 广播"的关键路径处理，MySQL 持久化改为**批量写后置（write-behind）**，不阻塞 ack。
- 发送方收到 `op-ack` 后从 pending outbox 删除。
- 收到 `op-ack` 时使用服务端合并后的 HLC 校准本地版本。
- WebSocket 断线重连。
- 重连后重新 join。
- 重连后恢复历史状态。
- 断线期间 pending op 补发。
- WebSocket join 校验 REST 返回的短期 `joinToken`。

已知限制:

- 慢客户端背压阈值当前是固定配置, 尚未做运行时指标暴露和动态调整。
- cursor / shape-preview 已低优先级化, 但仍会跨节点发布到 Redis transient 分片频道。

## 6. 分布式能力

已完成:

- Docker Compose 启动双后端节点:
  - `backend1`
  - `backend2`
- Nginx 统一入口。
- 节点专属 WebSocket 路径:
  - `/ws/backend1/collab`
  - `/ws/backend2/collab`
- Redis 节点心跳。
- Redis `cocanvas:nodes` Set 维护节点索引。
- 节点 TTL key 表示存活状态。
- 一致性哈希按 `roomId` 选择 owner 节点。
- 一致性哈希环缓存。
- Redis Pub/Sub 跨节点广播。
- Pub/Sub 频道按 hash 分为 64 个:
  - `cocanvas:room-events:0`
  - ...
  - `cocanvas:room-events:63`
- cursor / shape-preview 使用独立 transient 分片频道:
  - `cocanvas:room-transient-events:0`
  - ...
  - `cocanvas:room-transient-events:63`
- Redis 订阅端将远端 op 应用到本节点内存副本。
- 只有一致性哈希 owner 节点写 room snapshot。

已知限制:

- 节点列表变化后, 已连接 WebSocket 需要依赖断线重连迁移。
- 没有专门的节点负载指标和动态权重。
- Redis Pub/Sub 不提供消息持久化, op 可靠性依赖 MySQL 操作日志和客户端重放。

## 7. 一致性与历史恢复

已完成:

- HLC 时间戳。
- 属性级 LWW/CRDT 合并。
- delete tombstone。
- opId 幂等去重。
- MySQL 操作日志（批量写后置：缓冲队列 + 满 200 条 / 每 200ms 触发一次 `saveAll` 批量插入，配合 Hibernate JDBC batch 与 MySQL `rewriteBatchedStatements`）。
- snapshot / operation_log 的 payload 字段为 `LONGTEXT`（非 `TEXT`），避免大白板快照 JSON 超过 64KB 被静默截断。
- MySQL snapshot。
- versioned snapshot:
  - 每个属性保留 `value`
  - 每个属性保留 `hlc`
  - 每个属性保留 `writerId`
- 历史恢复优先读取目标时间之前最新 snapshot。
- 只回放 snapshot 之后的增量 op。
- 手动历史回放实际渲染到画布, 顶部横幅标记"历史回放模式"。
- 重连恢复仍会显式替换为 snapshot + 增量状态。
- snapshot 后自动裁剪已覆盖的 op 日志 (保留倒数第二个快照以后的 op, 防止 operation_log 无限增长)。
- `GET /api/rooms/{roomId}/history/anchors` 返回快照时间戳列表, 用于前端时间轴刻度渲染。
- 历史时间轴 UI: 可拖动 range 滑块, 蓝色刻度线标示快照位置, "在画布预览" / "返回实时" 双按钮。

可靠性权衡:

- 持久化由"同步写穿透"改为"批量写后置"后，实时性显著提升（ack 延迟下降约 78%），但单条 op 的强持久化保证减弱。
- 兜底机制：实时状态以内存副本 + 跨节点广播为准；60s 周期快照直接从内存副本生成，个别 op 落库失败也能由快照恢复最终状态。

已知限制:

- 缺少 snapshot 完整性校验字段。
- 批量写后置在节点非优雅退出时，缓冲区中尚未 flush 的 op 可能丢失（依赖周期快照兜底）。

### 2026-05-30 性能可观测性压测结果

执行命令: `node scripts/distributed-check.mjs`  
环境: Docker Compose 双后端节点 (backend1 + backend2), Nginx 统一入口, MySQL + Redis

#### 1. 房间元数据缓存命中率 (Caffeine)

脚本对同一房间连续发起 300 次 `GET /api/rooms/{id}`, 前后对比 `/api/cluster/cache-stats`:

| 指标 | 数值 | 说明 |
|---|---|---|
| route_get 请求次数 | 300 | 全部命中同一房间 |
| 平均延迟 | 5.14 ms | 含网络 + Nginx 代理开销 |
| p50 延迟 | 4.77 ms | |
| p95 延迟 | 7.48 ms | |
| 缓存命中 | +149 次 | 共 150 次请求, 仅首次查数据库 |
| 缓存未命中 | +1 次 | 首次 cold miss, 后续全走内存 |
| 最终命中率 | 99.0% | 10s TTL 内反复访问无 DB 查询 |
| DB 加载次数 | 0 | 本轮 TTL 窗口内未过期 |

**结论**: Caffeine 缓存将 300 次路由查询中的 299 次拦截在内存层。对比升级前每次均需 MySQL 查询的实现, 在高并发房间访问场景下显著降低数据库压力。

#### 2. WebSocket 发送队列指标 (顺序可靠消息)

两个跨节点 peer 建立连接, peer1 顺序发送 300 条 op 消息, 前后对比 `/api/cluster/queue-stats`:

| 指标 | 数值 | 说明 |
|---|---|---|
| 总耗时 | 3809.66 ms | 300 条 op 顺序确认 |
| 吞吐量 | 78.7 ops/s | 含持久化 + HLC + 广播全链路 |
| ack 平均延迟 | 12.67 ms | 含 MySQL 写入 |
| ack p50 | 11.75 ms | |
| ack p95 | 19.12 ms | |
| transient 丢弃数 | 0 | 可靠消息不触发丢弃逻辑 |
| overload 断开数 | 0 | 队列未超过 MAX_QUEUE_SIZE=256 |
| 跨节点首条延迟 | 513.97 ms | 含节点路由 + Redis pub/sub 传播 |

**结论**: 顺序负载下队列始终健康 (`queued=0` 稳定归零), 无任何消息丢失或强制断连。

#### 2b. 持久化优化前后 ack 延迟对比 (300 op 顺序压测)

将 MySQL 持久化从"同步写穿透"逐步优化为"批量写后置"后的实测对比:

| 阶段 | ack 平均 | ack p50 | ack p95 | 吞吐量 | 300 op 的 DB 写入 |
|---|---|---|---|---|---|
| 原始（同步写穿透，ack 前等落库） | 12.67 ms | 11.75 ms | 19.12 ms | 78.7 ops/s | 300 次独立事务 |
| 写后置（单条异步落库） | 5.02 ms | 4.41 ms | 8.75 ms | 196.7 ops/s | 300 次独立事务 |
| **批量写后置（缓冲 + saveAll 批量插入）** | **2.77 ms** | **2.49 ms** | **4.68 ms** | **356.5 ops/s** | **~6-7 次批量插入** |

**结论**: ack 平均延迟下降约 78%, 吞吐量提升约 4.5 倍; 历史查询仍返回完整 301 条 op, 持久化正确性不受影响。优化手段对标竞品 SyncCanvas 的 "Kafka 批写"。

#### 3. WebSocket 发送队列指标 (并发瞬态消息冲击)

20 个并发 peer 各发送 200 条 cursor/emoji 瞬态消息 (共 4000 条), 模拟大量用户同时移动光标或发送表情的极端场景:

| 指标 | 数值 | 说明 |
|---|---|---|
| 并发 peer 数 | 20 | 均匀分布在 backend1 / backend2 |
| 总发送消息 | 4000 条 | cursor 占 80%, emoji 占 20% |
| 发送耗时 | 42 ms | |
| 发送速率 | 94,311 msg/s | 客户端侧发送速率 |
| 队列峰值长度 | 217 条 | 接近 TRANSIENT_DROP_THRESHOLD=96, 未达 MAX=256 |
| 队列平均长度 | 58.5 条 | 500ms 排水窗口内的均值 |
| transient 丢弃数 | 0 (0.00%) | 本轮未触发丢弃阈值 |
| overload 断开数 | 0 | 无会话因队列满而被强制关闭 |
| 队列健康状态 | **OK** | 无丢失, 无会话断连 |

**结论**: 20 并发用户高频发送瞬态消息时, 系统通过背压队列平稳吸收冲击。队列峰值 217 达到设计容量 (256) 的 85%, 说明当前阈值配置在此负载下有足够裕量。若并发用户继续增加至 50+ 且持续发送, 预计开始出现 transient 丢弃 (设计允许), 但不会触发 overload 断开 (不影响用户会话连续性)。

## 9. AI 助手

已完成:

- 底部上拉菜单增加 AI 助手入口。
- AI 可读取当前会议阶段、白板对象、最近会议对话作为上下文。
- 支持生成白板内容并直接应用为协作 op。
- 支持 `create` / `update` / `delete` 混合操作。
- 支持复杂请求的多智能体编排:
  - Planner 主 AI 划分区域。
  - 多个子 AI 并行生成各区域内容。
  - Merge AI 补充跨区域连接线和总标题。
- 简单请求自动降级为单次 AI 调用。
- AI 返回 op 会经过后端和前端双层校验后再写入白板。
- 支持生成 Markdown 会议总结并下载。
- AI 接口使用 OpenAI-compatible Chat Completions 形式, 默认面向 DeepSeek 配置:
  - `AI_API_KEY`
  - `AI_API_URL`
  - `AI_MODEL`

已知限制:

- AI 生成结果仍依赖模型稳定性, 无法保证每次布局完全理想。
- 目前不是流式输出, 大型多区域生成需要等待完整返回。
- AI 调用失败时只做前端错误提示和后端降级, 尚未做任务队列/重试。
- 尚未为 AI token 消耗、调用耗时和失败率暴露独立指标。
