## 3. 测试结果

### 3.1 路由与缓存读取

**场景说明：** 对同一房间 ID 发起 80 次 `GET /api/rooms/{id}`，通过 Nginx 负载均衡分发到两个 backend 节点，验证路由延迟与 Caffeine 缓存命中情况。

| 指标 | 数值 |
| --- | --- |
| 请求总数 | 80 |
| **平均延迟** | **3.32 ms** |
| p50 延迟 | 3.18 ms |
| **p95 延迟** | **4.17 ms** |
| p99 延迟 | 7.50 ms |
| 最小延迟 | 2.24 ms |
| 最大延迟 | 7.93 ms |
| 本轮新增请求 | 29 次（其余命中上轮缓存） |
| 本轮缓存命中 | 28 次 hit / 1 次 miss |
| 节点全局命中率 | **97.5%**（累计 80 请求 78 命中） |

**分析：**
- 平均 3.32ms、p95 4.17ms，满足 < 20ms 的需求目标，是目标值的 1/5。
- p99 7.50ms 仍属健康，本轮最大值 7.93ms 仅为单次 Caffeine miss 触发数据库加载所致。
- 97.5% 全局命中率超过 95% 目标。缓存预热后所有节点均稳定命中，REST 查询路径无瓶颈。

---

### 3.2 可靠写入（WebSocket op）

**场景说明：** `peer-a` 连接 backend1，`peer-b` 连接 backend2。peer-a 串行发送 61 条 op（index 0 为 create，后续为 update），每条等待 `op-ack` 后再发下一条；同时验证 peer-b 通过 Redis Pub/Sub 收到第一条和最后一条 op。

| 指标 | 数值 |
| --- | --- |
| op 总数 | 61 |
| 总耗时 | 71.31 ms |
| **吞吐量** | **855.5 ops/s** |
| **首条 ack 延迟** | **1.65 ms** |
| **首条跨节点 fanout** | **3.86 ms** |
| **ack 平均延迟** | **1.17 ms** |
| ack p50 | 0.92 ms |
| **ack p95** | **2.05 ms** |
| ack p99 | 4.53 ms |
| ack 最小值 | 0.63 ms |
| ack 最大值 | 6.02 ms |
| 队列 transient 丢弃 | 0 |
| overload 断连 | 0 |

**一致性校验：**

| 检查项 | 结果 |
| --- | --- |
| 首条 op 跨节点 peer-b 收到 | **通过** |
| 最后一条 op 跨节点 peer-b 收到 | **通过** |

**分析：**
- ack 平均 **1.17ms**、p50 仅 **0.92ms**，系统稳态下单节点往返延迟极低，体现了"先 ACK 后异步持久化"设计的优势。
- **首条跨节点 fanout 3.86ms**（相比上次测试的 25ms 大幅改善），关闭后台进程后 Redis Pub/Sub 容器间通信延迟显著下降。
- 吞吐量 **855.5 ops/s**，超出需求目标（100 ops/s）约 8.5 倍，批量写后置（每 200 条或 200ms 一次 saveAll）对吞吐量贡献显著。
- p99 4.53ms 偶发毛刺来自 Redis Pub/Sub 协程调度抖动，不影响稳态性能。

**对比两次测试：**

| 指标 | 本次（无后台进程） | 上次（有后台进程） | 改善幅度 |
| --- | --- | --- | --- |
| ack 平均延迟 | **1.17 ms** | 1.37 ms | ↓ 15% |
| ack p95 | 2.05 ms | 1.74 ms | — |
| 首条 ack | **1.65 ms** | 11.09 ms | ↓ 85% |
| 首条跨节点 fanout | **3.86 ms** | 25.25 ms | ↓ 85% |
| 吞吐量 | **855.5 ops/s** | 821.6 ops/s | ↑ 4% |

> 关闭后台进程后，受益最明显的是首次操作延迟（-85%）和跨节点 fanout（-85%），这两项对应 Docker 内核网络栈的冷路径竞争，系统空闲时处理更快。稳态 ack 提升相对有限，说明稳定负载下批量写策略已将延迟压缩到接近底限。

### 3.3 瞬时消息突发（transient burst）

**场景说明：** 6 个 peer 交替连接 backend1 / backend2，并发发送 240 条 cursor（80%）和 emoji（20%）transient 消息，验证高并发非可靠消息下背压机制的行为。

| 指标 | 数值 |
| --- | --- |
| 并发 peer 数 | 6 |
| 每 peer 消息数 | 40 |
| 总发送消息数 | 240 |
| 发送总耗时 | 7.85 ms |
| **发送吞吐量** | **30,573 msg/s** |
| **队列峰值积压** | **0 条** |
| 队列平均积压 | 0 条 |
| **transient 丢弃数** | **0** |
| **overload 断连数** | **0** |
| 强制断开 peer 数 | 0 / 6 |

**分析：**
- 队列峰值积压为 **0 条**（上次为 4 条），资源充足时 transient 消息被立即消费，队列无积压。
- 背压保护阈值（丢弃阈值 96、断开阈值 256）在当前负载下完全未触发，系统处于健康状态。
- 240 条消息 30,573 msg/s 的发送吞吐量表明前端发送路径和服务端接收路径均无瓶颈。
