# 11 开发报告

> 更新日期: 2026-05-29  
> 当前阶段: Cocanvas 已从基础协作 Demo 演进为具备房间管理、产品化白板交互、实时协作、历史恢复和大画布性能优化能力的在线协作白板工作台。

## 1. 报告摘要

本阶段围绕“让 Cocanvas 真正可用”展开, 重点不再只是证明 WebSocket 可以同步图形, 而是把产品体验补齐到头脑风暴、会议讨论和产品协作场景可以持续使用的状态。

当前系统已经具备以下主线能力:

- 房间控制台: 创建、管理、归档、密码校验、房间号、权限字段和语音入口配置。
- 协作身份: 用户可自定义名称和颜色, 支持中文输入法, 并用于远端光标展示。
- 白板工作台: 无限画布、工具栏、图形创建、拖拽创建、编辑属性、连线、分组、多选、右键菜单、快捷键。
- 场景化讨论: 面向头脑风暴和产品讨论的模板、会议流程、卡片与导入导出能力。
- 实时协作: WebSocket 房间 join、远端光标、操作广播、shape-preview、断线重连和 pending op 补偿。
- 分布式底座: Redis Pub/Sub、多后端节点、一致性哈希路由、MySQL 操作日志和快照恢复。
- 性能优化: 大量节点场景下, 单节点拖拽已从整层重绘优化为临时拖拽层重绘, 实测 1/1000/3000 节点下拖拽体验接近一致。

## 2. 本阶段开发目标

本阶段的目标可以拆成三类。

第一类是产品化体验:

- 用户进入后看到的是一个完整白板工作台, 而不是孤立 canvas。
- 图形应支持文字、属性、尺寸、颜色、边框、分组和连线。
- 工具使用方式应符合常见办公软件习惯, 例如拖拽创建、Ctrl 多选、复制粘贴、撤销重做、右键菜单。
- 模板和会议流程应服务真实讨论场景, 而不是单纯展示按钮。

第二类是协作与房间:

- 同一房间内多个页面应看到一致状态。
- 房间应有基础管理入口, 包括房间名、房间号、密码、权限字段、语音开关等。
- 用户应能定义自己在房间中的显示名称和颜色。
- 远端光标应使用画布世界坐标, 避免不同用户视口不同导致错位。

第三类是稳定性与性能:

- 画布必须铺满浏览器可用空间。
- 画布应表现为无限画布, 支持持续平移、缩放和内容适配。
- 大量节点存在时, 拖动单个节点不应随节点总量线性变卡。
- 关键功能需要通过 lint、build 和浏览器自动化验证。

## 4. 性能优化专项

### 4.1 问题背景

用户反馈: 节点数量增加后, 拖拽单个节点明显变卡。目标是无论画布中有多少节点, 只拖动一个节点时, 体感应尽可能接近只有一个节点的情况。

原始问题主要来自两个方面:

- React 状态在拖拽过程中频繁更新, 触发大量渲染计算。
- Konva 同一个 Layer 是一张 canvas, 即使只移动一个节点, `batchDraw()` 也可能重绘整层所有节点。

### 4.2 已实施优化

已完成:

- 拖拽过程中不再每帧 setState 更新节点位置。
- 使用 Konva node ref 命令式移动正在拖拽的节点。
- 将正在拖拽的节点临时移动到独立 drag layer。
- 静态节点留在 static layer, 拖拽高频阶段不参与重绘。
- 背景网格、连线、静态图形、拖拽图形拆分为多个 Layer。
- 拖拽期间关闭静态层和连线层的命中检测, 减少鼠标移动时遍历大量图形。
- 连线跟随拖拽时优先读取 Konva 节点实时位置, 避免视觉慢半拍。
- 添加 viewport culling, 只渲染可视区域附近的节点和相关连线。
- 将 shape-preview 发送节流从 16ms 调整为 50ms。
- 增加开发环境性能种子函数 `window.__cocanvasPerfSeed(count)`, 用于本地快速构造大量节点压测。

### 4.3 实测结果

测试环境:

- 入口: `http://localhost:8088`
- 浏览器: Microsoft Edge headless
- 方法: 创建临时房间, 通过开发测试钩子分别生成 1 / 1000 / 3000 个节点, 使用 CDP 模拟鼠标拖拽第一个节点。
- 指标: CDP 单步耗时和页面 `requestAnimationFrame` 帧间隔。

测试结果:

| 节点数 | CDP 平均步耗时 | CDP 最大步耗时 | RAF 平均帧间隔 | RAF 最大帧间隔 | >50ms 长帧 |
|---:|---:|---:|---:|---:|---:|
| 1 | 16.45ms | 29.00ms | 17.45ms | 66.70ms | 1 |
| 1000 | 16.05ms | 29.30ms | 16.79ms | 33.30ms | 0 |
| 3000 | 16.11ms | 25.80ms | 16.67ms | 16.80ms | 0 |

结论:

- 在当前测试脚本下, 1000 和 3000 节点时拖拽单个节点的平均耗时已经与 1 个节点基本一致。
- 说明拖拽主路径已经从“随节点总数增长”优化为“主要与正在拖拽的节点数量相关”。
- CDP 指标包含调试协议往返开销, 不等同于真实用户 FPS, 但同脚本横向对比可以说明节点总量不再显著拖慢单节点拖拽。

## 7. 已知风险与后续计划

### 7.1 P0: 权限与房间安全落地

当前房间管理 UI 已经成型, 但权限还没有真正控制行为。

后续应完成:

- 房间密码带盐哈希。
- 创建房间后生成短期 join token。
- WebSocket join 校验 token。
- 后端按 `permissionMode` 拦截 op 写入。
- 只读用户仍可看光标和内容, 但不能创建、修改、删除对象。

### 7.2 P0: 历史回放隔离

当前历史能力仍需要从“恢复状态”升级为“只读回放”。

后续应完成:

- 独立 replay store。
- 历史画布只读渲染。
- 查看历史不污染当前实时协作状态。
- 从历史版本恢复到当前画布时需要明确确认。

### 7.3 P1: 协作会议能力

语音入口已经具备配置和 UI 基础, 但还不是完整会议系统。

后续应完成:

- WebRTC 语音通话。
- 语音房间成员列表。
- 主持人能力。
- 会议流程同步给房间所有成员。

### 7.4 P1: 模板与自动化导入

当前已经支持基础导入和模板, 但仍可增强为真正的白板生产力工具。

后续应完成:

### 7.5 P1: 大房间持续性能

后续应完成:

## 9. 总结

Cocanvas 当前已经具备完整的产品雏形: 它既能作为多人在线白板进行头脑风暴和产品讨论, 又保留了分布式系统项目需要展示的核心能力, 包括多节点广播、一致性哈希路由、CRDT 合并、操作日志、快照恢复和断线重连。

本阶段最重要的进展是把系统从“功能可以跑”推进到“交互能持续使用”。尤其是画布铺满、无限画布、多选分组、右键菜单、快捷键、房间管理和拖拽性能优化, 都显著降低了真实使用时的摩擦。

下一阶段应优先把权限安全、历史回放隔离和会议协作同步补齐, 再进一步沉淀压测脚本和演示脚本, 让项目在答辩和后续扩展中都更稳。
