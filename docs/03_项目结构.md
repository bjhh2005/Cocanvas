## 整体架构分层

Cocanvas 在概念上分四层,从外到内分别是客户端层、接入层、协同与业务层、基础设施层。

最外层是**客户端层**,运行在浏览器中的 React 应用,负责画布渲染、本地操作的产生、UI 交互、以及与服务端的双向通信。客户端持有一份本地的画布状态副本,所有用户的可见画面都是从这份本地副本渲染而来,而不是直接从服务端拉数据画到屏幕上。

**接入层**由多个 Spring Boot 实例组成,每个实例同时承担 HTTP REST 与 WebSocket 双重职责,共用同一个 Tomcat 端口。这一层负责接收前端连接、维护 WebSocket 会话、做房间到连接的路由、把消息发给同房间内本节点上的其他连接,以及把跨节点消息通过 Redis 转发出去。

**协同与业务层**在物理上和接入层同处一个 JVM 进程,但在代码组织上严格分开。这一层包含 CRDT 操作处理、HLC 时钟管理、房间会话注册表、操作日志持久化调度、节点路由计算、服务发现等核心业务逻辑。在 Spring 里它体现为一组 Service Bean,被接入层的 WebSocket Handler 和 REST Controller 调用。

最内层是**基础设施层**,包括 Redis 和 MySQL。Redis 同时承担两个角色,一是跨节点 Pub/Sub 广播通道(房间内操作通过它在多节点间同步),二是服务发现的轻量注册中心(各节点心跳上报、其他节点订阅活节点列表)。MySQL 负责操作日志和快照的持久化,按 roomId 哈希分片以支持后续水平扩展。

跟 Node.js 方案最显著的差异在并发模型。Node.js 是单线程事件循环,WS 消息天然串行;Spring Boot 的 WebSocket 消息分发是多线程的,每条消息可能被不同线程处理。这意味着所有跨连接的共享数据结构——房间会话注册表、CRDT 副本状态、操作缓冲区——都必须线程安全,会用到 `ConcurrentHashMap`、读写锁、原子变量这些并发原语。这反而是分布式系统课程里"单机并发 → 分布式协同"过渡的经典内容,答辩时可以顺势引出。

## 技术栈

前端栈是 **React 18 + TypeScript + Vite + Konva + Zustand**。TypeScript 是必须的,弱类型在 CRDT 场景下会埋雷;Vite 启动快、HMR 体验好;Konva 负责画布,在大量形状的命中检测与事件处理上比 Fabric 更友好;Zustand 做状态管理,比 Redux 轻得多,而且 Konva 频繁更新场景下细粒度订阅更友好。

后端栈是 **Spring Boot 3.x + Java 21**。选 Java 21 是因为它是 LTS,而且虚拟线程对 WebSocket 这种 IO 密集场景的契合度可以作为答辩亮点。构建工具用 **Gradle Kotlin DSL**,比 Maven 的 XML 清爽一个量级,但如果团队对 Maven 更熟也可以用 Maven,这点不影响架构。

**WebSocket 选型是后端最关键的决定**:Spring 提供两种 WebSocket 风格,STOMP(自带订阅、确认、广播的消息代理协议)和原生 WebSocket(`WebSocketHandler` 接口,自己掌控所有消息)。**必须选原生**。STOMP 默认假设有集中式 broker,房间路由和广播是它自己的一套,会和你们要做的一致性哈希路由、CRDT 排序直接冲突。直接用 `WebSocketHandler` 加自己维护 `Map<roomId, Set<WebSocketSession>>` 才有完全控制权。

HTTP 用 **Spring MVC**(不是 WebFlux,WebFlux 学习曲线陡且对 WS 场景无明显收益)。JSON 用 Spring 默认的 **Jackson**,WS 消息通过 `type` 字段做判别,后端用 `@JsonTypeInfo` + `@JsonSubTypes` 实现多态反序列化。Redis 用 **Spring Data Redis + Lettuce 客户端**(Lettuce 异步、线程安全,适合 Pub/Sub;不要用 Jedis)。MySQL 入手用 **Spring Data JPA**,后期分片复杂时可引入 **ShardingSphere-JDBC**。Lombok 可选,Java 21 的 record 已能覆盖大部分 DTO,DTO 用 record 即可。

## 仓库组织

切到 Spring Boot 后,Node.js 方案下 monorepo + shared 包共享代码的优势消失了——JVM 和 JS 共享代码不现实。所以仓库改为**顶层目录下并列两个独立子项目**的形式。

根目录下放四样东西:`frontend/`(Vite + React,独立 `package.json`)、`backend/`(Spring Boot Gradle 项目,独立 `build.gradle.kts`)、`protocol/`(协议文档,纯 Markdown)、`docker-compose.yml`(本地起 Redis 和 MySQL)。再加 `README.md` 和 `.env.example`。

**`protocol/` 这个目录非常关键**。失去 shared 包后,前后端协议同步只能靠纪律,纪律的载体就是这份文档。所有 WS 消息类型、所有 REST 端点都必须先在 protocol 里改完,再分别在前后端落实现,PR 评审时把这条作为硬性 checklist。后期如果觉得手工对齐风险大,可以引入 OpenAPI 描述 REST、JSON Schema 描述 WS 消息,用代码生成器分别为前端 TS 和后端 Java 生成类型——但这是优化项,当前阶段用 Markdown 文档对齐就够。

## 后端项目结构

后端用单模块 Gradle 项目,不要一上来就拆多模块,小团队多模块只会增加构建复杂度。

标准 Spring Boot 目录结构:`src/main/java/com/cocanvas/`、`src/main/resources/`、`src/test/java/com/cocanvas/`。`src/main/resources/` 下放 `application.yml`(默认配置)、`application-dev.yml`(开发覆盖)、`application-prod.yml`(生产覆盖)和 Logback 配置。

Java 包按职责从外到内拆分:

**`config/`** 是 Spring 的装配中心,放所有 `@Configuration` 类。当前阶段至少要有 `WebSocketConfig`,负责注册 WebSocketHandler 和握手拦截器,后期补 `RedisConfig`(Lettuce 连接、订阅容器)、`JpaConfig`、`AsyncConfig`、`CorsConfig` 等。Bean 关系出问题时先来这里找。

**`controller/`** 放 HTTP REST 控制器。当前阶段只有 `RoomController`(创建/查询房间)和 `HealthController`,后期加 `HistoryController`、`MetricsController`。控制器保持薄,所有业务逻辑下沉到 service。

**`ws/`** 是后端最核心的一块。内部细分:`CollabWebSocketHandler` 继承 `TextWebSocketHandler`,实现连接建立、消息接收、连接断开三个回调,是 WS 消息的入口;`HandshakeAuthInterceptor` 实现 `HandshakeInterceptor`,在 WS 升级前校验 query param(roomId、userId、token),把信息塞进 session attributes;`RoomSessionRegistry` 维护房间到 WebSocketSession 集合的映射,用 `ConcurrentHashMap<String, Set<WebSocketSession>>`,提供 join、leave、broadcastInRoom 等方法;`MessageDispatcher` 把入站 JSON 解析后按 `type` 字段路由到具体处理函数。

**`protocol/`** 放消息 DTO,**严格对应根目录 `protocol/` 文档**。内部分 `inbound/`(客户端→服务端消息)、`outbound/`(服务端→客户端消息)、`common/`(共用子结构,如 Operation、ShapeAttrs、UserInfo)。DTO 用 Java record,简单且不可变。抽象基类 `InboundMessage` 加 `@JsonTypeInfo(use = NAME, property = "type")` 和 `@JsonSubTypes` 列出所有子类型,Jackson 就能按 `type` 字段自动反序列化。

**`service/`** 放 Service Bean。当前阶段需要 `RoomService`(创建房间、查询房间所在节点)、`CollabService`(收到操作后的处理流程,当前直接广播);后期补 `CrdtService`(操作合并)、`HistoryService`、`SnapshotService`。

**`pubsub/`** 放 Redis Pub/Sub 封装,**当前阶段留空目录加占位类**。后期实现 `RedisOpsPublisher`(发布到 `room:{roomId}:ops` 频道)和 `RedisOpsSubscriber`(订阅并通过 RoomSessionRegistry 本地广播)。

**`crdt/`** 放 CRDT 算法,当前留空。后期放 `HybridLogicalClock`(线程安全的 HLC 实现)、`OperationLog`、`ConflictResolver`(LWW 比较器)、`ShapeReplica`。

**`routing/`** 放一致性哈希,当前留空。后期实现 `ConsistentHashRing`(Ketama 算法),`NodeRouter` 提供 `routeRoom(roomId)`。

**`discovery/`** 放服务发现,当前留空。简单方案是用 Redis 的 set 存活节点列表 + 心跳,这种自实现比引入 Nacos 或 Consul 更有故事可讲。

**`persistence/`** 放持久化层,当前留空。内部分 `entity/`(JPA 实体)和 `repository/`(Spring Data 仓库接口)。

**`exception/`** 放自定义异常和 `@RestControllerAdvice` 全局异常处理器,让 HTTP 错误响应格式统一。

**`util/`** 放工具类(ID 生成器、JSON 封装等)。

主启动类 `CocanvasApplication` 放在 `com.cocanvas` 包根,带 `@SpringBootApplication`。

**当前阶段所有留空的包都建好空目录加一个 README**,说明该模块的职责。这种"提前留好骨架"的做法很重要——它强制每个成员在自己的模块里干活,不会把 CRDT 逻辑写进 WebSocketHandler 这种应该薄的层里。

## 前端项目结构

前端按 React 项目常规组织,但要刻意把"网络层"和"渲染层"分清楚,因为后期 CRDT 逻辑会插在两者之间。

`src/` 下面分这几块:

**`pages/`** 放页面级组件,当前阶段就两个——主页(创建/加入房间)和房间页(协作画布)。

**`components/`** 放可复用 UI 组件,内部再细分:`Canvas/` 放 Konva 画布及图形组件;`Toolbar/` 放工具栏;`CursorLayer/` 放多人光标层,用绝对定位 div 实现而不是放进 Canvas;`TimelineSlider/` 是时光回放滑块,后期加。

**`store/`** 放 Zustand 状态,按领域拆分:`shapeStore`(画布上的所有图形)、`userStore`(当前用户与远端用户)、`connectionStore`(连接状态、节点地址、重连次数)。

**`network/`** 是网络层封装:`websocket.ts` 管理 WS 连接,封装实例、收发队列、心跳;`api.ts` 封装 HTTP 调用;`reconnect.ts` 处理指数退避重连。

**`types/protocol.ts`** 是关键文件——协议文档在前端的 TS 实现,所有消息类型在这里定义,这个文件改动时必须连带改后端 `protocol/` 包。

**`crdt/`** 占位,后期放本地 CRDT 状态机。

**`hooks/`** 放自定义 hooks,如 `useRoom`、`useWebSocket`。

一个关键约束:**WebSocket 实例不要放进 React 组件状态里**,放在模块单例或 Zustand 之外的专门管理器里,通过事件订阅让组件感知消息。React 重渲染和 WS 生命周期是两件事,混在一起会出诡异 bug。

## 前后端连接设计

通信分两条信道,严格不混。HTTP REST 用于一次性请求,WebSocket 用于持续双向实时通信。

### HTTP 信道

HTTP 端点由 Spring MVC 处理,默认 8080 端口。当前阶段必需的端点有三个:

`POST /api/rooms` 创建房间,请求体为空或带可选元信息,响应返回 `{ roomId, wsUrl }`。wsUrl 在当前单节点阶段固定指向自己,后期接入一致性哈希后由后端按 roomId 计算并返回该房间应该连接的具体节点地址。

`GET /api/rooms/{roomId}` 查询房间,返回 `{ roomId, exists, wsUrl }`,不存在时 exists 为 false。

`GET /health` 健康检查,Spring Boot Actuator 自带,启用即可。

跨域配置必须**在 WebSocket 和 Spring MVC 两处都开**,Spring MVC 的 `CorsConfiguration` 不会自动应用到 WebSocket 握手上,需要在 `WebSocketConfig` 里通过 `.setAllowedOrigins("*")` 单独配。

### WebSocket 信道

WS 端点路径定为 `/ws/collab`,在 `WebSocketConfig.registerWebSocketHandlers` 里注册 `CollabWebSocketHandler` 到这个路径并挂上 `HandshakeAuthInterceptor`。

**连接建立的完整流程**是:

前端先调 `POST /api/rooms` 拿到 roomId 和 wsUrl。然后发起 WS 连接,URL 形如 `ws://host:8080/ws/collab?roomId=xxx&userId=yyy&token=zzz`。Spring 收到 HTTP upgrade 请求,先走 `HandshakeAuthInterceptor.beforeHandshake`,在这里解析 query param,校验 token,通过则把 `{roomId, userId, displayName, color}` 塞进 attributes map(后续会绑定到 WebSocketSession 的 attributes),返回 true 放行;失败 setResponseStatus 给 401 并返回 false。

Spring 完成 HTTP 101 升级,创建 WebSocketSession,触发 `CollabWebSocketHandler.afterConnectionEstablished(session)`。这个回调里读 attributes 拿到 roomId/userId,**但不要立刻 join 房间**,等客户端发 join 消息再 join——握手通过只代表连接合法,加入房间是逻辑层动作,这样未来加房间密码等机制都在 join 消息里做。

客户端发送 `join` 消息,handler 解析后调 `RoomSessionRegistry.join(roomId, session)`,然后回送 `joined` 消息(当前空快照 + 空在线列表),并向同房间其他 session 广播 `peer-joined`。

后续消息走 `handleTextMessage(session, message)` 回调,经过 `MessageDispatcher` 分发到具体处理函数(`onMouseMove`、`onOp` 等),处理函数通过 RoomSessionRegistry 完成本地广播。**当前阶段不走 Redis**,所有广播都是本节点内部遍历 session 集合发送。

连接断开时触发 `afterConnectionClosed(session, status)`,handler 调 `RoomSessionRegistry.leave` 并向房间其他人广播 `peer-left`。

### 消息协议格式

所有 WS 消息用 JSON,顶层结构统一:`{ type: string, msgId: string, ts: number, ...payload }`。msgId 由客户端生成用于去重和确认;ts 是发送时间,后期换成 HLC。

Spring 端通过 `TextWebSocketHandler` 接收 `TextMessage`,用 ObjectMapper 反序列化到 `InboundMessage` 抽象基类。Jackson 配置好 `@JsonTypeInfo` 后会按 `type` 字段自动选具体子类。出站消息直接 `objectMapper.writeValueAsString(outboundMessage)` 序列化。

当前阶段入站消息类型:`join`、`leave`、`op`、`cursor`、`ping`。出站消息类型:`joined`、`peer-joined`、`peer-left`、`op`、`cursor`、`pong`、`error`。

### 心跳与重连

Spring 原生 WebSocket 不自动管心跳,需要自实现。**用应用层心跳**:客户端每 20 秒发 ping 消息,服务端回 pong,服务端用 `ScheduledExecutorService` 定时扫描所有 session 的 `lastActiveTime` 属性,超过 45 秒没活动就主动关闭。这种做法的好处是心跳也是普通业务消息,易于调试。不要用 WS 协议层的 ping/pong 帧——浏览器 WebSocket API 不暴露主动发 ping 的接口。

重连策略用**指数退避加抖动**,首次 1 秒,失败后 2、4、8 秒,封顶 30 秒,每次加随机抖动避免雪崩。重连成功后客户端要重新走 join 流程,后期还要拉取断线期间错过的操作。

### 跨节点广播的预留

当前单节点不需要 Redis,但代码结构要预留好接口。具体做法:`CollabService` 收到操作时调用 `OpsBroadcaster.broadcast(roomId, op)` 抽象,当前实现是 `LocalOnlyBroadcaster`(只走 RoomSessionRegistry 本地广播),后期换成 `RedisPubSubBroadcaster`(发布到 Redis 频道,所有节点订阅后再本地广播)。用 Spring 的 `@Profile` 或 `@ConditionalOnProperty` 按配置切换,不用改 Service 代码。这种"先抽象后替换"在 Spring 里成本很低,值得一开始就做。

## 本地开发环境

根目录 `docker-compose.yml` 起两个服务:Redis(6379)、MySQL 8(3306,创建初始库 `cocanvas`,简单的 root 密码)。

后端启动 `cd backend && ./gradlew bootRun`,默认读 `application-dev.yml`。模拟多节点用 `--server.port=8080` 和 `--server.port=8081` 起两个实例即可。

前端启动 `cd frontend && pnpm dev`,默认 5173,通过环境变量 `VITE_API_BASE` 指向后端 `http://localhost:8080`。

Spring Boot 启动比 Node.js 慢(几秒到十几秒),频繁重启影响效率。引入 **Spring DevTools** 依赖,它会监听 classpath 变更热重启 ApplicationContext,实测比冷启动快不少。IDE 用 IntelliJ IDEA 配合 DevTools 体验最好。

## 当前阶段的联通验证目标

成功标准定义为:**两个浏览器窗口分别加入同一个 roomId,在一个窗口里移动鼠标,另一个窗口能实时看到一个跟随的色点**。

达到这个目标需要的最小实现:后端创建 Spring Boot 骨架,装配 `WebSocketConfig` 注册 `/ws/collab`,实现 `HandshakeAuthInterceptor` 简单透传 query param(暂不真鉴权),实现 `CollabWebSocketHandler` 的三个回调,实现 `RoomSessionRegistry` 用 `ConcurrentHashMap`,实现 `RoomController` 提供 `POST /api/rooms`。Redis 和 MySQL 依赖可以引但不启动 Bean(用 `@ConditionalOnProperty` 控制)。前端实现 `pages/Home`(创建房间按钮)、`pages/Room`(连接 WS、监听鼠标、渲染远端光标),封装好 `network/websocket.ts` 和 `store/connectionStore`。

CRDT、HLC、Redis、MySQL、多节点全部不做。但占位包(`crdt/`、`pubsub/`、`routing/`、`persistence/`)在后端先建好空目录与 README 说明模块职责,避免后续填代码时归类混乱。

## 推进顺序

阶段一(当前)单节点联通,鼠标光标同步。

阶段二加图形 CRUD,服务端简单转发不做合并,验证 Konva 集成、Zustand 状态管理、协议多态序列化。

阶段三引入 HLC 和 CRDT 处理并发冲突,核心难点,要预留最多时间。Java 这边 HLC 实现注意线程安全,用 `AtomicLong` 或加锁。

阶段四接入 Redis Pub/Sub 实现跨节点广播,替换 `LocalOnlyBroadcaster` 为 `RedisPubSubBroadcaster`,本地多端口起两个实例验证。

阶段五加 MySQL 持久化、快照、时光回放。

阶段六做一致性哈希路由加简单服务发现(Redis 注册节点列表)加故障切换演示。

每一阶段都要保持端到端可演示,避免做完一堆后端才接前端——分布式系统的 bug 大多在前后端交互处,越晚联调坑越深。
