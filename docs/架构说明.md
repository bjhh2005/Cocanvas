# 架构说明

> 更新日期: 2026-05-30  
> 说明: 本文档描述当前代码的运行架构、模块职责和关键数据流。快速启动看根目录 `README.md`, 协议细节看 `接口文档.md`。

## 1. 总体架构

```text
Browser
  |
  | HTTP / WebSocket
  v
Nginx :8088
  |-- frontend: Vite React app
  |-- backend1: Spring Boot
  `-- backend2: Spring Boot

backend1/backend2
  |-- WebSocket 协作入口
  |-- session 发送队列与慢客户端隔离
  |-- 房间管理 REST API
  |-- join token 与权限校验
  |-- HLC/CRDT 房间副本
  |-- MySQL 操作日志与 snapshot
  |-- Redis 节点注册
  `-- Redis Pub/Sub 分片广播

Redis
  |-- 节点存活信息
  `-- 跨节点房间事件

MySQL
  |-- 房间元数据
  |-- 操作日志
  `-- 快照
```

系统把实时链路和恢复链路分开:

- 实时链路: WebSocket -> 后端应用内存副本 -> ack -> 本节点广播 -> Redis 跨节点广播；MySQL 持久化批量写后置，不在 ack 关键路径上。
- 恢复链路: REST history -> 最新 versioned snapshot -> snapshot 后增量 op -> 前端 CRDT apply。
- transient 链路: cursor / shape-preview -> 本节点低优先级队列 -> Redis transient 分片 -> 远端节点低优先级队列。

## 2. 前端结构

路径:

```text
src/frontend/app
```

主要模块:

- `src/pages/Home.tsx`: 房间控制台。
- `src/pages/Room.tsx`: 房间协作页, 管理连接、重连、history restore、pending outbox。
- `src/components/CanvasBoard.tsx`: Konva 白板画布。
- `src/components/Toolbar.tsx`: 工具栏。
- `src/components/ProductPanel.tsx`: 产品讨论面板、模板、筛选。
- `src/store/shapeStore.ts`: 图形 CRDT 状态。
- `src/store/userStore.ts`: 用户身份和远端用户状态。
- `src/store/connectionStore.ts`: WebSocket 状态。
- `src/network/api.ts`: REST API 封装。
- `src/network/websocket.ts`: WebSocket 客户端封装。
- `src/types/protocol.ts`: 前端协议类型。

关键机制:

- 本地操作先应用到 `shapeStore`, 保证交互即时反馈。
- 操作同时进入 pending outbox。
- 发送到服务端后等待 `op-ack`。
- 收到 `op-ack` 后删除 pending outbox, 并用服务端 HLC 校准本地版本。
- 重连时先恢复 history, 再 flush pending outbox。
- 手动历史回放实际渲染到画布 (`applyHistoryState`), 顶部横幅标记历史模式, 点"返回实时"恢复最新状态。
- 协作诊断条展示当前连接节点、`wsUrl`、重连次数、pending ops、最近恢复/回放/flush 数量。

## 3. 后端结构

路径:

```text
src/backend/java/src/main/java/com/cocanvas
```

主要模块:

- `controller/`: REST API。
- `ws/`: WebSocket handler 和房间 session registry。
- `protocol/`: WebSocket DTO。
- `service/`: 房间业务、历史恢复、内存副本。
- `crdt/`: Hybrid Logical Clock。
- `pubsub/`: Redis Pub/Sub 广播和订阅。
- `cluster/`: 节点身份和节点注册。
- `routing/`: 一致性哈希路由。
- `persistence/`: JPA entity 和 repository。
- `config/`: WebSocket、Redis Pub/Sub、CORS 配置。

关键类:

- `CollabWebSocketHandler`: WebSocket 消息入口。
- `RoomSessionRegistry`: 本节点房间 session 集合。
- `SessionSendQueue`: 每个 session 独立发送队列和慢客户端背压。
- `RoomReplicaService`: 本节点房间内存副本与 CRDT 合并。
- `HistoryService`: op log、snapshot、history restore。
- `JoinTokenService`: REST 发放短期 join token, WebSocket join 校验。
- `RedisRealtimeBroadcaster`: 本节点广播 + Redis 分片发布。
- `RedisRoomEventSubscriber`: Redis 订阅后同步本节点副本并广播本节点客户端。
- `RedisNodeRegistry`: Redis 节点心跳和节点发现。
- `NodeRouter`: 一致性哈希房间路由。

## 4. 实时 op 数据流

```text
Client A
  |
  | join + joinToken
  v
backend validates token
  |
  | op
  v
backend owner/current node
  |
  | validate permissionMode
  | merge HLC
  | apply RoomReplicaService（内存副本，关键路径）
  | send op-ack to Client A
  | broadcast local sessions
  | enqueue operation_logs（批量写后置，异步）
  v
Redis cocanvas:room-events:{shard}
  |
  v
other backend nodes
  |
  | apply remote op to RoomReplicaService
  | broadcast local sessions
  v
Client B
```

可靠性边界:

- op 应用到内存副本并 ack/广播后即对所有协作者生效；MySQL 持久化批量异步进行，失败仅记录日志。
- 持久化失败的兜底：60s 周期快照直接从内存副本生成，可恢复最终状态；CRDT/HLC 合并可交换，批量乱序落库不影响结果。
- 发送方没有收到 `op-ack` 时, pending outbox 会保留该 op。
- 重连后 pending op 会再次发送。
- opId 是幂等键, 重放不会重复写日志（批内去重 + 主键约束 + 批失败逐条回退）。
- `view` 房间拒绝 op; `comment` 房间只允许 comment 对象 op。
- 权衡：相比"同步写穿透"，实时性更好但单条 op 强持久化保证减弱（节点非优雅退出时缓冲区未 flush 的 op 可能丢失，靠快照兜底）。

## 5. transient 消息与背压

cursor 和 shape-preview 不进入 MySQL 操作日志, 也不参与历史恢复。它们作为 transient 消息处理:

```text
Client
  |
  | cursor / shape-preview
  v
backend
  |
  | local transient send queue
  v
Redis cocanvas:room-transient-events:{shard}
  |
  v
other backend nodes
  |
  | transient send queue
  v
local clients
```

背压策略:

- 每个 WebSocket session 独立队列, 避免慢连接拖慢整个房间。
- 队列达到 transient 阈值时丢弃 cursor / shape-preview。
- 队列达到最大阈值且消息为 op 等可靠消息时, 关闭该 session, 客户端通过重连和 history 恢复。

## 6. 历史恢复数据流

**断线重连恢复链路:**

```text
Client reconnect / refresh
  |
  | GET /api/rooms/{roomId}/history
  v
HistoryService
  |
  | find latest snapshot before at
  | find ops after snapshot.createdAt
  v
Client
  |
  | replaceWithSnapshot(versioned snapshot)
  | apply incremental ops
  | flush pending outbox
```

**手动历史回放链路 (2026-05-30 新增):**

```text
用户拖动时间轴滑块 → 选择目标时间点 historyAt
  |
  | GET /api/rooms/{roomId}/history?at=historyAt
  v
HistoryService (同上恢复链路)
  v
Client
  |
  | applyHistoryState() → replaceWithSnapshot + apply incremental ops
  | setHistoryMode(true) → 顶部横幅标记"历史回放模式 · 时间"
  |
用户点"返回实时"
  |
  | GET /api/rooms/{roomId}/history (最新状态)
  | restoreLatestState() → 重新加载当前画布
  | setHistoryMode(false)
```

**时间轴 anchors 接口:**

```text
GET /api/rooms/{roomId}/history/anchors
→ { roomCreatedAt, snapshots: [ts, ...], latestOpAt }

前端使用:
  - min = roomCreatedAt  (滑块左边界)
  - max = latestOpAt     (滑块右边界)
  - snapshots → 蓝色刻度线位置 (left% = (ts-min)/(max-min)*100)
```

**op 日志裁剪策略:**

```text
每次 saveSnapshot(roomId) 后:
  1. 查找本次快照时间戳 now 之前最近的一个旧快照 prev
  2. 删除 roomId 下 createdAt < prev.createdAt 的所有 op 日志
  3. 保留 prev.createdAt 之后的 op (用于 prev→now 区间回放)
  4. 回放 prev~now 之间任意时间点仍可精确还原
```

snapshot 格式:

- shapeId -> attrs。
- 每个属性可以是 versioned value:
  - `value`
  - `hlc`
  - `writerId`
- `shapeType` 保留为普通字段。

## 7. 节点发现与路由

Redis key:

```text
cocanvas:nodes              Set, 保存 nodeId
cocanvas:nodes:{nodeId}     String, JSON NodeInfo, TTL 15 秒
```

路由:

- 后端节点每 5 秒写心跳。
- `NodeRouter` 从 `NodeRegistry` 读取存活节点。
- 节点列表按 `nodeId` 排序。
- 构建一致性哈希环。
- 按 `roomId` 选择 owner 节点。
- 节点列表不变时复用缓存 hash ring。

## 8. Pub/Sub 分片

频道:

```text
cocanvas:room-events:{0..63}
cocanvas:room-transient-events:{0..63}
```

分片:

- 对 `roomId` 计算 SHA-256。
- 取前 4 字节转 int。
- `floorMod(hash, 64)` 得到分片编号。

当前每个后端订阅可靠事件和 transient 事件各 64 个分片。这样做的主要收益是避免所有房间挤在同一个 Redis 频道上, 并把 op 与 cursor/preview 的优先级语义拆开。

## 9. Snapshot 写入策略

每个后端都会通过 Redis 订阅维护本节点房间副本, 但 snapshot 只允许一致性哈希 owner 节点写入。

目的:

- 避免两个后端对同一房间重复写 snapshot。
- 避免 snapshot 写入放大。
- 保留多节点内存副本, 支持本节点客户端快速广播和后续恢复扩展。

## 10. 当前瓶颈

已测得 (2026-05-30, Docker Compose 双节点, `node scripts/distributed-check.mjs`):

- WebSocket 顺序 op 压测 300 条（持久化优化历程）：
  - 同步写穿透：`ack_avg=12.67ms`, `ack_p95=19.12ms`, 吞吐 78.7 ops/s。
  - 写后置（单条异步）：`ack_avg=5.02ms`, `ack_p95=8.75ms`, 吞吐 196.7 ops/s。
  - 批量写后置（缓冲 + saveAll）：`ack_avg=2.77ms`, `ack_p95=4.68ms`, 吞吐 356.5 ops/s（ack 延迟↓78%，吞吐↑4.5x）。
- 持久化层：op 日志改为批量写后置（满 200 条 / 200ms flush，Persistable 强制 persist 走真批量插入），300 op 落库从 300 次独立事务降为 ~6-7 次批量插入。
- snapshot / op payload 字段由 `TEXT`(64KB) 改为 `LONGTEXT`，消除大白板快照截断风险。
- Redis CPU 使用率很低, 当前不是瓶颈。
- 房间元数据已升级为 Caffeine 本地缓存 (TTL 10s, max 1000), 300 次路由查询命中率 99.0%, DB 查询降至 1 次。
- WebSocket 发送队列已补充可观测指标: 20 并发 peer × 200 条瞬态消息冲击下队列峰值 217/256, `transient_drops=0`, `overload_disconnects=0`, 系统稳定。
- 客户端渲染虚拟化：空间网格索引 + 视口裁剪 + overscan，仅渲染视口内对象，大白板下渲染开销与对象总数解耦。

下一步优先优化:

- 数据层虚拟化：join/恢复时按视口/区域懒加载，避免大白板一次性下载全量快照。
- 节点非优雅退出时的缓冲区 flush（`@PreDestroy`），减小写后置丢失窗口。
- 权限模型从房间级升级为成员级。
