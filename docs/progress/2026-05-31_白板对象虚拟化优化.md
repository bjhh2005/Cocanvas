# 2026-05-31 白板对象虚拟化优化

## 1. 本阶段目标

在大规模协作与多图元复杂场景下，降低前端的渲染和重绘开销，同时优化多用户协作时产生的高频瞬态消息。

## 2. 已完成工作

### 2.1 白板对象视口剔除 (Viewport Culling) 与空间索引

瓶颈判断:

- 随着画布图元 (Shape) 数量增加，React-Konva 及主线程渲染压力成正比增加。
- 绝大多数时间用户只能看到画布的一个局部视口，但原逻辑全量更新、计算与重绘。

完成内容:

- 引入二维空间索引网格 (Spatial Cell Grid，单元格大小 960x960)。
- 在 `CanvasBoard` 中缓存对象的包围盒 `bounds`，并根据图形分布计算其落在哪些网格中。
- 大画布发生平移、缩放时实时计算 `viewportBounds` (带防穿帮 `viewportOverscan` 缓冲)。
- 渲染时只求交并渲染当前视口范围内的网格中的 `visibleBoardShapes`。
- 对于体积过大横跨多个网格的特殊大图元 (`overflowShapes`) 单独求交。
- 只有端点所连图元处于可见视口内的连接线 (`visibleConnectors`) 才会参与判断及渲染。

效果:

- 将渲染时间复杂度从 O(总图元数) 降级为 O(视口地图元数量)，大幅降低大型白板浏览和交互时的页面卡顿。

### 2.2 多用户瞬态交互链路优化

瓶颈判断:

- 多用户同时移动光标或拖拽对象时, cursor / shape-preview 会形成 N 对 N 的瞬态消息放大。
- 这些消息只需要最新状态, 但旧消息仍可能在接收端队列和前端 state 更新中排队, 导致慢客户端看到延迟和主线程频繁渲染。

完成内容:

- 前端接收端按 animation frame 合并远端 cursor 和 shape-preview, 同一用户/对象只应用最新状态。
- 本地 cursor 发送增加细微移动过滤, 降低无意义坐标刷新。
- 后端 `SessionSendQueue` 对 cursor 和 shape-preview 做 latest-wins 队列合并, 新瞬态消息会替换同 key 的旧消息。
- Redis 订阅端在本节点没有该房间 WebSocket 会话时, 跳过 cursor / shape-preview 本地转发。

效果:

- 保持 op 可靠链路不变, 只对可丢弃的瞬态协作信号做降噪和合并。
- 多用户协作时可以减少过期 cursor / preview 的排队、跨节点无效转发和前端重复渲染。
