# 2026-05-30 分布式优化进度

## 1. 本阶段目标

本阶段目标是把 Cocanvas 的分布式协作链路从“能演示”推进到“可解释、可恢复、可压测”。

重点问题:

- Redis Pub/Sub 单一全局频道会造成扩容后无关消息浪费。
- op 实时广播和 MySQL 持久化之间缺少强约束。
- 历史恢复曾经存在从头回放或 snapshot 不可信的问题。
- Redis 节点发现使用 `KEYS`, 不适合生产级扫描。
- 一致性哈希环每次路由都重建。
- 前端 pending op 只在内存中, 页面刷新后可能丢失。
- 多后端副本都可能写同一房间 snapshot。

## 2. 已完成工作

### 2.1 Redis Pub/Sub 分片

完成内容:

- 将原先单一 `cocanvas:room-events` 频道改为 64 个分片频道。
- 发布时根据 `roomId` hash 到 `cocanvas:room-events:{shard}`。
- 后端启动时订阅所有 64 个分片。

效果:

- 避免所有房间共享一个全局频道。
- 为后续按房间/按分片做限流和监控留下入口。

### 2.2 op 持久化确认

完成内容:

- `CollabWebSocketHandler` 不再先应用副本再落库。
- 当前顺序为:
  1. 合并 HLC。
  2. 写入 MySQL 操作日志。
  3. 写入成功后应用本地副本。
  4. 给发送者返回 `op-ack`。
  5. 广播给本节点其他客户端。
  6. 发布到 Redis 分片频道。
- 新增 `OpAckMessage`。
- 前端收到 `op-ack` 后删除 pending outbox 中对应 op。

效果:

- 解决“实时看起来成功, 但历史恢复可能丢 op”的问题。
- 客户端具备基于 `opId` 的至少一次重放能力。

### 2.3 前端 pending outbox

完成内容:

- pending op 从内存数组升级为 localStorage outbox。
- key 按 `roomId:userId` 隔离。
- 在线发送的 op 也会先进入 outbox。
- 只有收到服务端 `op-ack` 才删除。
- 收到 `op-ack` 后使用服务端合并后的 HLC 校准本地图形版本。
- 服务端返回 `op_persist_failed` 时前端关闭连接并等待重连重放。

效果:

- 页面刷新、短暂断线后, 未确认 op 不会直接丢失。
- opId 幂等去重保证重放不会重复写入。

### 2.4 历史恢复与 versioned snapshot

完成内容:

- `HistoryService.history` 查询目标时间前最新 snapshot。
- 只回放 snapshot 之后到目标时间之间的增量 op。
- `RoomReplicaService.versionedSnapshot` 保存每个属性的:
  - `value`
  - `hlc`
  - `writerId`
- 前端 `shapeStore.replaceWithSnapshot` 支持恢复 versioned snapshot。
- 修复 Java `Map.of` 不支持 null 值导致 snapshot 可能静默失败的问题。

效果:

- 历史恢复成本不再随全部房间历史线性增长。
- 恢复后的 CRDT 属性版本仍可正确比较。

### 2.5 节点发现和路由

完成内容:

- Redis 节点发现去掉 `KEYS`。
- 心跳时写入:
  - `cocanvas:nodes` Set。
  - `cocanvas:nodes:{nodeId}` TTL key。
- `aliveNodes` 从 Set 读取节点索引, 对 TTL key 不存在的节点做清理。
- `NodeRouter` 缓存一致性哈希环, 节点列表不变时不重复构建。

效果:

- 避免 Redis 全库扫描。
- 降低房间路由接口频繁调用时的 CPU 开销。

### 2.6 snapshot 单 owner 写入

完成内容:

- `HistoryService.saveSnapshot` 先判断当前节点是否是 `roomId` 的一致性哈希 owner。
- 非 owner 节点不写 snapshot。

效果:

- 解决多后端都持有同一房间副本时重复写 snapshot 的问题。
- 降低 MySQL snapshot 写入放大。

### 2.7 Nginx WebSocket 长连接

完成内容:

- 节点专属 WebSocket location 增加:
  - `proxy_read_timeout 1h`
  - `proxy_send_timeout 1h`
- 通用 `/ws/` location 同步增加。

效果:

- 降低长时间协作时 WebSocket 被默认代理超时断开的概率。

### 2.8 分布式测试脚本

完成内容:

- 新增 `scripts/distributed-check.mjs`。

脚本覆盖:

- 健康检查。
- 集群节点查询。
- 创建房间。
- 房间路由接口性能统计。
- 分别连接 backend1/backend2 WebSocket。
- 发送 op 并等待 `op-ack`。
- 验证跨节点 op 广播。
- 连续发送 N 条 op, 统计吞吐与 ack 延迟。
- 查询历史恢复接口。

## 6. P0/P1 跟进完成情况

### 6.1 WebSocket 慢客户端背压

完成内容:

- 新增 `SessionSendQueue`。
- 每个 WebSocket session 拥有独立发送队列。
- `RoomSessionRegistry.broadcastInRoom` 不再同步逐个 `sendMessage`, 而是进入对应 session 队列。
- cursor / shape-preview 作为 transient 消息:
  - 队列达到 transient 阈值时直接丢弃。
  - 不阻塞 op 广播。
- op / joined / peer 等可靠消息:
  - 队列达到最大阈值时关闭该 session。
  - 客户端依赖重连、history restore 和 pending outbox 恢复。

效果:

- 慢客户端不会拖慢同房间其他 session 的发送。
- 高频 cursor / preview 在压力下可以主动降级。

### 6.2 cursor / shape-preview 低优先级通道

完成内容:

- `RealtimeBroadcaster` 增加 `broadcastTransient`。
- 本节点广播使用 `broadcastTransientInRoom`。
- Redis 跨节点发布拆出 transient 分片频道:
  - `cocanvas:room-transient-events:{0..63}`
- 原有可靠事件仍使用:
  - `cocanvas:room-events:{0..63}`
- Redis 订阅端识别 cursor / shape-preview 后走本地 transient 队列。

效果:

- op 与 cursor/preview 的通道语义分离。
- 后续可以分别统计、限流和调参。

### 6.3 后端权限校验

完成内容:

- `RoomService.canWrite` 根据 `permissionMode` 判断写入权限。
- `CollabWebSocketHandler.handleOp` 在持久化前校验权限。
- `view` 禁止所有 op。
- `comment` 只允许 `shapeType=comment` 的 op。
- 不允许写入时返回:
  - `permission_denied`

效果:

- 权限不再只是房间元数据, 后端已经参与写保护。

### 6.4 join token

完成内容:

- 新增 `JoinTokenService`。
- 创建房间和查询房间 REST 响应返回短期 `joinToken`。
- WebSocket `join` 消息必须携带 `joinToken`。
- join token 绑定:
  - `roomId`
  - `permissionMode`
  - `expiresAt`
  - HMAC-like SHA-256 signature
- token 默认 5 分钟有效。
- Docker Compose 为 backend1/backend2 显式配置同一个 `JOIN_TOKEN_SECRET`。
- token 无效、缺失或过期时返回 `invalid_join_token`, 并关闭 WebSocket。

效果:

- REST 授权查询和 WebSocket join 链路闭合。
- 多后端节点可以校验同一集群签发的 token。

### 6.5 房间元数据缓存

完成内容:

- `RoomService.findRoom` 增加 10 秒本地内存缓存。
- 创建、更新、归档房间时清理对应缓存。

效果:

- 降低高频 REST 查询对 MySQL 的直接压力。

当前限制:

- 仍是简单本地 Map, 尚未升级为 Caffeine/Redis cache。
- 尚未暴露命中率、过期数等指标。

### 6.6 历史回放与实时 store 隔离

完成内容:

- 手动历史查询不再直接覆盖实时 `shapeStore`。
- 手动查询改为生成 history preview 统计。
- 重连恢复仍走完整 `applyHistoryState`, 用 snapshot + 增量 op 重建实时状态。

效果:

- 用户查看历史信息时不会破坏当前协作中的实时画布状态。
- 恢复链路仍保持确定性重建。

### 6.7 协作诊断面板

完成内容:

- 房间顶部增加协作诊断信息:
  - 当前连接节点。
  - `wsUrl`。
  - reconnect 次数。
  - pending ops 数量。
  - last restored ops。
  - last replayed ops。
  - last flushed ops。

效果:

- 分布式演示时可以直接观察当前连接、重连和恢复状态。

## 10. 当前结论

本阶段后, Cocanvas 分布式链路已经具备:

- 多节点注册。
- 一致性哈希路由。
- Redis Pub/Sub 分片广播。
- cursor / shape-preview transient 分片广播。
- WebSocket 慢客户端背压。
- 后端房间权限写入校验。
- REST join token + WebSocket join 校验。
- op 持久化确认。
- 客户端 pending outbox。
- versioned snapshot + 增量恢复。
- 手动历史查询与实时 store 隔离。
- 协作诊断面板。
- owner-only snapshot。
- Docker 可复现压测脚本。

这意味着当前系统已经不只是展示 WebSocket 同步, 而是具备了完整的“提交、确认、广播、恢复、压测”闭环。
