## 2. 本次完成的阶段

### 第二阶段:WebSocket 最小双向通信

完成 Step 7 到 Step 10。

主要实现:

- 后端新增 `spring-boot-starter-websocket` 依赖。
- 新增 `WebSocketConfig`,注册 `/ws/echo` 和后续协作端点。
- 新增 `EchoWebSocketHandler`,支持浏览器发送文本后原样回显。
- 前端新增 `WSClient`,封装连接、发送、消息订阅、状态订阅和关闭逻辑。
- Nginx 和 Vite 开发代理均补充 `/ws/` WebSocket 转发配置。

### 第三阶段:房间机制与多客户端广播

完成 Step 11 到 Step 19。

主要实现:

- 新增 `src/protocol/messages.md`,记录 join、cursor、peer-joined、peer-left 等 WebSocket 协议。
- 后端新增 `RoomController`,支持 `POST /api/rooms` 创建房间和 `GET /api/rooms/{roomId}` 查询房间。
- 后端新增 `RoomSessionRegistry`,维护房间到 WebSocket session 的映射。
- 后端新增 `CollabWebSocketHandler`,处理 join、cursor、断开清理和房间内广播。
- 后端新增协议 DTO,包括入站 `JoinMessage`、`CursorMessage` 和出站 `JoinedMessage`、`PeerJoinedMessage`、`PeerLeftMessage`、`CursorBroadcastMessage`。
- 前端新增 React Router 页面结构:`Home` 创建/加入房间,`Room` 房间协作页。
- 前端新增 Zustand store:`connectionStore` 和 `userStore`。
- 前端新增 `CursorLayer`,可在两个浏览器窗口间同步显示远端用户光标。

### 第四阶段:图形 CRUD

完成 Step 20 到 Step 27。

主要实现:

- 协议扩展 `op` 消息,用于图形 create、update、delete。
- 后端新增 `ShapeOperation`、`OpMessage`、`OpBroadcastMessage`,并在 `CollabWebSocketHandler` 中转发 op。
- 前端新增 `shapeStore`,维护画布图形状态。
- 前端新增 `CanvasBoard`,基于 `react-konva` 渲染矩形、圆形和文本。
- 前端新增 `Toolbar`,支持创建 Rect、Circle、Text。
- 支持图形选择、拖动同步、Delete/Backspace 删除同步。

### 第五阶段:HLC 与 CRDT 冲突消解

完成 Step 28 到 Step 33 的最小可演示实现。

主要实现:

- 后端新增 `HybridLogicalClock`,支持 `now()`、`update(remoteHlc)` 和 HLC 比较。
- 前端新增 `HybridLogicalClock`,发送 op 前生成 HLC。
- `op` 消息贯通 `hlc` 和 `writerId` 字段。
- 前端 `shapeStore` 升级为属性级 LWW 寄存器:每个属性保存 value、hlc、writerId。
- 前端按 HLC + writerId 做确定性合并,实现并发冲突下最终收敛。
- 后端新增 `RoomReplicaService`,维护每个房间的内存副本,为后续持久化和恢复做准备。
- 前后端均保留删除 tombstone,避免旧 update 把已删除图形复活。

### 第六阶段:Redis Pub/Sub 与多节点

完成 Step 34 到 Step 38。

主要实现:

- 后端新增 `spring-boot-starter-data-redis` 依赖。
- 新增 `RealtimeBroadcaster` 抽象。
- 新增 `LocalRealtimeBroadcaster`,保持单节点本地广播能力。
- 新增 `RedisRealtimeBroadcaster` 和 `RedisRoomEventSubscriber`,支持跨后端节点广播房间事件。
- 新增 `NodeIdentity`、`NodeRegistry`、`LocalNodeRegistry`、`RedisNodeRegistry`,通过 Redis TTL 心跳维护活跃节点。
- 新增 `ClusterController`,提供 `GET /api/cluster/nodes` 查看活跃节点。
- 根目录 `docker-compose.yml` 改为显式启动 `backend1` 和 `backend2`,Nginx upstream 轮询两个后端节点。

### 第七阶段:持久化与时光回放

完成 Step 39 到 Step 43 的基础实现。

主要实现:

- 后端新增 `spring-boot-starter-data-jpa` 和 MySQL 驱动。
- 新增 JPA 实体 `OperationLogEntity` 和 `SnapshotEntity`。
- 新增 Repository:`OperationLogRepository` 和 `SnapshotRepository`。
- 新增 `HistoryService`,在收到 op 后写入操作日志,并周期性保存房间快照。
- 新增 `HistoryController`,支持 `GET /api/rooms/{roomId}/history?at=...` 查询历史。
- 前端新增 `getRoomHistory()` API。
- 房间页新增 Time travel 面板,可以按时间戳加载历史快照和增量操作进行回放预览。
- 测试环境新增 H2 内存数据库配置,避免本地测试必须依赖 MySQL。

### 第八阶段:一致性哈希与故障切换

完成 Step 44 到 Step 48 的基础演示链路。

主要实现:

- 后端新增 `ConsistentHashRing`,使用虚拟节点进行一致性哈希路由。
- 新增 `NodeRouter`,根据活跃节点列表计算房间应该路由到哪个节点。
- `RoomController` 创建/查询房间时,根据 `roomId` 返回对应节点的 `wsUrl`。
- 前端房间页增加 WebSocket 断线重连逻辑。
- 断线后前端使用指数退避重新查询房间信息,拿到新的 `wsUrl`,重新连接并重新 join。

## 4. 启动方式

需要先启动 Docker Desktop。

在项目根目录运行:

```powershell
cd D:\Cocanvas
.\run.bat dev
```

启动后访问:

```text
http://localhost/
```

如果希望后台启动:

```powershell
.\run.bat up
.\run.bat logs
```

停止服务:

```powershell
.\run.bat down
```

清理容器和 MySQL 数据卷:

```powershell
.\run.bat clean
```

## 5. 演示流程

推荐按下面顺序演示:

1. 打开 `http://localhost/`,点击 `Create room`。
2. 复制房间 URL,在第二个浏览器窗口打开同一个房间。
3. 移动鼠标,观察另一个窗口中的远端光标。
4. 点击 `Rect`、`Circle`、`Text` 创建图形。
5. 拖动图形,观察另一个窗口同步移动。
6. 选中图形后点击 `Delete` 或按 Delete/Backspace,观察同步删除。
7. 在 Time travel 面板输入时间戳,点击 `Load history`,验证历史查询和回放链路。
8. 使用 Docker 停掉其中一个后端容器,观察前端断线后自动重连。

## 7. 注意事项

- 访问 `http://localhost/` 前必须先启动 Docker Desktop 和项目容器。
- 如果浏览器显示 `ERR_CONNECTION_REFUSED`,通常表示 Docker 没启动、容器没起来,或 Nginx 没监听 80 端口。
- 如果 80 端口被占用,需要调整 `docker-compose.yml` 中 Nginx 的端口映射。
- 当前历史回放是基础演示版,会把历史状态应用到当前画布 store 上,不是独立只读回放视图。
- 当前故障切换是演示级实现,前端能重新查询路由并重连,完整生产级恢复还需要更严格的断线期间操作重放和幂等去重。
