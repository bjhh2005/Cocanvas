# Cocanvas 需求文档

## 1. 项目概述

Cocanvas 是一个面向多人实时协作的分布式白板系统，核心定位是 Miro 的功能子集复刻 + 分布式系统能力演示。系统允许多个用户同时进入一个"房间"，在共享的无限画布上并发地创建、移动、修改图形元素，所有操作通过分布式后端实时同步给房间内所有成员，并支持任意时间点的历史回放。

项目的设计目标不是追求 Miro 的功能完备度，而是在受控的功能子集下，**完整展示分布式系统的核心问题与解决方案**：WebSocket 集群下的房间路由、跨节点消息广播、CRDT 并发冲突消解、节点故障切换、操作日志持久化与回放、最终一致性收敛等。同时系统内置 AI 辅助创作与会议协作功能，支持通过 AI 自动生成白板内容并应用为实时协作操作。

## 2. 术语定义

| 术语 | 含义 |
| --- | --- |
| 房间 (Room) | 一个独立的协作单元，拥有唯一 roomId，内部包含一份共享画布状态和加入其中的所有用户 |
| 图形 (Shape) | 画布上的一个可编辑元素，拥有前端生成的全局唯一 UUID，终生不变 |
| 操作 (Operation) | 对画布状态的一次原子修改，如 create / update / delete，是 CRDT 同步的基本单位 |
| HLC | Hybrid Logical Clock，混合逻辑时钟，作为操作的逻辑时间戳，用于跨节点排序与冲突消解 |
| 副本 (Replica) | 同一房间在不同客户端 / 服务节点上的画布状态拷贝 |
| 快照 (Snapshot) | 某一时刻房间画布的全量状态序列化结果，用于加速历史回放与故障恢复 |
| LWW | Last-Writer-Wins，最后写入胜出，按 HLC 时间戳决定属性最终值的冲突消解策略 |
| Tombstone | 删除墓碑，标记某图形已被删除的元数据（含删除 HLC 和删除者），用于跨副本一致删除 |
| Join Token | 短期 JWT，由 REST 接口颁发，WebSocket join 时携带以完成身份校验 |
| Owner 节点 | 通过一致性哈希为某房间选出的主节点，负责该房间的周期快照写入 |

## 3. 功能需求

### 3.1 用户身份与认证

#### 3.1.1 匿名协作身份

- 用户首次访问时，浏览器本地生成匿名用户 ID，无需注册即可参与协作
- 用户可自定义显示名称和用户颜色，用于光标标识与操作归属显示
- 匿名身份信息持久化存储到 localStorage，刷新页面后保持一致

#### 3.1.2 账号体系

- 用户可注册账号（用户名 + 密码），账号密码使用 **PBKDF2 + 随机盐** 存储，不可逆
- 注册账号后可登录，系统颁发 Auth Token 存储到 localStorage
- 登录用户创建房间时自动成为房间 Owner；进入他人房间时按房间默认权限加入成员列表

#### 3.1.3 Join Token 机制

- REST 接口（`GET /api/rooms/{roomId}`）返回短期 Join Token（JWT 格式）
- WebSocket 建立连接后发送 join 消息时必须携带有效 Join Token
- 后端校验 Token 合法性（签名、过期、roomId 匹配）后允许加入房间

### 3.2 房间管理

#### 3.2.1 房间生命周期

- 用户可创建新房间，指定房间号（roomId）、房间名称
- 用户可通过 roomId 加入已存在的房间
- Owner 可更新房间配置（名称、访问模式、权限模式、密码等）
- Owner 可归档（软删除）房间；房间列表仅展示未归档房间
- 系统提供房间存在性查询，密码房间需传入密码参数校验

#### 3.2.2 访问控制模式

支持三种访问模式：

| 模式 | 说明 |
| --- | --- |
| `link` | 获得 roomId 即可访问，无需额外凭证 |
| `password` | 访问需提供房间密码 |
| `private` | 仅白名单成员可访问 |

#### 3.2.3 成员与权限

支持四种成员角色：

| 角色 | 权限 |
| --- | --- |
| `owner` | 完整管理权限，可修改房间、管理成员、转让所有权 |
| `edit` | 可创建、修改、删除所有白板对象 |
| `comment` | 仅可创建和操作 comment 类型对象，不能修改其他图形 |
| `view` | 只读，不可执行任何写操作 |

- 房间设有默认权限模式（`edit` / `comment` / `view`），未显式指定角色的用户按此权限加入
- 成员角色优先于房间默认权限
- Owner 可添加成员、调整成员角色、移除成员、转让 Owner 身份（系统保证至少保留一个 Owner）
- 后端在 WebSocket 层对每条 op 消息进行权限校验，拒绝越权操作

### 3.3 画布与图形编辑

#### 3.3.1 画布交互

- 画布为无限大平面，支持平移（鼠标拖拽、右键拖动）与缩放（滚轮）
- 提供"适应内容"（Fit to content）功能，一键将视口缩放到所有对象范围
- 底部状态条实时展示：当前缩放比例、画布对象总数、已选中对象数、当前背景模式
- 支持五种画布背景模式：`grid`（网格）、`dots`（点阵）、`paper`（纸张）、`blueprint`（蓝图）、`plain`（纯色）
- 可开启/关闭坐标轴网格标签显示

#### 3.3.2 支持的图形类型

系统支持以下 12 种图形类型：

| 类型 | 说明 |
| --- | --- |
| `sticky-note` | 便利贴，支持文本内容 |
| `product-card` | 产品讨论卡片，含标题、描述、标签、优先级、状态、投票字段 |
| `text` | 大标题文本 |
| `rectangle` | 矩形 |
| `rounded-rectangle` | 圆角矩形 |
| `circle` | 圆形 |
| `diamond` | 菱形 |
| `triangle` | 三角形 |
| `connector` | 连接线和箭头，支持锚点吸附到其他图形 |
| `pen` | 手绘笔迹 |
| `comment` | 批注，comment 角色用户可操作 |
| `frame` | 框架/容器 |

每个图形拥有以下基本属性：
- 全局唯一 UUID（前端生成，创建后不可变）
- 位置（绝对坐标 x, y）
- 尺寸（宽 w、高 h；圆形为半径）
- 视觉属性（填充色、描边色、描边宽度、透明度、字号等）
- 文本类图形额外携带文本内容
- connector 额外携带起止图形的 UUID 引用及锚点信息

#### 3.3.3 支持的图形操作

| 操作 | 说明 |
| --- | --- |
| `create` | 在指定位置创建一个新图形 |
| `update` | 修改图形的视觉属性、文本内容或其他可编辑字段 |
| `delete` | 删除一个或多个指定 UUID 的图形 |

- 移动和缩放最终通过 `update` 操作提交坐标/尺寸变化
- 拖拽移动过程中通过 `shape-preview` 消息广播预览状态，不写入操作日志
- 单个图形被删除时，以该图形为端点的 connector 应自动级联删除

#### 3.3.4 编辑辅助功能

- **多选**：框选、Ctrl/Cmd 点击多选、Shift 扩展选区
- **分组/取消分组**：将多个图形组合为一个逻辑组，支持整体操作
- **撤销/重做**：Ctrl/Cmd+Z 撤销，Ctrl/Cmd+Y 或 Ctrl/Cmd+Shift+Z 重做
- **复制/粘贴**：Ctrl/Cmd+C 复制，Ctrl/Cmd+V 粘贴，支持 Duplicate 快速复制一份
- **右键菜单**：提供上下文相关操作入口
- **快捷键查询**：独立浮层展示全部快捷键列表
- **导出**：支持导出为 PNG（当前视口）、Markdown、JSON 三种格式
- **导入**：支持将 Markdown / TXT / CSV / JSON 文件内容转换为白板图形

### 3.4 实时协同

#### 3.4.1 操作广播

- 房间内任一用户的 op 操作实时广播给同房间所有其他用户
- 本地操作先于服务端确认立即在本地画布生效（乐观更新），服务端回执 `op-ack` 后以服务端 HLC 校准版本
- 后端收到 op 后：应用到内存副本 → 立即回执 ACK → 异步广播 + 批量持久化（ACK 路径不等待持久化完成）
- 每个 WebSocket session 有独立发送队列（FIFO，容量上限 256 条）

#### 3.4.2 光标同步

- 每位用户的鼠标光标位置实时广播给同房间其他用户
- 光标以用户分配的颜色标注，显示在所有其他用户的视图中
- 光标消息属于 transient（非可靠）消息，细微移动（< 3px）过滤不发送，减少无效广播

#### 3.4.3 拖拽预览

- 用户拖拽图形时，`shape-preview` 消息实时广播拖拽预览位置给其他用户
- 预览消息属于 transient 消息，按图形 ID 做 latest-wins 合并，不写入操作日志

#### 3.4.4 背压控制

- 发送队列中 transient 消息积压超过阈值时，新到 transient 消息直接丢弃（不影响可靠消息）
- 可靠消息积压超过队列上限时，强制断开该客户端连接
- 接收端按 animation frame 合并远端 cursor / shape-preview，避免过度重绘

### 3.5 冲突消解（CRDT）

- 系统采用基于操作的 CRDT 模型，属性级 LWW 策略
- 每条操作携带 HLC 时间戳，格式：`physicalMs.logicalCounter.nodeId`
- 多副本对同一图形同一属性的并发修改，按如下优先级消解冲突：
  1. 物理时间（毫秒）大者优先
  2. 物理时间相同时，逻辑计数器大者优先
  3. 逻辑计数器相同时，nodeId 字典序大者优先（确定性打破平局）
- 删除操作使用 tombstone 机制（记录删除 HLC 和删除者），create 操作 HLC 必须晚于 tombstone 才能"复活"图形
- opId 全局唯一，后端对重复 opId 做幂等去重，保证操作恰好应用一次

### 3.6 历史记录与时光回放

#### 3.6.1 操作持久化

- 房间内所有 op 操作均持久化到 MySQL `operation_logs` 表，不因节点重启或用户离线丢失
- 操作日志字段：roomId、opId、userId、opType、shapeId、shapeType、HLC 时间戳、操作载荷（LONGTEXT）、写入时间
- 操作到达后先 ACK 客户端，由后台写入队列批量 flush（缓冲满或定时触发 saveAll），减少数据库压力

#### 3.6.2 快照机制

- 后端定期由 Owner 节点对房间画布生成全量 versioned snapshot（每个属性保留 value、HLC、写入者）
- 快照写入后自动裁剪已被覆盖的旧操作日志，控制存储增长
- 历史查询接口优先读取目标时刻之前最近快照，叠加快照之后到目标时刻的增量 op，避免从零重放

#### 3.6.3 时光回放 UI

- 前端提供时间轴滑块 UI，用户拖动滑块可查看房间在任意历史时刻的画布状态
- 时间轴上以蓝色刻度线标示快照锚点位置（通过 `GET /api/rooms/{roomId}/history/anchors` 获取）
- 顶部横幅标记"历史回放模式"，提供"在画布预览"和"返回实时"双按钮
- 历史回放视图为只读，不影响当前协作中的画布状态

### 3.7 断线重连与状态恢复

- 客户端在 WebSocket 连接断开后自动尝试重连，重连延迟策略为指数退避：`[1s, 2s, 4s, 8s, 15s]`
- 断线期间本地产生的 pending op 缓存到 localStorage，重连成功后自动补发
- 重连成功后客户端重新发送 join 消息，后端返回最新 snapshot + 增量 ops 完成状态重建
- 收到 `op-ack` 时使用服务端 HLC 校准本地副本版本

### 3.8 会议协作

#### 3.8.1 会议阶段管理

- 支持会议阶段的实时同步切换，所有用户实时看到当前激活阶段
- 支持自定义会议阶段列表（阶段名称、顺序）
- 内置 16 种阶段模板，覆盖以下场景：
  - 基础讨论：SWOT、User Journey、Decision Matrix 等
  - 产品评审：PRD Review、Risk Map 等
  - 优先级：ICE Prioritization、RICE Scoring 等
  - 复盘增长：GTM Plan、Experiment Plan 等

#### 3.8.2 会议聊天与表情

- 支持房间内实时聊天消息，消息历史保留在会话期间
- 支持 8 种 emoji 表情消息，以浮层动画形式展示在画布上

### 3.9 AI 辅助创作

#### 3.9.1 AI 白板内容生成

- 用户可通过底部 AI 助手入口发起自然语言指令
- AI 读取当前会议阶段、白板已有对象和最近会议聊天作为上下文
- AI 返回的内容直接转换为协作 op（create / update / delete），应用到画布并广播给所有成员
- AI 生成的 op 经过后端和前端双层白名单校验（shape 类型、属性范围、op 类型）

#### 3.9.2 多智能体编排

- 复杂生成请求自动触发多智能体流程：
  1. **Planner**：主 AI 解析意图并划分画布区域
  2. **Workers**：多个子 AI 并行生成各区域内容
  3. **Merger**：合并子 AI 结果，补充跨区域连接线和总标题
- 简单请求自动降级为单次 AI 调用，减少延迟和 token 消耗

#### 3.9.3 会议总结生成

- 用户可发起会议总结请求，AI 读取白板内容和会议聊天生成 Markdown 格式总结
- 总结可直接下载为 `.md` 文件

#### 3.9.4 AI 安全约束

- 后端对 AI prompt 长度、上下文长度、op 类型、shape 类型和属性做白名单清洗，拒绝非法生成内容
- AI 请求有超时限制和频率限制（均可通过环境变量配置）
- 后端使用 OpenAI-compatible Chat Completions 接口，默认配置为 DeepSeek 服务

### 3.10 可观测性与诊断

- 前端提供协作诊断抽屉，展示：当前连接节点、WebSocket URL、重连次数、pending ops 数量、最近恢复/回放操作数
- 前端定期轮询拉取后端 `cache-stats`（缓存命中率）和 `queue-stats`（队列积压、丢弃、断开数）
- 后端暴露以下运维接口：
  - `GET /api/cluster/nodes`：当前活跃节点列表
  - `GET /api/cluster/cache-stats`：房间元数据缓存指标（命中率、加载耗时等）
  - `GET /api/cluster/queue-stats`：WebSocket 发送队列指标（活跃会话、积压消息、丢弃数、断开数）

## 4. 非功能需求

### 4.1 性能

- 同节点内操作的 ACK 往返延迟应低于 **50ms**（局域网，Docker Compose 环境）
- 跨节点协同操作（经 Redis Pub/Sub 广播）的端到端延迟应低于 **2000ms**（单机双容器环境）
- 房间元数据 REST 路由查询平均延迟应低于 **20ms**，缓存热身后命中率应达到 **95%** 以上
- 单个房间支持的图形数量不少于 1000 个，在此规模下画布交互（拖动、缩放）保持流畅（≥ 30 FPS）
- op 批量持久化吞吐量目标：单节点不低于 **100 ops/s**，批量写入应将独立事务数压缩至 op 总数的 5% 以内
- 历史快照查询（含 snapshot 反序列化 + 增量 op 加载）应在 **500ms** 内完成
- transient 消息（光标、预览）高并发时不应引发 WebSocket 会话强制断开，丢弃降级可接受

### 4.2 可用性

- 单个 WebSocket 节点故障不影响其他节点和其承载的无关房间的可用性
- 客户端在连接断开后自动重连，重连后自动恢复房间状态
- 已持久化到 MySQL 的操作保证不丢失
- 系统提供 `GET /api/health` 健康检查端点，用于容器编排层的存活探测

### 4.3 一致性

- 系统提供**最终一致性**保证：在没有新操作产生的前提下，所有副本最终收敛到相同的画布状态
- 系统提供**因果一致性**保证：op 依赖关系通过 HLC 偏序关系保持，后续操作在所有副本上按相同因果顺序应用
- 系统**不**提供强一致性（线性一致性）保证，这是在 CAP 权衡中为可用性与分区容忍性做出的设计选择
- 操作幂等性：重复 opId 的操作由服务端去重，不会被二次应用
- history 接口对同一参数的多次重复读取应返回稳定结果（无竞态翻转）

### 4.4 可扩展性

- WebSocket 服务节点支持水平扩展（默认部署 backend1 / backend2 两个节点，可继续增加）
- 房间到节点的路由采用**一致性哈希**（SHA-256，支持虚拟节点），同一房间的 Owner 节点在节点列表稳定时保持不变
- Redis 跨节点广播按房间哈希分片到 64 个频道，减少无关订阅消息处理
- 节点注册与发现基于 Redis 心跳，新增节点后自动纳入路由环

### 4.5 容错性

- 批量写后置策略：整批 flush 失败时降级为逐条保存，跳过重复/格式错误数据，不因单条坏数据丢失全批
- 客户端发送的操作在网络断开时缓存于 localStorage，重连后自动补发
- Redis Pub/Sub 订阅端在本节点无该房间会话时跳过消息转发，避免无效处理
- WebSocket 发送队列背压机制保护服务端内存，避免慢客户端引发 OOM

### 4.6 安全性

- 账号密码存储使用 PBKDF2 + 随机盐，不可逆
- WebSocket 接入使用短期 JWT（Join Token）校验，Token 绑定 roomId，防止越房间访问
- Auth Token 用于 REST API 的用户身份认证
- 后端在每条 op 消息处理时校验用户权限，`view` 角色写操作请求被拒绝并返回 error 消息
- AI 接口对 prompt 和返回 op 做白名单清洗，防止恶意注入

## 5. 技术约束

| 层次 | 技术选型 |
| --- | --- |
| 前端框架 | React 19 + TypeScript + Vite |
| 画布渲染 | Konva |
| 前端状态管理 | Zustand |
| 后端框架 | Java 21 + Spring Boot 3.5 + Spring WebSocket |
| 持久化 | MySQL 8，Spring Data JPA + JDBC 批量插入 |
| 缓存 | Caffeine（本地缓存） |
| 跨节点广播 | Redis 7 Pub/Sub，64 分片频道 |
| 路由策略 | 一致性哈希（SHA-256，支持虚拟节点），节点发现基于 Redis 心跳 |
| 时钟方案 | Hybrid Logical Clock (HLC)，格式 `physicalMs.logicalCounter.nodeId` |
| 冲突消解 | 属性级 LWW + HLC，tombstone 删除标记 |
| AI 接口 | OpenAI-compatible Chat Completions（默认 DeepSeek） |
| 容器编排 | Docker Compose，Nginx 统一入口（端口 8088） |

## 6. 系统架构概览

```
浏览器
  │  REST /api/*
  │  WebSocket /ws/backend1/collab
  │  WebSocket /ws/backend2/collab
  ▼
Nginx :8088
  ├── / → frontend (Vite :5173)
  ├── /api/* → backend_servers (负载均衡)
  ├── /ws/backend1/collab → backend1:8080
  └── /ws/backend2/collab → backend2:8080

backend1 / backend2 (Spring Boot :8080)
  ├── CollabWebSocketHandler   ← 消息分发入口
  ├── RoomReplicaService       ← 内存副本 + CRDT merge
  ├── HistoryService           ← 操作日志批量写 + 快照
  ├── RoomSessionRegistry      ← session 管理 + 广播
  ├── SessionSendQueue         ← 每 session 发送队列 + 背压
  ├── RedisRealtimeBroadcaster ← 跨节点 Pub/Sub 发布
  ├── RedisRoomEventSubscriber ← 跨节点 Pub/Sub 订阅
  ├── ConsistentHashRing       ← 房间→节点路由
  ├── RedisNodeRegistry        ← 节点心跳注册与发现
  └── AiService                ← AI 多智能体编排

Redis 7
  ├── cocanvas:room-events:{0..63}           ← op 广播分片频道
  ├── cocanvas:room-transient-events:{0..63} ← cursor/preview 分片频道
  └── cocanvas:nodes:{nodeId}               ← 节点心跳（TTL 15s）

MySQL 8
  ├── rooms              ← 房间元数据
  ├── operation_logs     ← 操作日志（索引：roomId+hlc）
  ├── snapshots          ← 定期全量快照
  ├── users              ← 账号信息
  └── room_members       ← 成员角色表
```

## 8. WebSocket 消息协议

### 入站消息（客户端 → 服务端）

| 消息类型 | 说明 |
| --- | --- |
| `join` | 携带 joinToken 加入房间 |
| `op` | 白板操作（create / update / delete），含 opId、shapeId、payload、hlc |
| `cursor` | 光标位置广播，transient |
| `shape-preview` | 拖拽预览广播，transient |
| `room-chat` | 聊天消息 |
| `room-emoji` | 表情消息 |
| `room-phase` | 单个阶段切换 |
| `room-phases` | 自定义阶段列表更新 |

### 出站消息（服务端 → 客户端）

| 消息类型 | 说明 |
| --- | --- |
| `joined` | join 回执，含当前在线 peer 列表 |
| `peer-joined` / `peer-left` | 成员加入/离开通知 |
| `op-ack` | 操作确认，含服务端 HLC |
| `op-broadcast` | 操作广播（来自其他用户） |
| `cursor-broadcast` | 光标广播 |
| `shape-preview-broadcast` | 预览广播 |
| `room-chat-broadcast` | 聊天广播 |
| `room-emoji-broadcast` | 表情广播 |
| `room-members` | 成员列表变化通知 |
| `room-phase-broadcast` / `room-phases-broadcast` | 阶段变化广播 |
| `error` | 错误信息（权限拒绝、token 无效等） |

## 10. 非目标（明确不做）

以下功能明确不在本项目范围内：

- 富文本编辑（文本仅支持纯文本内容）
- 图形图层管理（z-index 手动排序）
- 图片/附件上传（图片 shape 类型已定义但未实现渲染）
- 多选整体缩放（框选后统一缩放尺寸）
- 完整内容范围 PNG 导出（当前为视口快照）
- 移动端适配（仅保证桌面浏览器可用）
- 端到端加密（仅保证传输层 TLS）
- 跨房间数据共享或模板市场
- 操作回放的逐步动画演示（回放直接跳到目标时刻状态）
- 批注/评论的线程化讨论（comment 对象仅支持创建，无回复链）
- 实时语音/视频（voiceEnabled 字段预留，功能未实现）
